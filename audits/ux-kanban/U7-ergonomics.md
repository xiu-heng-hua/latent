# U7 — Editing ergonomics & workflow

> Depends on **U3** (panels & controls) and **U4** (file & session), board sequencing step 7 ([`README.md`](README.md) §"Recommended sequencing"; this epic + U8 are the last two commits, 9–10). It turns the editor's *interaction* — once it has organized panels (U3) and a real file/config layer (U4) — into a fluent workflow: a clickable history, named variants with thumbnails, copy/paste + reset of develop settings, saved presets, a complete discoverable shortcut set, and a runtime CPU↔GPU toggle. Every card here is **ergonomics over a fixed pipeline**: per Global rule 2 ([`README.md`](README.md) §"Global rules") **no rendered pixel changes** — these cards re-order, name, copy, snapshot, and re-target the *same* `Settings` the engine already renders, and (U7-C6) swap *which* backend renders them, never the render math.
>
> **Reuse, don't reinvent.** The per-variant [`History<Settings>`](../../latent-edit/src/history.rs) is the single source of truth ([`gui.rs:111`](../../latent-app/src/gui.rs)) — every workflow action (jump-to-step, paste, reset-all, apply-preset) lands as one `begin`/`commit` gesture on the active history so undo parity is automatic. Reuse **U3**'s collapsible-section widget (U3-C1) for the new side panels (history/variants), **U3**'s reset affordances (U3-C5) for reset-all, and **U3-C4**'s cheat-sheet as the *single sink* the shortcut table (U7-C5) feeds. Reuse **U4**'s persisted app config (U4-C6) for presets, recent state, and the GPU preference — presets and the backend pref are config data, not sidecar data. The whole epic is **single-image scope** ([`README.md`](README.md) §"Product decisions" 1): variants and presets are within one open RAW, not a catalog.
>
> **Shared surface:** the live-render machinery — `RenderState` (one render at a time, latest-wins coalescing, [`gui.rs:153-166`](../../latent-app/src/gui.rs)), `render_preview`/`spawn`/`poll_render` ([`gui.rs:254-329`](../../latent-app/src/gui.rs)) — is touched by **every** card that changes settings (they set `dirty`/call `render_preview`) and is *load-bearing* for **U7-C6**: the backend may only be swapped when no render is in flight (`RenderState::is_busy()`), then a re-render is forced. The variant list (`Vec<History<Settings>>` + `active: usize`, [`gui.rs:111-113`](../../latent-app/src/gui.rs)) is the shared state for C1/C2/C3; the [`Document`](../../latent-edit/src/lib.rs)/sidecar round-trip ([`gui.rs:228-247`](../../latent-app/src/gui.rs), `from_ron`/`to_ron` [`latent-edit/src/lib.rs:912-940`](../../latent-edit/src/lib.rs)) is the shared persistence for C2. Take the cards roughly in order: **C1**/**C2** build the history/variant panels on U3's section widget; **C3**/**C4** add copy-paste/reset and presets over U4's clipboard/config; **C5** centralizes shortcuts (and wires C1/C2/C3/C6 actions into one table for U3-C4's sheet); **C6** adds the runtime backend toggle last, since it leans on U4-C6 (pref) and U0-C5 (status bar).
>
> **Additive data-model note (forward-compat, Global rule 4):** two cards want new persisted state. **U7-C2** needs a *variant name* — `Document.variants: Vec<Settings>` ([`latent-edit/src/lib.rs:904`](../../latent-edit/src/lib.rs)) has no name field; this epic adds one **without breaking old sidecars** (see C2 for the exact shape + the `#[serde(default)]` rule). **U7-C1** wants a clickable history — [`History<T>`](../../latent-edit/src/history.rs) today exposes only `current`/`undo`/`redo`/`can_*` and no entry list or jump-to-index; C1 specifies the **minimal additive API** (a `len` + a `jump_to`) kept forward-compatible. Neither change touches render math.

### U7-C1 — History panel
- Implements: drastic-UI goal (visible, navigable edit history)            Priority: Medium
- Crates/files: [`latent-app/src/gui.rs`](../../latent-app/src/gui.rs) → `gui/panels/history.rs` (post-U0 module tree); [`latent-edit/src/history.rs`](../../latent-edit/src/history.rs) (additive read/jump API)
- Depends on: **U3-C1** (collapsible section widget the panel lives in)            Blocks: **U7-C5** (the undo/redo shortcuts are wired through the same actions)
- Heads-up:
  - **The panel is a window onto the active variant's [`History<Settings>`](../../latent-edit/src/history.rs)** ([`gui.rs:111`](../../latent-app/src/gui.rs), `active_history` [`gui.rs:222-224`](../../latent-app/src/gui.rs)). Today undo/redo are only the two toolbar buttons ([`gui.rs:373-384`](../../latent-app/src/gui.rs)) and the `Cmd/Ctrl+Z` / `Shift+Z` / `Y` handler ([`gui.rs:354-368`](../../latent-app/src/gui.rs)); there is **no list view and no jump-to-step**. This card adds a section (use U3-C1's collapsible) listing the steps with the current position marked, where clicking step *k* navigates straight to it.
  - **`History` does not expose its steps and cannot jump — add the minimal API.** [`history.rs`](../../latent-edit/src/history.rs) holds `undo: VecDeque<T>`, `redo: Vec<T>`, `current: T` (all private, [`history.rs:21-29`](../../latent-edit/src/history.rs)) and offers only `undo()`/`redo()` (one step, [`history.rs:90-110`](../../latent-edit/src/history.rs)) and `can_undo`/`can_redo`. To back a clickable list, add **forward-compatible, additive** methods on `impl<T: Clone + PartialEq> History<T>` — do **not** restructure the `VecDeque`/`Vec` representation (the bounded-`undo` + cleared-on-commit-`redo` invariant is documented and tested, [`history.rs:14-19`](../../latent-edit/src/history.rs), `history_caps_undo_depth` [`history.rs:179-202`](../../latent-edit/src/history.rs)):
    - `undo_len(&self) -> usize` and `redo_len(&self) -> usize` (or one `len()` = total steps + a `position()` index) so the panel can size and index the list.
    - `jump_to(&mut self, target_undo_len: usize) -> bool` (or an equivalent step-delta) that calls `undo()`/`redo()` the right number of times to land at the requested position — **implemented purely in terms of the existing `undo`/`redo`**, so the redo-clear-on-commit and capacity invariants are unchanged and no new failure mode is introduced. Returns whether it moved (so the caller can gate `dirty`).
    - Optionally a *cheap label* per step. The steps are full `T` snapshots, not deltas, so there is no stored action name; the panel can label generically ("Edit 1", "Edit 2", …, "Current") or, if a per-step label is wanted later, that is a **separate, additive** change (a parallel `Vec<String>` of labels pushed in `commit`) — **note it, don't build it now** unless the acceptance asks. Keep the first cut label-free to stay minimal and forward-compatible.
  - **Navigation must be `Ctrl+Z`-parity-preserving.** A jump is just a run of `undo`/`redo`, which is exactly what the keyboard handler and toolbar buttons already do — so a panel click and a `Ctrl+Z` press converge on the same `History` state. After a jump, set `dirty = true` so `render_preview` ([`gui.rs:254-266`](../../latent-app/src/gui.rs)) refreshes the texture, identical to the post-undo `dirty` set at [`gui.rs:495-499`](../../latent-app/src/gui.rs). Do **not** add a second navigation path: route the toolbar Undo/Redo, the keyboard shortcuts, and the panel clicks through the same `History` calls.
  - **Per-variant, switches with `active`.** The panel shows *the active variant's* history; switching variants ([`gui.rs:388-396`](../../latent-app/src/gui.rs)) repaints it against the new history — no extra state, just read `self.variants[self.active]`. A brush stroke is already one history step (begin-on-press/commit-on-release, [`gui.rs:536-565`](../../latent-app/src/gui.rs)), so strokes appear as single entries — no special-casing.
  - Cross-cutting: do **not** change the autosave gate (`is_idle`, [`gui.rs:228-231`](../../latent-app/src/gui.rs)) — a jump is a completed gesture (no open `begin`), so the existing autosave-on-idle persists the landed state correctly. Keep the new `History` methods `#[must_use]` where they return a moved-bool, matching the crate's style.
- Acceptance:
  - `History` gains additive, tested read/jump methods (`undo_len`/`redo_len` or `len`+`position`, and `jump_to`) implemented over the existing `undo`/`redo`; new tests `history_reports_len_and_position` and `history_jumps_to_an_arbitrary_step` (jump back N then forward M lands on the same state as N `undo`s + M `redo`s) pass, and the existing `undo_redo_round_trips` / `history_caps_undo_depth` / `a_new_edit_clears_the_redo_branch` still pass unchanged.
  - The history panel lists the active variant's steps with the current position marked; clicking a step navigates to it and triggers a re-render; `Ctrl+Z` / the toolbar Undo land on the *same* state as clicking the adjacent step (parity), and switching variants re-targets the panel to that variant's history.
  - `cargo fmt --check`, `cargo clippy --workspace --all-targets` (zero warnings), `cargo test --workspace` stay green (Global rule 1).

### U7-C2 — Variant management
- Implements: drastic-UI goal (named, browsable variants)            Priority: Medium
- Crates/files: [`latent-app/src/gui.rs`](../../latent-app/src/gui.rs) → `gui/panels/variants.rs`; [`latent-edit/src/lib.rs`](../../latent-edit/src/lib.rs) (a variant-name field, serde-default)
- Depends on: **U3-C1** (the section/panel widget)            Blocks: —
- Heads-up:
  - **Today variants are a bare numbered row.** The toolbar renders `selectable_label("{i+1}")` per variant plus a `"+"` that copies the active one ([`gui.rs:386-403`](../../latent-app/src/gui.rs)); state is `variants: Vec<History<Settings>>` + `active: usize` ([`gui.rs:111-113`](../../latent-app/src/gui.rs)). There is **no name, no thumbnail, no delete, no reorder**. This card turns that into a managed list: **name**, **small thumbnail**, **duplicate / delete / reorder**.
  - **Naming needs a parallel/wrapper structure — `Settings` has no name and shouldn't.** `Document.variants: Vec<Settings>` ([`latent-edit/src/lib.rs:904`](../../latent-edit/src/lib.rs)) is what the sidecar stores; `Settings` ([`latent-edit/src/lib.rs:19-28`](../../latent-edit/src/lib.rs)) is pure develop data (a name is UI metadata, not an adjustment, and putting it on `Settings` would pollute `History<Settings>` equality and the no-op-gesture detection at [`history.rs:77-87`](../../latent-edit/src/history.rs)). Two forward-compatible options — **pick one and document it**:
    - **(preferred) a `names: Vec<String>` parallel field on `Document`** (`#[serde(default)]`, parallel to `variants`, `names[i]` ↔ `variants[i]`, a missing/short entry defaulting to empty → the UI shows "Variant {i+1}"). Smallest diff, keeps `Settings`/`History` untouched, and an old sidecar (no `names`) loads with the field defaulting to `Vec::new()`.
    - **(alternative) a `Variant { name: String, settings: Settings }` wrapper** changing `variants: Vec<Settings>` → `Vec<Variant>`. Cleaner long-term but a wider change (every `variants` access in `gui.rs` + the `from_ron`/`to_ron` round-trip + the four-plus sidecar tests at [`latent-edit/src/lib.rs:1230+`](../../latent-edit/src/lib.rs)), and the serde-compat story is harder (an old `Vec<Settings>` won't deserialize into `Vec<Variant>` without a custom adapter). **Prefer the parallel-vector option** unless a later card needs per-variant metadata beyond a name.
  - **Forward-compat is the gate (Global rule 4 / `Document::VERSION`).** Whatever shape, it must keep `Document::VERSION = 1` loading old sidecars: a pre-names sidecar still deserializes (the field defaults), and a names-carrying sidecar round-trips (`from_ron(to_ron(doc)) == doc`). Mirror the existing serde-default round-trip tests ([`latent-edit/src/lib.rs:1230`, `:1237`, `:1340-1360`](../../latent-edit/src/lib.rs)) with a `document_names_round_trip_and_old_sidecar_loads` test. **Do not bump `VERSION`** — an additive `#[serde(default)]` field is exactly the compatible-evolution the version guard ([`latent-edit/src/lib.rs:929-935`](../../latent-edit/src/lib.rs)) is designed to *not* trip on. Keep the `gui.rs` load path (`Document::from_ron`, the empty-variants backfill at [`gui.rs:44-46`](../../latent-app/src/gui.rs), and the `saved`/`autosave` comparison at [`gui.rs:54`, `:228-247`](../../latent-app/src/gui.rs)) in sync — `autosave` compares `Vec<Settings>`; it now must also persist/compare names so a rename autosaves.
  - **Thumbnails: render each variant tiny, cached.** Reuse the existing render path — `render(base, settings, backend)` ([`gui.rs:193`](../../latent-app/src/gui.rs)) over a *very small* base (downscale `self.preview` further, e.g. longest side ~96px, à la `PREVIEW_MAX_DIM` [`gui.rs:22`](../../latent-app/src/gui.rs)) → `to_color_image` ([`gui.rs:1232-1235`](../../latent-app/src/gui.rs)) → a per-variant `egui::TextureHandle`. **Respect the one-render-at-a-time gate:** thumbnails must **not** contend with the live preview render. Either (a) render them lazily off the same `RenderState`/`spawn` worker when idle and cache by a settings hash/clone so an unchanged variant isn't re-rendered, or (b) render them synchronously at a trivial size only when the panel is visible and a variant changed — **(a) is preferred** to never block the UI (Global rule 5). Invalidate a variant's cached thumbnail when its `History::current()` changes (compare a cheap fingerprint). Pixel math is untouched — thumbnails go through the exact same `render`/`to_srgb8` transform, just smaller (Global rule 2).
  - **Duplicate / delete / reorder operate on the parallel lists together.** Duplicate = today's `"+"` copy ([`gui.rs:397-402`](../../latent-app/src/gui.rs)) but also copying the name (e.g. "Foo copy") and seeding a fresh `History::new(copy)`. Delete must keep **at least one variant** (the invariant at [`gui.rs:44-46`, `:111`](../../latent-app/src/gui.rs): "never empty"; deleting the last is disallowed) and clamp `active` (reuse the `clamp_selection` pattern, [`gui.rs:216-218`](../../latent-app/src/gui.rs)). Reorder must move the `History`, the name, *and* the cached thumbnail in lockstep, and re-point `active` to follow the moved item. These are UI-state mutations (not `Settings` edits) — they do **not** go through `History`; they change which/what variants exist, persisted by `autosave`.
- Acceptance:
  - Variants can be **named**, **duplicated**, **deleted** (never below one), and **reordered**; renaming, duplicating, deleting, and reordering all autosave and survive reload.
  - Names persist in the sidecar (parallel `names` vector or `Variant` wrapper, `#[serde(default)]`); an **old sidecar with no names still loads** (names default to empty → "Variant N" shown), proven by `document_names_round_trip_and_old_sidecar_loads`; `Document::VERSION` is unchanged.
  - Each variant shows a small thumbnail rendered through the existing `render`/`to_color_image` path at a tiny size, cached and invalidated on edit, and thumbnail rendering never blocks or starves the live preview (uses the same one-at-a-time gate).
  - `cargo fmt --check`, clippy (zero warnings), `cargo test --workspace` green (Global rule 1); the existing sidecar round-trip tests still pass.

### U7-C3 — Copy/paste develop settings + reset-all
- Implements: drastic-UI goal (transfer & reset edits)            Priority: Medium
- Crates/files: [`latent-app/src/gui.rs`](../../latent-app/src/gui.rs) → `gui/app.rs` + menu/toolbar (post-U0 shell, U0-C5)
- Depends on: **U4-C1** (in-app file switching — paste-onto-another-image is meaningful once a file can be opened into the running app)            Blocks: **U7-C5** (Copy/Paste/Reset get shortcuts)
- Heads-up:
  - **An in-app settings clipboard, held on `App`.** Add a field like `clipboard: Option<Settings>` (or `Option<Adjustments>` for develop-only — see below) to `App` ([`gui.rs:104-136`](../../latent-app/src/gui.rs)). **Copy** snapshots the active variant's `Settings` (`self.variants[self.active].current().clone()`, [`gui.rs:222-224`, `:261`](../../latent-app/src/gui.rs)) into it. This is **process-local** UI state — *not* the OS clipboard (that's for text), not persisted to config or sidecar. It survives variant switches and (with U4-C1) file switches within the session.
  - **Paste is one undo step on the target history.** Paste onto the active variant (or, post-U4-C1, after switching image) must be a single `begin`/`commit` gesture: `history.begin(); *history.current_mut() = pasted; history.commit();` (the same transaction shape the slider helpers use, e.g. `opt_point_slider` [`gui.rs:604-613`](../../latent-app/src/gui.rs)), then `dirty = true`. Because `commit` only records a step if the value actually changed ([`history.rs:77-87`](../../latent-edit/src/history.rs)), pasting an identical settings is a free no-op — correct by construction.
  - **Decide the copy scope and document it: develop vs. spatial.** `Settings` = `{ global: Adjustments, locals, geometry }` ([`latent-edit/src/lib.rs:19-28`](../../latent-app/src/gui.rs)). Geometry/crop is **image-specific** (a crop or keystone tuned for one frame is wrong on another; lens is per-lens) — so the useful default for paste-across-images is **develop only** (`global` + `locals`, excluding `geometry`), while paste-within-the-same-image may want everything. Offer the two as "Paste settings" (develop only — copies `global`+`locals`, keeps the target's `geometry`) and optionally "Paste all" (everything incl. geometry). **State the chosen default in the menu label and a code comment.** Mechanically, "develop only" = build the new `Settings` from the clipboard's `global`/`locals` but the *target's* existing `geometry`. (This mirrors the U7-C4 preset decision — keep the two consistent.)
  - **Reset-all develop = back to default, one undo step.** A "Reset all develop" command sets the active variant to a neutral develop while (per the same develop/spatial split) **preserving geometry by default** — i.e. `Settings { global: Adjustments::default(), locals: Vec::new(), geometry: <keep current> }` — or a full `Settings::default()` ([`latent-edit/src/lib.rs:17`](../../latent-edit/src/lib.rs)) for "reset everything". Decide and label which; **default to resetting develop and keeping geometry** (a user rarely wants reset to also undo their crop), and offer "Reset everything" if both are wanted. Wrap it in one `begin`/`commit` so it's a single undoable step (reuses U3-C5's reset affordance shape; don't duplicate reset logic the panels already expose per-section). After reset, `dirty = true`.
  - Cross-cutting: these are `Settings` edits, so they flow through the **same** autosave (a completed gesture → `is_idle` true → persisted, [`gui.rs:228-247`](../../latent-app/src/gui.rs)) and the **same** `render_preview` path — no new render plumbing. The Copy/Paste/Reset menu items live in U0-C5's menu/toolbar; disable Paste when `clipboard` is `None`.
- Acceptance:
  - Copy snapshots the active `Settings` to an in-app clipboard; Paste applies it to the active variant (or, with U4-C1, another image) as **one undo step** that re-renders and autosaves; pasting identical settings records no step (no-op).
  - The develop-vs-geometry scope is decided and documented (paste-develop keeps the target's geometry; an explicit "paste all" includes it) and matches the U7-C4 preset decision.
  - "Reset all develop" returns the variant to default develop (geometry kept by the documented default) in **one** undoable step; Paste is disabled when the clipboard is empty.
  - `cargo fmt --check`, clippy (zero warnings), `cargo test --workspace` green; the copy/paste/reset transforms carry a unit test on the pure mapping (e.g. `paste_develop_keeps_target_geometry`, `reset_all_returns_default_develop`).

### U7-C4 — Develop presets
- Implements: drastic-UI goal (saved, reusable looks)            Priority: Medium
- Crates/files: `gui/config.rs` (the U4-C6 app config), `gui/panels/…` (a preset list/menu), [`latent-app/src/gui.rs`](../../latent-app/src/gui.rs)
- Depends on: **U4-C6** (persisted app config is where presets live)            Blocks: **U7-C5** (apply-preset can take a shortcut/menu accelerator)
- Heads-up:
  - **A preset is a named `Settings` snapshot stored in config, applied like a paste.** Save = name the active variant's current `Settings` (`current().clone()`, [`gui.rs:261`](../../latent-app/src/gui.rs)) and write it into the U4-C6 app config (the persisted store — *not* the per-image `.ron` sidecar; presets are global to the app, reusable across images, exactly like recent-files/GPU-pref live in config per [`README.md`](README.md) U4-C6). Apply = the **same one-`begin`/`commit` transaction** as U7-C3 paste, so apply-preset is one undo step and re-renders; **share the apply code path with U7-C3's paste** rather than duplicating it.
  - **Exclude geometry/crop from a preset — decide and document (consistent with U7-C3).** A preset is a *look* (exposure, tone, color, curves, detail), reused across different framings; baking a crop/keystone/lens/straighten into a preset would mis-apply it to a differently-framed image. So a preset stores and applies the **develop** part (`global` + `locals`) and leaves the target's `geometry` ([`latent-edit/src/lib.rs:535-559`](../../latent-edit/src/lib.rs)) untouched on apply — the same develop/spatial split U7-C3 chose. **Document this in the save dialog and a code comment.** (If "include geometry" is ever wanted it's an explicit opt-in checkbox at save time, like Lightroom's preset checkboxes — note it as a future option, don't build the full matrix now.) Storing the whole `Settings` but *applying* only the develop part is fine and keeps the data shape simple; storing only `{ global, locals }` is equally valid — pick the simpler serialize.
  - **Config persistence reuses U4-C6's atomic write (Global rule 9).** Presets are config data: written temp-then-rename with the rest of the app config, a failed write raising a toast (U8-C2), never truncating. They're `Serialize`/`Deserialize` `Settings` (or `{global,locals}`) — already serde-derived ([`latent-edit/src/lib.rs:19`, `:364`](../../latent-edit/src/lib.rs)) — so a `Vec<(String, ...)>` of named presets drops straight into the config struct with `#[serde(default)]` (an old config with no presets loads with an empty list). No new crate.
  - **Apply re-renders and autosaves through the existing paths.** Because apply mutates the active variant's `Settings` via `History`, the existing autosave ([`gui.rs:228-247`](../../latent-app/src/gui.rs)) and `render_preview` ([`gui.rs:254-266`](../../latent-app/src/gui.rs)) handle persistence + redraw with no new plumbing. The preset *list* itself persists in config, not the sidecar.
  - Cross-cutting: deleting/renaming presets edits the config list (clamp the selection like [`gui.rs:216-218`](../../latent-app/src/gui.rs)); a sanitize pass on apply is free if you route through `Settings` (the sidecar load already sanitizes, [`latent-edit/src/lib.rs:936-938`](../../latent-edit/src/lib.rs)) — a preset from a hand-edited config is untrusted, so sanitize before applying.
- Acceptance:
  - Saving the current edit as a named preset stores its **develop** settings (`global`+`locals`, geometry excluded) in the U4-C6 app config; presets persist across launches (atomic config write).
  - Applying a preset re-applies its develop adjustments to the active variant as **one undo step** (sharing U7-C3's apply path), re-renders, and leaves the target's geometry intact; the geometry-exclusion decision is documented in the UI + code.
  - An old config without presets loads (empty list, `#[serde(default)]`); a hand-edited/non-finite preset is sanitized before apply.
  - `cargo fmt --check`, clippy (zero warnings), `cargo test --workspace` green; a test pins `apply_preset_excludes_geometry` and config preset round-trip.

### U7-C5 — Keyboard shortcuts
- Implements: drastic-UI goal (discoverable, complete shortcuts)            Priority: Medium
- Crates/files: `gui/shortcuts.rs` (the central table), wired across `gui/panels/*` and the canvas; feeds **U3-C4**'s cheat-sheet
- Depends on: **U3-C4** (the cheat-sheet that renders the table)            Blocks: —
- Heads-up:
  - **One source of truth for shortcuts.** Today shortcuts are scattered and minimal: only `Cmd/Ctrl+Z` / `Shift+Z` / `Y` (undo/redo) are handled, inline in `update` ([`gui.rs:354-368`](../../latent-app/src/gui.rs)); everything else is button-only. This card defines a **single table** (`gui/shortcuts.rs`) — a list of `{ action, KeyboardShortcut, description }` — that both (a) the input handler dispatches from and (b) **U3-C4's cheat-sheet renders from**, so the help is generated, never hand-maintained alongside the bindings. *Don't* let any card add a binding that isn't in this table.
  - **The complete set** (each entry an action the existing code already has or a sibling card adds):
    - **File:** Open (U4-C1), Export (the existing `export`, [`gui.rs:272-286`](../../latent-app/src/gui.rs)).
    - **Edit:** Undo / Redo (the existing handler — *move it into the table*), Copy / Paste / Reset-all (U7-C3), Apply-preset (U7-C4, optional accelerator).
    - **View / zoom:** Fit / 100% / Zoom-in / Zoom-out (U1-C3's zoom levels), Before/after toggle (U1-C4), panel toggle (U3-C3's hide-panel).
    - **Tools:** tool-select — e.g. `C` = crop (U2-C2), `R` = a rotate/straighten or radial tool, brush size `[` / `]` (U2-C7 already implies size shortcuts) — name the letters in the table, don't hardcode them in the canvas.
    - **Variants:** next / previous variant (cycles `active`, [`gui.rs:388-396`](../../latent-app/src/gui.rs)).
  - **Don't fire while typing — this is the load-bearing correctness constraint.** The export-path field ([`gui.rs:476-479`](../../latent-app/src/gui.rs), `text_edit_singleline`), the variant-name field (U7-C2), numeric slider entry (U3-C2), and the preset-name field (U7-C4) all take keyboard focus. ### U7-C6 — In-app GPU/CPU backend toggle + status
- Implements: drastic-UI goal (runtime backend choice + status)            Priority: Medium
- Crates/files: [`latent-app/src/gui.rs`](../../latent-app/src/gui.rs) → `gui/app.rs`; reuse [`main::select_backend`](../../latent-app/src/main.rs) logic (move it to a shared spot)
- Depends on: **U4-C6** (persist the GPU preference), **U0-C5** (the status bar that shows the active backend)            Blocks: —
- Heads-up:
  - **Backend is fixed at launch today — make it runtime-switchable.** `select_backend(use_gpu)` ([`main.rs:67-78`](../../latent-app/src/main.rs)) builds a `GpuBackend::new()` (with CPU fallback on error) or `CpuBackend` *once*, at the composition root, and `gui::run` stores it as `backend: Arc<dyn Backend>` ([`gui.rs:33`, `:135`](../../latent-app/src/gui.rs)); there is **no in-app way to switch**. This card adds a View/Settings toggle that rebuilds the backend, swaps `App.backend`, and forces a re-render.
  - **Reuse `select_backend` — move it somewhere shared.** The toggle must build the new backend with the *exact same* logic the launch path uses (GPU-with-CPU-fallback), so move `select_backend` out of `main.rs` into a shared module (e.g. `gui::app` or a small `backend` module) and call it from **both** `main` and the toggle. Don't fork the selection logic. The result is `Box<dyn Backend>` → `Arc::from` ([`gui.rs:33`](../../latent-app/src/gui.rs)), assigned to `self.backend`. The `Backend` trait is `Send + Sync` ([`README.md`](README.md) §"Ground in"; `gui.rs:32` comment, [`gui.rs:18`, `:135`](../../latent-app/src/gui.rs)), so a freshly-built backend swaps into the worker's `Arc::clone(&self.backend)` ([`gui.rs:263`, `:280`](../../latent-app/src/gui.rs)) cleanly.
  - **Never swap while a render is in flight — this is the load-bearing gotcha.** A `RenderJob` captures `backend: Arc::clone(&self.backend)` at spawn time ([`gui.rs:259-264`, `:276-283`](../../latent-app/src/gui.rs)) and runs on the worker; the in-flight render owns its `Arc` and must finish on the backend it started with. So the toggle must **wait for the worker to be idle** before swapping: gate the swap on `!self.render.is_busy()` ([`gui.rs:163-165`](../../latent-app/src/gui.rs)) — if busy, record a *pending backend switch* and perform it in `poll_render` ([`gui.rs:307-329`](../../latent-app/src/gui.rs)) right after the in-flight render is consumed and before the coalesced re-render is spawned (so the next render uses the new backend). This reuses the **same** one-at-a-time gate and latest-wins coalescing that already serialize renders — don't add a second synchronization mechanism. After the swap, set `dirty`/call `render_preview` so the image re-renders on the chosen backend; a `RenderState::is_busy` check already exists and is unit-tested (`render_state_gate_coalesces_to_one`, [`gui.rs:1287-1310`](../../latent-app/src/gui.rs)) — extend that pattern.
  - **GPU init can fail → fall back to CPU + toast.** `GpuBackend::new()` returns `Err` when no device is available; `select_backend` already prints and falls back to `CpuBackend` ([`main.rs:69-77`](../../latent-app/src/main.rs)). In-app, that fallback must surface to the user: a toast (U8-C2) "GPU unavailable, using CPU" and the **status bar reflecting the *actually-active* backend** (U0-C5), not the requested one. So the toggle needs to know *which* backend it got — either have the shared `select_backend` return the active kind, or probe it; the status bar shows "CPU" / "GPU" from the real result. Don't claim GPU in the status bar if it fell back.
  - **Persist the preference (U4-C6), not the backend.** The desired GPU on/off is a config field ([`README.md`](README.md) U4-C6: "GPU pref"), written through U4-C6's atomic config save; on next launch `main` seeds `select_backend` from the persisted pref instead of only the `--gpu` flag ([`main.rs:114`](../../latent-app/src/main.rs)) — decide precedence (an explicit `--gpu` flag overrides the stored pref, or the flag sets it). Persist the *preference*; the active backend is always re-derived via `select_backend` (so a saved "GPU on" still gracefully degrades to CPU if the device is gone). The status bar shows active; the pref records desired.
  - Cross-cutting (Global rule 2): swapping the backend must **not change pixels** — the CPU↔GPU equivalence is the project's invariant (the CPU/GPU equivalence tests, [`README.md`](README.md) §"Global rules" 2). This card relies on that equivalence; it does not touch backend/shader math. A toggle that produced visibly different output would be a regression in *those* tests, not this card — keep this card to swapping the `Arc`, never the math.
- Acceptance:
  - A View/Settings toggle switches CPU↔GPU at runtime: it rebuilds the backend via the shared `select_backend`, swaps `App.backend`, and re-renders the preview on the chosen backend.
  - The swap **respects the in-flight worker** — it never replaces `self.backend` while `RenderState::is_busy()`; a toggle during a render is deferred to the next idle point (in `poll_render`) and then re-renders. A test extends `render_state_gate_coalesces_to_one` to cover a pending-backend-swap deferred past a busy render.
  - GPU-init failure **falls back to CPU gracefully** with a toast, and the **status bar shows the actually-active backend** (not the requested one); the GPU **preference persists** (U4-C6) and is re-derived through `select_backend` on next launch (a stored "GPU on" still degrades to CPU when no device).
  - `select_backend` is shared (moved out of `main.rs`, called by both `main` and the toggle — not forked); `cargo fmt --check`, clippy (zero warnings), `cargo test --workspace` green (Global rule 1), and the CPU/GPU equivalence tests are untouched and still green (Global rule 2).
