# U2 — Direct on-canvas manipulation

> Depends on **U1** (board sequencing step 4, [`README.md`](README.md) §"Recommended sequencing"). This is the single biggest UX jump in the overhaul: today every spatial edit in `latent` is a **numeric slider** — crop is four sliders (Left/Top/Width/Height, [`gui.rs:902-930`](../../latent-app/src/gui.rs)), straighten is one Angle slider ([`:835-849`](../../latent-app/src/gui.rs)), keystone is two ([`:871-898`](../../latent-app/src/gui.rs)), gradient/radial mask geometry is four more ([`local_shape_block:1134-1227`](../../latent-app/src/gui.rs)), and the brush paints by mapping `resp.hover_pos()` against `resp.rect` with **its own ad-hoc screen↔image math** ([`:542-562`](../../latent-app/src/gui.rs)). You cannot *see* a mask at all. This epic replaces that with **draggable handles drawn on the image**: a crop rectangle you grab by its corners, a horizon line you drag to level, keystone corners you pull, gradient/radial handles on the shape, a colored overlay that shows the selection, and a brush cursor ring — with the numeric sliders kept as a precise fallback.
>
> **Shared surface — one overlay, one transform.** Every card in this epic draws into and hit-tests against the **same** canvas overlay, and every screen↔image conversion goes through **U1-C2's `ViewTransform`** ([`README.md`](README.md) §"Global rules" rule 3: *"U2 … consume it — they never recompute screen↔image math"*). The brush's current hand-rolled `(pos − rect) / rect.width()` mapping ([`gui.rs:544-546`](../../latent-app/src/gui.rs)) is the *anti-pattern* this epic deletes: it is fit-only, breaks the moment U1-C3 adds zoom/pan, and is duplicated math the `ViewTransform` exists to own. **C1** builds the framework (handle drawing + hit-test + drag/begin-commit wiring + a tool enum) once; **C2–C7** each add one tool on top of it and never recompute the mapping. Masks and crop stay **normalized `[0, 1]` over the oriented image** and are evaluated in SOURCE space exactly as today (rule 3) — a handle drag is just a different *input device* writing the same normalized field the slider wrote.
>
> **One-resample rule (the pipeline is sacred — rule 2).** UI work must not change rendered pixels: the preview transform and `latent_pipeline::render` math stay byte-identical, the CPU↔GPU equivalence tests stay green. **C3 is the only card permitted to touch pipeline math, and only additively** — a new `Geometry` orientation field (manual rotate 90° / flip) folded into the *existing* single resample, never a second pass. Every other card writes existing `Geometry`/`Mask` fields the engine already consumes.
>
> **Handles primary, numeric kept.** Each tool's slider block from `gui.rs` stays as a numeric-entry fallback alongside the handles (the user can still type an exact crop width or a `1.5°` straighten) — the handles become the *primary* surface, not the only one. Do **not** delete the numeric controls.
>
> **One undo step per drag.** Every drag is one `History<Settings>` gesture: `begin()` on press, mutate `current_mut()` each move, `commit()` on release — the exact pattern the sliders use through `gesture()` ([`gui.rs:582-589`](../../latent-app/src/gui.rs)) and the brush stroke uses today ([`:536-565`](../../latent-app/src/gui.rs)). `commit()` only records a step if the value changed, so a click that grabs and releases a handle without moving it costs nothing.
>
> **What `ViewTransform` must expose (from U1-C2, consumed here):** an `image→screen` and `screen→image` mapping built from the fit/zoom/pan state plus the displayed (EXIF-oriented) image size, where *image space* is **normalized `[0, 1]`** over the oriented image — the same space `Crop`/`Mask`/`Dab` already live in. C1 names the exact methods it needs (point map both directions, and a scalar for "how many screen pixels is a normalized radius") so U1-C2 provides them; if U1-C2 lands first, C1 adopts its surface verbatim. The overlay is painted in the `CentralPanel` after the image, in **screen** coordinates obtained by mapping normalized handle positions forward through the transform.
>
> **Authoritative anchors in `latent-app/src/gui.rs`** (today, all sliders): `straighten_slider` `:835-849`; `vignette_slider` `:853-867`; `keystone_block` `:871-898`; `crop_block` `:902-930`; `local_shape_block` `:1134-1227` (gradient `:1136-1154`, radial `:1155-1177`, luminosity `:1178-1195`, colorrange `:1196-1220`, brush label `:1221-1224`); brush settings `brush_radius/feather/erase` `:127-130` + UI `:461-472`; brush painting in the `CentralPanel` `:529-566` (the `resp.rect` → `nx,ny` map at `:544-546`). **In `latent-edit/src/lib.rs`:** `Geometry` `:534-573` (with `is_identity` `:564-572`); `Crop {x,y,width,height}` `:678-684`; `Perspective {vertical,horizontal}` `:579-583`; `Mask` `:60-95` + `weight_at` `:76-94`; `MaskShape` `:112-123`; `Gradient {x0,y0,x1,y1}` `:265-281`; `Radial {cx,cy,radius,feather}` `:292-308`; `Dab {x,y,radius,feather,erase}` `:327-354`. **In `latent-pipeline/src/lib.rs`:** `apply_geometry` `:1355-1464` (vignetting-pre, then straighten+keystone homography compose `:1378-1394`, then lens warp, then `crop_image` `:1446-1448`); `Transform::rotation` `:143-161`; `keystone_transform` `:1199-1213`; `crop_image` `:1499-1506`.

### U2-C1 — Canvas interaction framework
- Implements: the on-canvas interaction surface every other U2 card sits on            Priority: High
- Crates/files: `latent-app/src/gui/canvas.rs` (new, in the U0-C1 module tree); the `CentralPanel` call in `gui.rs`/`gui/…` (the image-paint site, today [`gui.rs:523-568`](../../latent-app/src/gui.rs))
- Depends on: **U1-C2** (the `ViewTransform`)            Blocks: **U2-C2, U2-C3, U2-C4, U2-C5, U2-C6, U2-C7**; **U6-C4** (mask handles follow the selected shape), **U6-C6** (WB gray eyedropper picks a canvas pixel)
- Heads-up:
  - **One place owns handle drawing, hit-testing, drag state, and the begin/commit wiring — every later card plugs into it instead of re-rolling pointer math.** Today the brush is the *only* on-canvas interaction and it hand-rolls everything inline in the `CentralPanel` ([`gui.rs:529-566`](../../latent-app/src/gui.rs)): its own `resp.rect`→normalized map ([`:544-546`](../../latent-app/src/gui.rs)), its own begin/commit gating on `drag_started`/`drag_stopped` ([`:538-565`](../../latent-app/src/gui.rs)). C1 generalizes that into a reusable layer so C2–C7 don't each duplicate it (and so the duplication doesn't drift — the same reason `gesture()` ([`:582-589`](../../latent-app/src/gui.rs)) and `clamp_selection()` ([`:216-218`](../../latent-app/src/gui.rs)) were factored out).
  - **Active-tool enum.** Add a `CanvasTool` enum — `None`, `Crop`, `Straighten`, `Keystone`, `MaskShape` (gradient/radial handles, C5), `Brush`, `WbPick` (U6-C6) — held as app state (next to `local_sel`/`brush_*` at [`gui.rs:125-130`](../../latent-app/src/gui.rs)), defaulting to `None` (plain view/pan, U1's job). The toolbar (U0-C5) selects it; the panel that owns a tool (crop section, brush section) also activates it on open. Only the active tool draws handles and consumes pointer input on the canvas — so e.g. Crop handles don't intercept a brush stroke. `None` means the canvas is pure pan/zoom (U1-C3 owns the drag then). The brush's existing "is the selected local a brush?" gate ([`:461-466`](../../latent-app/src/gui.rs), [`:531-535`](../../latent-app/src/gui.rs)) becomes "tool == Brush", and C5's mask handles activate when a gradient/radial local is selected.
  - **Consume the `ViewTransform`, never recompute it (rule 3).** The framework takes the `ViewTransform` from U1-C2 and offers the overlay two primitives: map a normalized image point → screen `Pos2` (to *draw* a handle), and map a screen `Pos2` → normalized image point (to *read* a drag). Both come straight from the transform — **delete** the brush's `(pos − rect)/rect.width()` map ([`:544-546`](../../latent-app/src/gui.rs)) and route C7's brush through this instead (that deletion is the proof C1 unified the math). It also needs a scalar "normalized-length → screen-pixels" (for drawing a radial ring or a brush ring at the right on-screen size) and its inverse "screen-pixels → normalized-length" (for a handle hit radius in image space) — both derived from the transform's scale, not re-measured off `resp.rect`.
  - **Handle + drag abstraction.** A handle is a normalized anchor point the user can grab; provide: (a) a draw helper (a filled/outlined circle or line endpoint at the screen position, themed via U0-C2 tokens, sized in screen px so it stays grabbable at any zoom); (b) a hit-test (is the pointer within a screen-px tolerance of this handle? — convert the tolerance to image space *once* via the scalar above, or test in screen space after mapping the handle forward, whichever is cleaner — but only through the transform); (c) drag state — which handle (if any) is currently grabbed, so a drag that starts on a corner keeps editing *that* corner even as the pointer moves off it. Egui gives you `drag_started`/`dragged`/`drag_stopped`/`interact_pointer_pos` on the image `Response` (the curve editor at [`:678-701`](../../latent-app/src/gui.rs) is the in-repo model for this exact begin-drag-commit shape — reuse the structure). Keep the picked-handle index in app/overlay state across frames the way the brush keeps its stroke across the drag.
  - **begin/commit wiring is part of the framework, not each card.** Expose a tiny "gesture" around a drag: on `drag_started` over a handle, call `history.begin()`; on each `dragged` frame, hand the card the new normalized pointer position to write its field and set `dirty`; on `drag_stopped`, `history.commit()`. This is exactly the brush's begin/commit-per-stroke ([`:538-565`](../../latent-app/src/gui.rs)) generalized — one undo step per drag (epic rule). A discrete click (grab + release, no move) must commit harmlessly (no-op if unchanged, like `gesture()`'s `discrete` path at [`:587`](../../latent-app/src/gui.rs) and `commit()`'s change check).
  - **Paint order & input precedence.** The overlay paints *after* the image in the same `CentralPanel`/`ScrollArea` and reads the image `Response`. When a tool is active and the pointer is over a handle, the tool consumes the drag (so U1-C3's pan doesn't also fire); when no handle is hit (or tool is `None`), the drag falls through to pan. Decide this precedence once here so each card inherits it. (The image `Response` is already `Sense::click_and_drag()` at [`:527`](../../latent-app/src/gui.rs); keep that.)
  - **No pipeline change, no render change (rule 2).** C1 is pure UI plumbing — it only draws overlays and writes existing `Settings` fields. Rendered pixels are untouched; the overlay is egui paint on top of the texture, never baked into it.
- Acceptance:
  - A reusable handle/drag abstraction exists in `gui/canvas.rs`: draw-a-handle, hit-test-a-handle (screen-px tolerance), grabbed-handle drag state held across frames, and a begin/commit-per-drag wrapper — with a `CanvasTool` active-tool enum (`None`/`Crop`/`Straighten`/`Keystone`/`MaskShape`/`Brush`/`WbPick`) in app state, defaulting to `None`.
  - **Nothing in the overlay recomputes screen↔image** — all mapping (point both directions, and the length scalar) goes through U1-C2's `ViewTransform`; the brush's old `(pos − rect)/rect.width()` map ([`gui.rs:544-546`](../../latent-app/src/gui.rs)) is gone, replaced by a `ViewTransform` call (this deletion is the rule-3 proof).
  - The brush (C7) and at least one handle tool (C2) drive their begin/commit through the framework's per-drag gesture — one undo step per drag, verified by an undo after a crop-corner drag restoring the prior crop.
  - Pointer precedence is defined: an active tool's handle consumes the drag, otherwise it falls through to pan; `None` is pure pan/zoom. The render output is byte-identical (rule 2): the CPU/GPU preview tests and `to_color_image_matches_the_export_transform` ([`gui.rs:1241-1257`](../../latent-app/src/gui.rs)) are untouched.
  - The interaction logic that *can* be unit-tested without a window (hit-test tolerance in normalized space, the grabbed-handle state machine, clamping) carries tests; the egui-paint/drag surface is documented as display-unverifiable, matching the curve editor's stance ([`gui.rs:633-634`](../../latent-app/src/gui.rs)). `cargo fmt --check`, `cargo clippy --workspace --all-targets` (0 warnings), `cargo test --workspace` stay green (rule 1).

### U2-C2 — Crop tool
- Implements: replace the four crop sliders with a direct crop rectangle (the headline spatial-edit jump)            Priority: High
- Crates/files: `latent-app/src/gui/canvas.rs` (the crop overlay + handles), `latent-app/src/gui/panels/…` (the crop section: aspect presets, lock, numeric entry)
- Depends on: **U2-C1**            Blocks: —
- Heads-up:
  - **A draggable crop rectangle replaces `crop_block`'s four sliders ([`gui.rs:902-930`](../../latent-app/src/gui.rs)) as the primary UI.** The rectangle is `geometry.crop` ([`Crop {x,y,width,height}`, `latent-edit:678-684`](../../latent-edit/src/lib.rs)) drawn in normalized space and mapped to screen by the `ViewTransform`. `None` is the full frame (the `unwrap_or(Crop{0,0,1,1})` the slider block uses at [`:903-908`](../../latent-app/src/gui.rs) is the same convention — keep it: a crop equal to the full frame can normalize back to `None`).
  - **8 handles + interior move.** Four corners + four edge midpoints (8 grab handles via C1's hit-test), plus an interior drag that translates the whole rectangle. A corner drag moves two edges; an edge-midpoint drag moves one edge; an interior drag moves `x`/`y` keeping `width`/`height`. Each is one begin/commit gesture (C1). **Clamp to `[0, 1]`** on every edit (the crop is normalized over the oriented image; `crop_image` ([`pipeline:1499-1506`](../../latent-pipeline/src/lib.rs)) already clamps and `.round()`s to pixels, but clamp in the UI so the handles can't be dragged off-frame). Enforce a minimum size (don't let width/height collapse to 0 — `crop_image` does `.max(1.0)` px, but the UI should stop at a small normalized floor so the rect stays grabbable).
  - **Aspect-ratio presets + lock.** A dropdown/segmented control (U3 widgets) with **Free / Original / 1:1 / 3:2 / 4:3 / 16:9**, and a lock toggle. *Original* is the oriented-image aspect (from U1's displayed size — note the image is a non-square pixel grid, so an aspect ratio in normalized `[0,1]²` space must be multiplied by the displayed `width/height`; a 1:1 *visual* square is **not** `width==height` in normalized units — convert through the displayed aspect, which the `ViewTransform`/U1 size provides). When locked, a corner/edge drag adjusts the free dimension to keep the ratio (this is the **gotcha**: respect the lock while dragging — recompute the constrained dimension each move, anchored at the opposite corner/edge so the rectangle grows from the side the user is *not* dragging). Free mode is unconstrained. Switching to a ratio while a crop exists re-fits the current rectangle to that ratio (centered, clamped).
  - **Overlays: rule-of-thirds + dim outside.** Draw a rule-of-thirds grid inside the crop rect (two vertical + two horizontal lines at the thirds), optionally a finer grid toggle, and **dim the area outside** the crop (a translucent dark fill over the four border regions, or a single dark overlay with the crop punched out) so the kept region reads clearly. All painted in screen space via the transform; pure overlay, no pixel change.
  - **Interaction with straighten/keystone (gotcha).** Crop lives in the **same normalized space** as the rest of geometry but is applied **last** — `apply_geometry` does vignetting → straighten+keystone homography → lens warp → **then** `crop_image` on the result ([`pipeline:1446-1448`](../../latent-pipeline/src/lib.rs)). So the crop rectangle is in the coordinates of the *straightened/keystoned* output, not the raw source. For C2 alone (straighten/keystone may be zero), drawing the crop in the displayed (oriented, post-geometry) image space the `ViewTransform` already represents is correct — the displayed preview *is* the post-geometry image. Just be aware the crop fractions are relative to the geometry-corrected frame; don't try to pre-rotate them. (Lightroom-style "crop overlay rotates with straighten" is out of scope; the crop stays axis-aligned to the corrected output, matching the engine.)
  - **Numeric entry stays (epic rule).** Keep the four-field numeric crop (the `crop_block` values, possibly reworked into compact X/Y/W/H number entries via the U3-C2 slider/entry widget) alongside the handles, so an exact crop is still typable. The handles are primary; the numbers are the fallback.
  - **One undo per gesture, via C1.** A corner drag = `begin()` on press, write `geometry.crop = Some(Crop{…})` each move, `commit()` on release — exactly the `crop_block` gesture ([`:914-929`](../../latent-app/src/gui.rs)) but driven by the handle instead of four sliders.
- Acceptance:
  - A draggable crop rectangle with 8 edge/corner handles + interior move writes `geometry.crop` (and normalizes a full-frame rect back to `None`), clamped to `[0, 1]` with a minimum size; it replaces the four `crop_block` sliders as the primary UI while a numeric entry remains.
  - Aspect presets **Free/Original/1:1/3:2/4:3/16:9** with a lock work: with the lock on, a drag holds the ratio (anchored at the opposite corner/edge), and the ratio is computed through the displayed image aspect (a 1:1 visual square is not `width==height` in normalized units) — a **test** on the aspect-constrain helper (pure math: given a drag delta, an anchor, and a target ratio + displayed aspect, returns the constrained rect) pins this and the corner-anchoring.
  - A rule-of-thirds overlay (and the optional grid toggle) draws inside the rect and the area outside the crop is dimmed; both are pure egui paint (no pixel/render change — rule 2).
  - A crop-corner drag is one undo step (C1 begin/commit), and undo restores the prior crop. Clamp/normalize/min-size helpers carry unit tests; `cargo fmt --check`, clippy (0 warnings), `cargo test --workspace` stay green.

### U2-C3 — Straighten-by-horizon + manual rotate/flip
- Implements: drag-to-level straighten + discrete rotate 90°/flip; the user-driven complement to U1-C1's auto EXIF orientation. **The only U2 card permitted to touch pipeline math, and only additively (rule 2).**            Priority: High
- Crates/files: `latent-edit/src/lib.rs` (add a manual-orientation field to `Geometry`, serde-default; update `is_identity`), `latent-pipeline/src/lib.rs` (apply it in `apply_geometry`, folded into the existing resample), `latent-app/src/gui/canvas.rs` (the horizon-line tool) + toolbar (rotate/flip buttons)
- Depends on: **U2-C1**, **U1-C1** (EXIF auto-orientation is the base orientation this composes on top of)            Blocks: —
- Heads-up:
  - **Two features, one card: drag-a-line to set the straighten angle, and discrete rotate/flip via a new `Geometry` field.**
  - **Straighten by horizon (writes the existing `straighten_degrees`).** With the Straighten tool active (C1), the user drags a line across the image along a reference horizon/vertical; on release, the angle that makes that line level (or plumb) is written to `geometry.straighten_degrees` ([`latent-edit:540`](../../latent-edit/src/lib.rs)) — `auto-level`. Compute the angle from the two normalized endpoints **after** mapping them through the `ViewTransform` to a consistent space (or compute in normalized space but **correct for the displayed aspect** — a line's screen angle depends on the pixel aspect, so `atan2(dy·H, dx·W)` using the displayed `width/height`, not raw normalized deltas, or the horizon won't actually level). Positive `straighten_degrees` is counter-clockwise ([`latent-edit:540`](../../latent-edit/src/lib.rs); the engine rotates via `Transform::rotation(extent, degrees.to_radians())` at [`pipeline:1378-1379`](../../latent-pipeline/src/lib.rs)) — match that sign so dragging a tilted horizon flat produces the correct rotation direction. This **writes the field the existing slider writes** ([`straighten_slider:835-849`](../../latent-app/src/gui.rs)); the slider stays as numeric entry.
  - **Manual rotate 90° / flip — the new `Geometry` field (the additive pipeline change).** `Geometry` has **no** manual orientation today ([`:534-559`](../../latent-edit/src/lib.rs)) — only the continuous `straighten_degrees`. Add a field for discrete orientation, e.g. `pub orientation: Orientation` (or a packed `rotate90: i8` + `flip_h: bool`/`flip_v: bool`) with **`#[serde(default)]`** (rule 4 — old `.ron` sidecars must still load; `Geometry` is already `#[serde(default)]` at [`:535`](../../latent-edit/src/lib.rs), so a defaulted field deserializes from an old file as the no-op). The 8 orientations form the dihedral group (the same space as EXIF's `sizes.flip` that U1-C1 reads); model it so rotate-CW/CCW and flip-H/V compose correctly (e.g. an enum of the 8 D4 states with `rotate_cw`/`rotate_ccw`/`flip_h`/`flip_v` transition methods, *or* `rotate90: i8` mod 4 plus a flip — pick the representation whose composition you can test). Default = identity (no rotation, no flip). **Toolbar buttons** (U0-C5): Rotate 90° CW, Rotate 90° CCW, Flip Horizontal, Flip Vertical — each a discrete `begin()`/`commit()` (one undo step), like the discrete-click path in `gesture()` ([`:587`](../../latent-app/src/gui.rs)).
  - **Apply it in `apply_geometry`, folded into the one resample (gotchas: composition + single resample).** The manual orientation must compose with **both** the EXIF base orientation (U1-C1, applied in develop) **and** straighten. U1-C1 bakes EXIF orientation into the developed image up front, so by `apply_geometry` the image is already EXIF-oriented; the manual orientation is *additional* user rotation on top. Fold it into the **existing** straighten+keystone homography compose ([`pipeline:1378-1394`](../../latent-pipeline/src/lib.rs)) so there is **no extra resample pass** (rule 2 / the one-resample rule): a 90° rotate + flip is an exact axis-swap/sign-flip of the source mapping — express it as a `Transform` (its own `output` extent swaps width/height for an odd 90° turn, like `Transform::rotation`'s bounding-box expansion at [`pipeline:143-161`](../../latent-pipeline/src/lib.rs) but exact for multiples of 90°) and `compose` it into the homography that already feeds the single `backend.resample`/`backend.warp` call ([`:1427`/`:1438`](../../latent-pipeline/src/lib.rs)). Order matters: decide and document whether the manual rotate is applied before or after straighten (most natural: discrete orientation is the *base* the continuous straighten refines, i.e. orientation first, then straighten — so straighten always tilts within the chosen 90° framing). A 90° turn is lossless and axis-aligned, so the right-angle path can even short-circuit the resampler if you want, but folding it into the compose keeps it to the one existing interpolation and is the simpler invariant to hold. **A pure flip or 180° must not introduce any interpolation** — it's a coordinate remap; verify the composed map lands on exact pixel centers for the right-angle cases.
  - **Update `is_identity` (the no-op check).** `Geometry::is_identity` ([`:564-572`](../../latent-edit/src/lib.rs)) currently checks crop/straighten/perspective/lens/vignette/auto_scale/output_sharpen — **add the new orientation field** so a non-identity orientation isn't treated as a no-op (and a defaulted/identity orientation still reports identity). This is the same forward-compat discipline E3-C1 followed for `LensProfile` defaults — a fresh/default value must keep `is_identity` true.
  - **Test that rotate90 composes correctly (acceptance).** A pipeline test: rotating a known non-square asymmetric image 90° CW lands the expected corner pixel at the expected output position and the output extent swaps W↔H; two 90° CW turns equal one 180°; flip-H then flip-H is identity; the right-angle cases introduce no interpolation (a synthetic image with sharp pixel values comes back bit-exact, not blurred). Compose-with-straighten is covered by checking that orientation+straighten produces the same map as applying orientation then the straighten rotation. Model these on the existing `straighten_expands_the_canvas_and_keeps_the_center` ([`pipeline:2914`](../../latent-pipeline/src/lib.rs)) and `crop_reduces_dimensions_and_keeps_the_region` ([`:2883`](../../latent-pipeline/src/lib.rs)) geometry tests.
  - **Sanitize/forward-compat.** If you add an enum, derive `Default` to the identity variant and ensure `Settings::sanitize` ([`latent-edit:698+`](../../latent-edit/src/lib.rs), which already scrubs `straighten_degrees` at [`:851`](../../latent-edit/src/lib.rs)) handles the new field (an enum needs no float scrub; a packed `i8` should be taken mod 4). Old sidecars without the field deserialize to the identity (rule 4) — a serde-default round-trip test, like E7's data-model rule.
- Acceptance:
  - Dragging a horizon/vertical line sets `geometry.straighten_degrees` to auto-level it, with the angle computed through the displayed aspect (so the horizon actually ends up level on a non-square frame) and the sign matching `Transform::rotation`'s CCW-positive convention; the straighten slider remains as numeric entry.
  - `Geometry` gains a `#[serde(default)]` manual-orientation field (the 8 dihedral states), with toolbar Rotate 90° CW/CCW and Flip H/V each a single undo step; `is_identity` ([`latent-edit:564-572`](../../latent-edit/src/lib.rs)) includes it (default orientation ⇒ still identity), and an old sidecar without the field still deserializes (serde-default test).
  - `apply_geometry` applies the orientation **folded into the existing straighten/keystone homography compose** — no second resample pass (rule 2 / one-resample); a right-angle/flip case is bit-exact (no interpolation). A **pipeline test** `rotate90_composes_correctly` (or similar) asserts: 90° CW swaps the output extent and lands the expected corner; two CW turns = one 180°; flip∘flip = identity; orientation∘straighten matches the documented order. The CPU/GPU equivalence tests stay green (the additive map is mirrored on both backends or the right-angle case is backend-agnostic).
  - `cargo fmt --check`, clippy (0 warnings), `cargo test --workspace` stay green (rule 1). This is the only U2 card that changed pipeline math, and the change is additive (new field, folded into the existing resample) — no existing render output moved (the default render is byte-identical: `is_identity` short-circuits an all-default geometry exactly as before).

### U2-C4 — Keystone corner handles
- Implements: replace the two keystone sliders with draggable corner/guide handles            Priority: Medium
- Crates/files: `latent-app/src/gui/canvas.rs` (the keystone overlay + handles)
- Depends on: **U2-C1**            Blocks: —
