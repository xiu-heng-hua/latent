# U3 — Panel & control redesign

> Depends on **U0** (the `gui.rs` module split — every file below is a module U0-C1 carves out — plus the theme/spacing tokens U0-C2 and the typography U0-C3 the new widgets render with). Turns the right `SidePanel::right("controls")` — today one long, non-collapsible column of `ui.heading()` sections separated by `ui.separator()` (`gui.rs:407-493`) — into an organized, scrollable, resettable, tooltip'd panel built from a small set of **reusable widgets**.
>
> **Shared surface:** two things, and the whole epic guards them. (1) A new **widgets module** (`gui/widgets.rs`) that owns one `adjust_slider`/`opt_adjust_slider` (U3-C2) replacing the four hand-rolled slider helpers (`opt_point_slider:593`, `value_slider:934`) and the seven bespoke multi-slider blocks (`tone_block:746`, `white_balance_block:727`, `sharpen_block:773`, `clarity_block:792`, `noise_reduction_block:811`, `keystone_block:871`, `crop_block:902`, plus the per-shape sliders in `local_shape_block:1134`). (2) A **collapsible panel layout** (`gui/panels/adjustments.rs`, U3-C1) that hosts those widgets in remembered-open/closed sections inside a `ScrollArea`. Tooltips (U3-C4) and per-section reset / modified-dots (U3-C5) decorate both. **U6's new panels (HSL, channel mixer, full locals, lens, WB) slot in as sections built from these same widgets** — this epic is the reusable substrate U6 consumes; do not let U6-specific controls leak in here.
>
> **The one invariant every card preserves: the begin/commit gesture-to-history contract.** Today each helper runs the same dance — `gesture(&[&responses])` derives `(begin, commit, changed)` from the egui `Response`s (`gui.rs:582-589`), then `history.begin()` on begin, `set(...)` on changed, `history.commit()` on commit — so **one drag (or one discrete click-set) is exactly one undo step**, and a net-zero gesture records nothing (`history.rs:62-87`). The new `adjust_slider` must reproduce this *byte-for-byte* in behavior: same `gesture` derivation, same begin-once/commit-once, same `current_mut()` write only on change. Reset and numeric-entry are **new gesture sources** that each wrap a `begin()…set…commit()` into one step. `is_idle()` must still be false mid-drag so `autosave` (`gui.rs:228-247`) saves once per completed edit, not per frame.
>
> **Cross-refs (consume, don't reimplement):** panel/section open-state and side-panel width **persist via U4-C6** (the app config) — this epic stores/reads through that, it does not build a config writer. The shortcuts cheat-sheet (U3-C4) is **populated by U1/U2/U7** as they add bindings; U7-C5 owns the complete shortcut set. The global "reset all develop" lives in **U7-C3** (copy/paste + reset-all); U3-C5 cross-refs it for the panel button, it does not own the clipboard. No card here touches pipeline/render math — **Global rule #2 (the render output is sacred)** holds trivially: these are input widgets, the rendered pixels are unchanged.

### U3-C2 — Reusable adjust-slider widget (double-click reset, numeric entry, fine drag, neutral marker)
- Implements: drastic-UI control consistency                Priority: High
- Crates/files: `gui/widgets.rs` (new `adjust_slider` / `opt_adjust_slider`, replacing `gui.rs:593-615` `opt_point_slider` and `gui.rs:934-955` `value_slider`, and the hand-rolled blocks `gui.rs:727-930` / `1134-1227`).
- Depends on: U0-C2 (spacing/accent tokens the widget renders with)                Blocks: U3-C1 (it consumes the widget), U3-C4 (tooltips attach here), U3-C5 (modified-dot + reset reuse the default knowledge), **U6** (all new panels build on it)
- Heads-up:
  - **This is the load-bearing widget of the epic. Its single hard constraint: keep the `gesture`/begin/commit history wiring *exactly* as `opt_point_slider`/`value_slider` have it today, and add reset/entry/marker as new, equally-correct one-undo-step gestures.** Get this contract right and every section inherits it; get it wrong and undo granularity breaks panel-wide.
  - **The history contract, restated precisely (do not deviate):** for a drag, derive `let (begin, commit, changed) = gesture(&[&response])` from the slider's `egui::Response` exactly as `gui.rs:604` does, then `if begin { history.begin() }`, `if changed { set(history.current_mut(), value) }`, `if commit { history.commit() }`. The `gesture` helper (`gui.rs:582-589`) already encodes both paths — a continuous drag (`drag_started`→`drag_stopped`, one transaction) **and** the discrete click-set (`changed && !dragged && !started && !stopped` ⇒ begin *and* commit on the same frame). Reuse `gesture` verbatim; do not re-derive begin/commit. A net-zero drag records nothing because `History::commit` compares `prev != current` (`history.rs:78-80`) — preserve that by writing through `current_mut()` only on `changed`, never unconditionally.
  - **Two flavors, mirroring today's two helpers.** `opt_adjust_slider` carries `neutral: f32` + `get(&Settings)->Option<f32>` + `set(&mut Settings, Option<f32>)`, showing `neutral` when the field is `None` and setting `Some(value)` on change (the `opt_point_slider:602-609` shape; note the variants where `None`-clearing on a sentinel matters — `vignette_slider:861` clears to `None` at 0, `keystone_block:887` at both-zero — expose that via the `set` closure, the widget stays generic). `adjust_slider` carries a plain `get->f32`/`set(f32)` (the `value_slider` shape, for `opacity`, `straighten_degrees`). Keep the **closure-based get/set** signature both helpers already use — it is what lets one widget drive a global field, a `locals[i]` field, or a geometry field without the widget knowing the path.
  - **The multi-field blocks fold in incrementally, preserving their shared gesture.** `tone_block`/`white_balance_block`/`sharpen_block`/`noise_reduction_block`/`keystone_block`/`crop_block` each build **one** transaction over *several* sliders by passing all their `Response`s into a single `gesture(&[&r0,&r1,…])` (`gui.rs:732`, `:754`, `:817`, `:914`) — so dragging contrast then highlights inside one block is still bracketed correctly. Preserve this: `adjust_slider` must return its `Response` (or expose a "collect responses, then one begin/commit at block end" form) so a block of N adjust-sliders can still commit as **one** gesture, exactly as the `unwrap_or_default()`→edit-locals→`Some(struct{…})` blocks do today. Do **not** silently turn one four-slider block into four separate undo steps — that is a behavior regression. (One workable shape: `adjust_slider` does the per-slider `get`/`set` but a caller-supplied `GestureScope` accumulates responses and runs the single begin/commit; document which blocks share a scope.)
  - **New gesture sources — each is one undo step:**
    - *Double-click-to-reset-to-default.* On the slider's `Response`, `if r.double_clicked() { history.begin(); set(current_mut(), default); history.commit(); dirty=true }`. "Default" = the field's neutral: for `opt_adjust_slider` that is `set(None)` (Global rule: defaults are neutral, `Default`/`None`); for `adjust_slider` it is the field's `Default` value (e.g. `straighten_degrees` → `0.0`, `opacity` → its default). A double-click that hits an already-default value nets no change, so `commit`'s `prev != current` check records nothing — correct, free.
    - *Numeric entry (`DragValue`).* Add an optional `egui::DragValue` next to the slider for precise typing. Its `Response.changed()` flows through the **same** `gesture`/begin/commit (a typed-and-committed edit is one undo step — egui fires `changed` on commit of the text edit). Wire the `DragValue` `Response` into the same `gesture(&[&slider, &drag_value])` call so slider-then-type inside one interaction stays one step; a standalone typed edit is the discrete-click path (begin+commit same frame).
    - *Fine drag.* egui `Slider` already slows with a modifier; expose `.drag_value_speed`/the slider's fine-step so a held modifier (note which — likely the egui default) gives sub-step precision. No history change — it is still one drag = one step.
    - *Neutral-value marker.* Draw a subtle tick at the neutral/default position on the slider track (so the user sees where "off" is). Pure painting via `egui::Painter` over the slider rect (U0-C2 accent token), **no** interaction, **no** history effect.
  - **Testability (the acceptance's careful wording):** the *gesture* is display-driven — it reads live `egui::Response` flags that only exist under a running window — so it is **not** unit-testable headless (state this in a doc-comment, as `curves_block:633-634` already notes for its drag). What **is** unit-testable, and what the card asks for, is the **reset/neutral pure logic**: a free function like `reset_value(neutral_or_default) -> NewValue` and the `Option` `None`-vs-`Some(neutral)` mapping. Factor those out of the egui path and test them directly (`double_click_resets_to_default`, `opt_neutral_maps_to_none`), the way the codebase factors `export_status`/`clamp_selection` out of the UI to test them (`gui.rs:206`, `:216`).
- Acceptance:
  - One `adjust_slider` + `opt_adjust_slider` in `gui/widgets.rs` drives a slider with **identical** begin/commit/changed wiring to `opt_point_slider`/`value_slider` (same `gesture` call, same one-undo-step-per-drag, same net-zero-records-nothing). At least the four single-field helpers and two of the multi-field blocks (e.g. `tone_block`, `white_balance_block`) are migrated to it; the rest follow incrementally.
  - **Double-click resets to default in one undo step:** a **test** `double_click_resets_to_default` on the extracted pure reset logic asserts an `opt` field resets to `None` and a plain field to its `Default`; the gesture wiring (begin/commit bracketing the reset) is documented as display-driven and not headless-tested.
  - **Numeric entry edits commit one undo step:** the `DragValue` `Response` is threaded into the same `gesture` so a typed edit is one transaction (documented; behavior-checked where the harness allows). Multi-field blocks that shared one gesture still commit as **one** step (no regression to N steps) — a structural note/assert in the migrated block.
  - The neutral marker renders at the default position and is non-interactive; fine-drag works under the modifier. `cargo fmt --check`, clippy (0 warnings), `cargo test --workspace` green.

## Epic done when

The right control panel is no longer one clipping, non-collapsible column but an organized, scrollable stack of **CollapsingHeader** sections — Basic / Tone / Color / Curves / Detail / Effects / Geometry / Masks (Export moves to a dialog in U4-C4) — each remembering its open/closed state across restarts through the app config (U3-C1 / U4-C6, keyed by a stable `SectionId`). Every control is one reusable `adjust_slider`/`opt_adjust_slider` (U3-C2) that **preserves the begin/commit gesture-to-history contract exactly** — one drag or one discrete click-set is one undo step, a net-zero gesture records nothing, multi-field blocks still commit as a single transaction — while adding double-click-to-reset, a `DragValue` numeric entry that commits one undo step, fine-drag, and a neutral-value marker, with the reset/neutral logic factored out and unit-tested (the gesture itself documented as display-driven). The panel is user-resizable with its width persisted and a `Tab` toggle to hide it for a full-screen canvas (U3-C3). Every control and section header carries a concise range-stating tooltip, and a `?`-opened modal lists the current keyboard shortcuts from a shared registry that U1/U2/U7 append to (U3-C4, complete set owned by U7-C5). Each section has a one-undo-step reset (and a global reset deferring to U7-C3), and a subtle modified-dot — derived purely from the current `Settings`, never a stored flag — marks any control or collapsed section that differs from default (U3-C5). No rendered pixel and no slider value/range changed; the widgets module and the collapsible panel are left as the reusable substrate **U6** builds its HSL, channel-mixer, full-local, multi-mask, lens, and white-balance panels on.
