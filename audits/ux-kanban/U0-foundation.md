# U0 — Visual foundation & app shell

> Foundational epic. Every other UI epic (U1–U8) lands on the two surfaces this epic creates: the **`latent-app/src/gui/` module tree** (C1) and the **`theme` module** — design tokens, `Visuals`, fonts, icons — that all panels and widgets reuse (C2–C4). Today the entire GUI is one ~1340-line file, `latent-app/src/gui.rs`, on default egui look-and-feel: no menu bar, no status bar, the image floats on egui's default panel color with a faint blue accent (`gui.rs:716` `Color32::LIGHT_BLUE`). C1 carves that monolith into a home for later work *without changing a pixel*; C2 lays a tuned dark theme over it with a **neutral mid-gray (~18%) canvas surround** (locked decision: a photo editor's working surface must have **no color cast**, or it biases every color judgement the user makes); C3/C4 embed UI/mono/icon fonts as OFL TTFs; C5 frames the window with a menu bar / toolbar / status bar that later epics wire; C6 sets sane window and high-DPI defaults.
>
> **Shared surface:** the new `gui/` tree (`mod.rs`, `app.rs`, `state.rs`, `theme.rs`, `widgets.rs`, `icons.rs`, `panels/…`, `canvas.rs`, `dialogs.rs`) and `theme.rs`'s token constants are imported by **U1–U8**. Take the cards in order: **C1** is a pure mechanical move (behavior-preserving, the 5 unit tests move with their code and stay byte-identical); **C2** introduces the token module + `Visuals` everything themes against; **C3** adds typography to that theme; **C4** adds the icon font + helper; **C5** builds the shell panels on top of C2–C4; **C6** finishes the window chrome.
>
> **The no-pixel-change rule (Global rule 2, load-bearing here):** the rendered *image* — `to_color_image` → `latent_export::to_srgb8` and the `latent_pipeline::render` math — stays byte-identical through this whole epic. A *theme* changes egui chrome (panels, sliders, text, the surround **around** the image), never the developed pixels inside the `egui::Image`. The `to_color_image_matches_the_export_transform` test is the tripwire and must stay green unchanged. No card here touches backend/shader/export math.
>
> **Dependency deps (read before starting):**
> - `main.rs:3` declares `mod gui;` and `main.rs:114` calls `gui::run(&input, select_backend(gpu))`. C1 must keep `gui::run` exported with the **same signature** (`pub fn run(input: &Path, backend: Box<dyn Backend>)`), or `main.rs` stops compiling. `main.rs:84` also exposes `pub fn develop_to_image`, which `gui.rs:29` calls as `crate::develop_to_image` — that crate-root path must survive the move (it's a sibling of the `gui` module, not inside it).
> - egui/eframe are pinned at **0.32.3** (`Cargo.lock`). Theme/fonts/icons/viewport use that version's APIs: `egui::Visuals`, `ctx.set_visuals`/`set_style`, `egui::FontDefinitions` + `FontData::from_static`, `egui::FontFamily::{Proportional, Monospace, Name}`, `egui::TextStyle`, `eframe::NativeOptions` + `egui::ViewportBuilder`.
> - The workspace license is **MIT** (`Cargo.toml` `[workspace.package] license = "MIT"`). Embedded fonts must be **OFL/MIT/Apache-2.0** and their license files committed beside them (C3/C4) — MIT app + OFL fonts is fine, but the OFL text must ship.

### U0-C2 — Theme & design tokens (tuned dark `Visuals` + neutral-gray canvas surround)
- Implements: board "themed shell"; the no-color-cast canvas decision            Priority: High
- Crates/files: new `latent-app/src/gui/theme.rs` (token constants + `fn apply(ctx: &egui::Context)`); call site in `gui/app.rs` (`run`/first-frame setup); the canvas surround in `gui/canvas.rs` (the CentralPanel `frame`).
- Depends on: U0-C1            Blocks: U0-C3, U0-C4, U0-C5; U3 (panels theme against these tokens).
- Heads-up:
  - **Build the single source of truth for the app's look here. Every later UI card must read spacing/rounding/accent/gray from `theme`'s constants, never hard-code a `Color32`/`Rounding`/magnitude inline. Today the code is full of ad-hoc literals — `Color32::from_gray(80)` (`gui.rs:706`), `Color32::LIGHT_BLUE` (`:717`), `Color32::WHITE` (`:720`), the `220.0`/`160.0` curve-editor sizes (`:672`), `0.08`/`0.04` brush defaults (`gui.rs:74-75`). C2 doesn't have to migrate all of them, but it must *define the tokens* so C3–C6 and U1–U8 stop inventing literals.**
  - **A tuned dark `Visuals`, applied once at startup.** Start from `egui::Visuals::dark()` and override: neutral gray ramp (no blue tint in `window_fill`, `panel_fill`, `faint_bg_color`, `extreme_bg_color`), a **single accent** for `selection.bg_fill` / `hyperlink_color` / active widget strokes (replace the stray `LIGHT_BLUE`), consistent widget rounding (`Visuals::widgets.{noninteractive,inactive,hovered,active,open}.corner_radius` — egui 0.32 renamed `rounding`→`corner_radius`; confirm against the pinned 0.32.3 API), consistent `bg_stroke`/`fg_stroke` widths, and a calm `selection.stroke`. Apply via `ctx.set_visuals(...)` (and `ctx.set_style(...)` for spacing) **once** — in the `eframe::run_native` creation closure (`gui.rs:60-80`, the `|_cc|` where `App` is built) or the first `update`, guarded so it runs once. The `cc.egui_ctx` is available in that closure — prefer setting it there over a per-frame check.
  - **The canvas surround is neutral ~18% gray — the locked, called-out decision.** A RAW editor's working area must surround the image with a **color-neutral mid-gray** so the surround doesn't push the eye's white balance and bias every color/exposure call the user makes. egui's default `CentralPanel` paints `panel_fill` (a cool dark). Give the CentralPanel an explicit `egui::Frame` whose `fill` is a neutral gray at ~18% luminance — i.e. `Color32::from_gray(n)` with **equal R=G=B** (no cast), `n` chosen so the *displayed* (already sRGB-encoded chrome) reads ~18% — roughly `from_gray(118)` for a perceptual mid, but pick by eye against a neutral test image and pin the constant in `theme.rs` as e.g. `CANVAS_SURROUND`. Document *why* equal-channel gray matters in the constant's doc-comment. This frame wraps the `egui::Image` in `gui/canvas.rs` (`gui.rs:523` `CentralPanel::default().show`) — it changes the area *around* the photo, **never the photo's own pixels** (the `Image` still draws `to_color_image`'s bytes unaltered; `to_color_image_matches_the_export_transform` stays green).
  - **Spacing/size constants live here too:** define `item_spacing`, panel inner margins, default side-panel width, and the curve-editor size (`220.0×160.0` at `gui.rs:672`) as named constants so U3's panel redesign and U1's viewport read one source. Set `Style::spacing` from them in `apply`.
  - **No new crate, no font work** (C3 owns fonts). `theme::apply` is fonts-agnostic in this card; C3 extends it to also register `FontDefinitions`.
  - **Cross-cutting / test:** a `Visuals` swap is display-only and largely unverifiable by unit test, but two things *are* checkable and should be pinned: (1) the canvas-surround constant is a true neutral — a **test** `canvas_surround_is_neutral_gray` asserts `CANVAS_SURROUND.r() == .g() && .g() == .b()` (catches an accidental cast on a future edit) and that its gray is in the mid range (e.g. `90..=150`); (2) `to_color_image_matches_the_export_transform` still passes (the theme didn't leak into pixel math). Keep `apply` idempotent.
- Acceptance:
  - A `theme` module exists as the **single source** for the accent color, the neutral gray ramp, rounding/stroke, spacing, and the `CANVAS_SURROUND` gray; nothing in `gui/` reintroduces an inline `LIGHT_BLUE`/ad-hoc panel color for chrome (the curve editor's accent now reads `theme::ACCENT`).
  - `theme::apply(ctx)` runs **once** at startup (set in the `run_native` closure or first-frame-guarded), setting `Visuals` + `Style::spacing`; the window renders dark, neutral, single-accent.
  - The CentralPanel wraps its image in a frame filled with the neutral ~18% gray surround (no color cast); the photo itself is unchanged.
  - **Tests:** `canvas_surround_is_neutral_gray` passes (equal channels, mid-range); `to_color_image_matches_the_export_transform` still passes unchanged. Baseline green (fmt/clippy zero-warnings/test, **313+** the one new test).

### U0-C3 — Typography (embed a UI sans + a mono, register `TextStyles`)
- Implements: board "themed shell" (legible, intentional type)            Priority: Medium
- Crates/files: `latent-app/src/gui/theme.rs` (extend `apply` to register fonts + text styles); new `latent-app/assets/fonts/` (the `.ttf` files + their license texts), loaded with `include_bytes!`.
- Depends on: U0-C2            Blocks: U0-C4 (icon font registers alongside these), U0-C5 (status-bar numeric readouts use the mono style).
- Heads-up:
  - **Embed the fonts as TTFs via `include_bytes!` + `FontDefinitions` — NO new font crate (decision 2 / Global rule 6). The fonts are committed binary assets under `latent-app/assets/`, not a dependency.** This keeps the dep list at "rfd only" and makes the build hermetic.
  - **Two faces:** a clean UI sans for body/headings (Inter is the suggested face — SIL OFL 1.1), and a monospace for *numeric readouts* (slider values, zoom %, RGB loupe in U1-C5, histogram counts in U5) where tabular/aligned digits matter — e.g. JetBrains Mono (OFL) or IBM Plex Mono (OFL). Pick whatever you can ship the OFL text for. A photo editor's numbers shifting width as digits change is the specific ugliness the mono fixes.
  - **Wiring (egui 0.32.3 API):** load each face into `egui::FontData::from_static(include_bytes!("../../assets/fonts/Inter-Regular.ttf"))`, insert into a `FontDefinitions`'s `font_data`, then prepend the sans to `families[&FontFamily::Proportional]` and the mono to `families[&FontFamily::Monospace]` (prepend so they take precedence over egui's built-ins but the built-ins remain as glyph fallback — important so missing glyphs don't tofu). Call `ctx.set_fonts(defs)` inside `theme::apply` (after/with the `Visuals` from C2 — one startup pass). Then set `Style::text_styles` (`ctx.style_mut` or the `Style` built in `apply`) for `TextStyle::{Heading, Body, Monospace, Small, Button}` to sane sizes (e.g. heading ~18, body ~14, small ~12, mono ~13) using the `FontFamily` each should resolve through — headings/body/small/button → `Proportional`, monospace → `Monospace`. egui doesn't expose line-height directly; size + the default row spacing is the lever — set `spacing.interact_size`/row height in C2's spacing block if rows feel cramped.
  - **License obligation (the gotcha):** the app is MIT but OFL fonts must ship their license. Commit `OFL.txt` (and any `AUTHORS`/`README`) **beside each font** in `latent-app/assets/fonts/`, and add a one-line attribution in the repo (e.g. a `latent-app/assets/fonts/README.md` listing each font + its license). OFL permits embedding/redistribution; it forbids selling the font alone and requires the license to travel with the font — committing `OFL.txt` satisfies it. Do **not** rename the font files in a way OFL's "Reserved Font Name" clause restricts. Verify each face's license is genuinely OFL/MIT/Apache before committing the bytes.
  - **No-pixel-change check:** fonts are chrome — they never touch `to_color_image`/render. The image tripwire test stays green. Keep the built-in font fallback so non-Latin paths in the title (`gui.rs:34`, the window title is a file path) still render.
  - **Cross-cutting:** `include_bytes!` makes the binary larger by the font sizes (~hundreds of KB) — acceptable for a desktop app and the price of zero font deps; note it in the commit. Pin the exact font version you commit.
- Acceptance:
  - `latent-app/assets/fonts/` holds the committed `.ttf`(s) for a UI sans + a mono, **each with its `OFL.txt`/license text** beside it and a short attribution note; the licenses are OFL/MIT/Apache.
  - `theme::apply` registers both faces in `FontDefinitions` (sans → `Proportional`, mono → `Monospace`, built-ins kept as fallback) via `include_bytes!`/`from_static`, and sets `TextStyle` sizes for heading/body/small/button/monospace.
  - The app renders in the embedded faces (visual check); numeric values intended for the mono style use `TextStyle::Monospace`. No new crate added (Cargo.toml deps unchanged but for nothing — fonts are assets).
  - Baseline green; `to_color_image_matches_the_export_transform` unchanged. (Typography is display-unverifiable by unit test; the binding checks are the committed license files and the no-new-dep rule.)

### U0-C4 — Icon set (embed an icon font + `icon()` / `icon_button` helper)
- Implements: board "themed shell" (toolbar/menu affordances)            Priority: Medium
- Crates/files: `latent-app/src/gui/icons.rs` (the helper + the codepoint name→glyph map) and `theme.rs` (register the icon font family); `latent-app/assets/fonts/` (the icon `.ttf` + its license).
- Depends on: U0-C3            Blocks: U0-C5 (the toolbar/menu use `icon_button`); U7 (toolbar actions reuse icons).
- Heads-up:
  - **Embed an icon font as an extra `FontFamily` — NO SVG crate, NO icon crate (decision 2). Icons are glyphs from one committed TTF, drawn as text in the accent/foreground color, scaling crisply with `pixels_per_point`.** This is the egui-idiomatic path and keeps deps flat.
  - **Face:** a permissively-licensed icon font — Lucide (ISC), Phosphor (MIT), or Material Symbols (Apache-2.0) as a TTF. Confirm the license is ISC/MIT/Apache and ship its text beside the file (same obligation as C3). Pick one whose glyph set covers the actions the shell needs: open, save/export, undo, redo, zoom-in/out/fit/100%, crop, brush, mask, histogram, settings, info — the U0-C5 toolbar and U1/U2/U7 toolbars draw from this set.
  - **Wiring (egui 0.32.3):** register the icon TTF under a **named** family — `FontFamily::Name("icons".into())` — in the same `FontDefinitions` C3 builds (`font_data` entry + `families.insert(Name("icons"), vec!["icon-font".into()])`). Do **not** prepend it to `Proportional`/`Monospace` (you don't want icon codepoints shadowing text). Then `icon(name) -> char`/`&str` maps a stable name (`"undo"`, `"zoom_in"`) to the font's private-use-area codepoint, and `icon_button(ui, name, tooltip) -> Response` builds a `egui::Button` whose label is a `RichText` in the `"icons"` family at the icon size, with `.on_hover_text(tooltip)`. Source codepoints from the font's own glyph manifest (Lucide/Phosphor publish a name→codepoint JSON).
  - **Gotcha — codepoints, not names, are what egui draws.** The icon font maps *glyphs to private-use codepoints*; the human name is yours. Get the mapping from the font's manifest and pin a test that the table is internally consistent (no two names share a codepoint, every codepoint is in the PUA range) so a future icon add can't silently collide.
  - **License:** commit the icon font's license text beside it; add it to the `assets/fonts/README.md` attribution from C3. ISC/MIT/Apache all allow embedding.
  - **No-pixel-change:** icons are chrome (toolbar/menu), never on the canvas image. Tripwire test stays green.
- Acceptance:
  - `latent-app/assets/fonts/` holds the icon `.ttf` + its license; `theme` registers it as `FontFamily::Name("icons")` in the same `FontDefinitions` pass (not shadowing text families).
  - `icons::icon(name)` resolves a stable set of names (the toolbar/menu set) to codepoints via one table; `icons::icon_button(ui, name, tooltip)` renders the glyph in the icons family with a hover tooltip and returns a clickable `Response`.
  - **Test** `icon_table_is_consistent`: every mapped codepoint is in the private-use range and no two names collide (guards future additions). Visual check: a sample `icon_button` renders the glyph, not tofu.
  - No SVG/icon crate added. Baseline green; image tripwire test unchanged.
