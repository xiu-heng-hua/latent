# U1 — Viewport: orientation, zoom, pan, before/after

> Depends on **U0** (the module split + shell): this epic lands on the `gui/` module tree U0 carves out of the 1340-line `latent-app/src/gui.rs`, and uses U0's status bar / toolbar surfaces. Cards write to `gui/canvas.rs`, `gui/app.rs`, `gui/panels/statusbar.rs` — the files U0-C1 establishes; if a card runs before its U0 file exists, create the module rather than re-inflating `gui.rs`.
>
> **The one rule this epic exists to enforce (README global rule 3): there is exactly one screen↔image coordinate mapping.** U1-C2 builds a `ViewTransform` (fit + zoom + pan) and is its *sole* owner; **U2** (crop/straighten/keystone/mask/brush handles), **U5-C2** (clipping overlay), and **U6-C4/C6** (mask handles, WB eyedropper) consume it and **never recompute** screen↔image math. Today that math is open-coded once, inside the brush block (`gui.rs:544-546`: `(pos.x - r.left()) / r.width()`), against the raw `Image` response rect — that is the *only* existing screen→image mapping, and C2 migrates it onto the transform so there is one place the mapping can ever be wrong.
>
> **Orientation comes first, by ordering, because every coordinate downstream is in display orientation.** U1-C1 reads LibRaw's `flip` and bakes the rotation/flip into the developed `ImageBuf` at the end of `develop_to_image` — so the texture, the fit, the brush normalized coords, the masks, and the crop all already operate on an *upright* image. C2's `ViewTransform` then maps screen↔**oriented**-image; masks/crop stay normalized `[0,1]` over that oriented image and are evaluated in SOURCE space exactly as today (README global rule 3). Take the cards in order: **C1** orients the pixels, **C2** fits them and owns the transform, **C3** adds zoom/pan to the same transform, **C4** caches a settings-off "before", **C5** reads a pixel back through the transform.
>
> **The render output stays byte-identical (README global rule 2).** This epic adds exactly one pipeline-adjacent change — orientation applied in `develop_to_image` (C1), which is display *geometry*, not sensor processing: it permutes/rotates pixels, it does **not** change a single pixel's RGB value. Everything else here is pure presentation (fit, zoom, pan, a cached before, a hover readout); `to_color_image` → `latent_export::to_srgb8` and `latent_pipeline::render` are untouched, the CPU↔GPU equivalence tests stay green, and the off-thread render worker (`RenderState`/`spawn`/`poll_render`, `gui.rs:153-329`) is preserved unchanged (README global rule 5).

### U1-C1 — EXIF auto-orientation (the portrait bug)
- Implements: Product decision 4 (orientation honored); *the reported portrait-sideways bug*            Priority: High
- Crates/files: `latent-raw/src/lib.rs` (`Metadata`, `read_metadata`), `latent-image/src/lib.rs` (a new `oriented`/rotate-flip helper), `latent-app/src/main.rs` (`develop_to_image` applies it).
- Depends on: —            Blocks: U1-C2 (the transform assumes an oriented image), all of U2 (canvas coords are in display orientation), U2-C3 (manual rotate composes *on top of* auto-orientation).
- Heads-up:
  - **This is the headline correctness fix of the epic. Portrait shots currently display sideways because LibRaw's orientation flag is read by nothing — `read_metadata` (`latent-raw:812-851`) copies black/white levels, CFA, color matrix, make/model/lens, but never touches `sizes.flip`. Develop hands the sensor-orientation pixels straight to the texture, so a camera held vertically shows landscape-rotated. Read the flag, apply the rotation/flip once at the end of develop, and the image is upright like every other viewer.**
  - **Where the flag lives.** The FFI struct `libraw_image_sizes_t` carries `pub flip: c_int` (bindings.rs:669, offset 32 in `(*raw).sizes`) — confirmed present in the generated bindings; it is simply never read. Add `orientation: u8` (or a small enum) to `Metadata` (`latent-raw:166-209`) and set it in `read_metadata` from `sizes.flip` (`sizes` is already bound at `:815`). Keep it a plain integer code on `Metadata` and decode it where it's applied, OR decode to an enum at read time — either is fine, but **pin the mapping in one place** so the 8-case table has a single home.
  - **The `flip` encoding is NOT the EXIF 1..8 numbering — this is the gotcha that makes naïve mappings wrong.** LibRaw stores `flip` in its own convention (the dcraw-derived code), where the common cases are `0`, `3`, `5`, `6` (these four cover the overwhelming majority of real files; the mirrored variants `1/2/4/7` exist but are rare). Document and implement the exact table:

    | LibRaw `flip` | meaning (how to make upright) | dims swap? |
    |---|---|---|
    | `0` | identity (already upright / landscape) | no |
    | `3` | rotate 180° | no |
    | `5` | rotate 90° **counter-clockwise** (CCW) — i.e. transpose + flip, "rotate CW 270") | **yes** (W↔H) |
    | `6` | rotate 90° **clockwise** (CW) | **yes** (W↔H) |
    | `1` | mirror horizontal | no |
    | `2` | mirror vertical | no |
    | `4` | transpose (mirror across main diagonal) | **yes** |
    | `7` | transverse (mirror across anti-diagonal) | **yes** |

    The four mirrored cases (`1/2/4/7`) are the transposed/flipped variants; implement them for completeness but the four that matter for the portrait bug are `0/3/5/6`. **A negative or out-of-range `flip` (LibRaw uses `-1` for "unknown" on some paths) maps to identity** — never panic, never rotate on a garbage code.
  - **Get the 5 vs 6 direction right — it is the easiest thing to invert and the test is what proves it.** LibRaw `flip == 6` is the portrait shot taken with the grip *down-right* (the common "rotate 90° CW to view upright"); `flip == 5` is grip *down-left* (90° CCW). If you flip these, every portrait comes out upside-down-and-mirrored-feeling instead of upright. The synthetic-image test (below) pins the direction concretely: a known corner pixel must land where a CW (resp. CCW) rotation puts it.
  - **The rotate/flip helper goes in `latent-image`, not the app.** `ImageBuf` has no rotate/transpose today (confirmed: `latent-image/src/lib.rs` exposes only `new`/`try_new`/`get`/`set`/`pixels`/`cropped`/`downscaled` — no orientation helper). Add one method, `oriented(&self, code: u8) -> ImageBuf` (or an `Orientation` enum arg), that allocates the destination at the correct dimensions (**swapping width/height for codes 4/5/6/7**) and copies each source pixel to its rotated/mirrored destination via `get`/`set`. It belongs in `latent-image` because (a) it is a pure pixel-geometry operation on `ImageBuf` with no app or RAW dependency, (b) U2-C3's manual rotate-90/flip wants the *same* primitive (compose manual orientation on top of auto), and (c) it is unit-testable in the crate that owns the buffer. Keep it allocation-clean (one fresh `ImageBuf` via `try_new` with the swapped dims; identity short-circuits to a `clone`, mirroring `downscaled`'s early-return style at `:166`).
  - **Apply it AFTER the color matrix, as the last step of develop.** In `develop_to_image` (`main.rs:84-94`) the chain is decode → `normalized` → `apply_white_balance` → `demosaic_mhc` → `reconstruct_highlights` → `color::apply_matrix`. Orientation is *display geometry*, not sensor processing: it must not sit before demosaic (the CFA phase is tied to the sensor grid) or before the color matrix (which is per-sensor-channel). Apply `oriented(meta.orientation)` to the final working `ImageBuf` right before the `Ok((…, meta))` return. From that point on, `full`, `preview = full.downscaled(...)`, the texture, brush coords, masks, and crop all operate in **display orientation** — which is exactly the invariant C2 and U2 assume. *(Note: `develop` the CLI command (`main.rs:100-104`) calls `develop_to_image` too, so the orientation fix also corrects CLI exports — a free win, and the right behavior.)*
  - **`pixel_aspect` is out of scope.** Some sensors set `sizes.pixel_aspect != 1.0` (anamorphic/non-square photosites). That is a separate squash/stretch concern; this card is rotation/flip only. Note it and leave it.
  - **Interaction with U2-C3 (manual rotate/flip).** Auto-orientation is the *base* — it makes the developed image upright. U2-C3's manual rotate-90°/flip is *additive on top*, a `Geometry` field applied in the pipeline's geometry stage (OUTPUT space), not here. The two must not double-apply: this card bakes the EXIF orientation into the develop base once; U2-C3 layers user intent over that already-upright base. Leave a one-line note at the apply site so U2-C3 composes rather than re-reads `flip`.
  - **Forward-compat (README global rule 4).** `Metadata` is not serialized to the sidecar (it is re-read from the RAW each open), so the new field needs no `#[serde(default)]` — but give it a sane default (`0`/identity) in the test constructor (`latent-raw:873-892` builds a literal `Metadata`; add the field there) so existing tests keep compiling.
- Acceptance:
  - `Metadata` carries an `orientation` (the LibRaw `flip` code), set in `read_metadata` from `sizes.flip`; the test-constructor `Metadata` literal includes it (default identity).
  - `ImageBuf::oriented` rotates/flips per the table: a **test** `oriented_rotates_each_flip_code` builds a small asymmetric image (e.g. 3×2 with a unique marker pixel at one corner) and asserts, for each of `0/3/5/6` (at minimum), that the marker lands at the expected destination corner and that **dimensions swap for `5`/`6`** (W↔H) but not for `0`/`3`. A second assertion pins `5` vs `6` as opposite rotation directions (CW vs CCW) so the direction can't silently invert.
  - `oriented(0)` (and any out-of-range/negative code) returns an image equal to the input (identity / no-op) — a **test** `oriented_identity_and_unknown_are_no_ops`.
  - `develop_to_image` applies orientation as its last step: a developed portrait base comes out with portrait dimensions (height > width) for a `flip == 6` fixture; landscape unchanged for `flip == 0`. (If a real RAW fixture is impractical, assert at the `oriented` seam — the develop wiring is a one-liner over the tested helper.)
  - `cargo test --workspace`, `cargo clippy --workspace --all-targets` (0 warnings), `cargo fmt --check` stay green; **no rendered RGB value changes** (orientation permutes pixels, it does not recolor them) — the CPU↔GPU and `to_color_image` tests are untouched and still pass.

### U1-C2 — Fit-to-window canvas + neutral surround + `ViewTransform`
- Implements: drastic-UI viewport goal; README global rule 3 (single coordinate transform)            Priority: High
- Crates/files: `gui/canvas.rs` (new; the central-panel draw + the `ViewTransform`). Migrates the brush pointer-mapping out of the inline `CentralPanel` block (`gui.rs:523-568`).
- Depends on: U0-C1 (the module split), U1-C1 (the image is oriented)            Blocks: U1-C3, U1-C4, U1-C5, all of U2, U5-C2, U6-C4, U6-C6.
- Heads-up:
  - **This card replaces the native-pixel blit with a fitted draw and introduces the single screen↔image transform the whole canvas stack is built on. It is the structural heart of the epic — get the transform's contract right and U2/U5/U6 inherit a correct mapping for free; get it wrong and every handle in U2 is subtly off.**
  - **What's there today.** The `CentralPanel` wraps the preview in `egui::ScrollArea::both()` and adds an `egui::Image::new(SizedTexture::new(tex_id, tex_size))` at the **native texture pixel size** (`gui.rs:523-528`) — no fit, no zoom, no pan; a 1600px-long preview just overflows into a scroll region. `tex_size = texture.size_vec2()` (`:521`) is the texture's pixel dimensions. The only screen→image mapping in the whole file is the brush block's open-coded `nx = (pos.x - r.left()) / r.width()`, `ny = (pos.y - r.top()) / r.height()` against the `Image` response rect (`:544-546`) — it works *only because* the image is drawn at 1:1 filling its own rect. Once we fit/letterbox, that assumption breaks, so the mapping must move onto the transform.
  - **The draw.** Replace the `ScrollArea` + native `Image` with: allocate the full available `CentralPanel` rect (`ui.available_rect_before_wrap()` / `ui.allocate_response` with `click_and_drag`), paint a **neutral-gray** fill across it (the photo surround — pull the gray from U0-C2's design tokens; a mid-neutral that doesn't tint the eye, not pure black), compute the fit, and draw the texture into the fitted sub-rect with `painter.image(tex_id, dest_rect, uv (0..1), Color32::WHITE)`. Letterbox = the gray showing around a portrait/landscape image in a differently-shaped panel. The image **fits** (preserve aspect, longest side bounded by the panel, centered) and **stays centered on resize** — recompute the fit from the live panel rect every frame; hold no pixel offsets in state, only the zoom/pan *intent* (C3 extends this).
  - **The `ViewTransform` — the contract.** A small value type built each frame from `(image_size_px, panel_rect, zoom, pan)`; in this card zoom is implicitly "fit" and pan is zero, but **define the full shape now** so C3 only fills in state, never changes the API:
    - `fit_scale` = `min(panel.w / img.w, panel.h / img.h)` (the letterbox-fit scale; the displayed scale is `fit_scale * zoom`, zoom = 1.0 here).
    - `offset` = the top-left of the image's drawn rect in screen space (centered: `panel.center - displayed_size/2`, plus pan in C3).
    - `image_norm_to_screen(norm: [f32;2]) -> Pos2` — normalized `[0,1]` over the oriented image → screen position. Used to *draw* handles/overlays (U2, U5).
    - `screen_to_image_norm(pos: Pos2) -> [f32;2]` — screen → normalized `[0,1]`, the inverse. Used to *read* the pointer (brush, eyedropper, handle drags). Clamp to `[0,1]` at the call site that needs it (brush does), but the transform itself returns the raw mapping so callers outside the image can detect "off-image".
    - Keep it pure and `Copy` — no egui `Ui` inside, just rects and scalars — so it is unit-testable and so U2 can stash it per-frame and hit-test against it.
  - **Migrate the brush onto it (the acceptance pin).** The brush block computes `nx/ny` from `resp.rect` (`gui.rs:544-546`); replace that with `transform.screen_to_image_norm(pos)` (then the existing `.clamp(0.0, 1.0)`). The dab still records normalized `[0,1]` coords (it must — masks are normalized over the oriented image, README global rule 3), so nothing downstream of the dab changes; only the *source* of `nx/ny` moves from the raw rect to the transform. This is the proof the transform is the one true mapping: the existing brush painting must still work, unchanged in behavior, through it.
  - **Texture vs source dimensions.** The transform's `image_size` is the **texture** size (`texture.size_vec2()`), which for the preview is the downscaled 1600px image — and that is correct, because normalized `[0,1]` is resolution-independent: the same `[0,1]` maps to preview, full-res export, and the source-space mask evaluation alike. Do not mix in `full`'s dimensions here.
  - **Don't regress the off-thread render.** This card only changes how the *existing* texture is drawn; the "Rendering…" placeholder path (`gui.rs:513-518`) and `poll_render`/`load_texture` stay exactly as-is (README global rule 5). The fit is computed from the texture that's already there.
- Acceptance:
  - The central panel paints a neutral-gray surround and draws the preview **fitted** (aspect preserved, centered, letterboxed); resizing the window keeps it centered and re-fitted — no scroll region, no native-size overflow. (Manual/visual for the paint; the math is pinned by the transform test below.)
  - A **unit test** `view_transform_round_trips` constructs a `ViewTransform` for a few `(image_size, panel_rect)` shapes (square-in-wide, portrait-in-square, landscape) and asserts `screen_to_image_norm(image_norm_to_screen(p)) ≈ p` within `f32` epsilon for several interior points, and that image corners `[0,0]`/`[1,1]` map to the fitted rect's corners. A second case pins the letterbox: a point in the gray surround maps to a normalized coord outside `[0,1]`.
  - The brush still paints through the transform: the brush pointer-mapping reads `screen_to_image_norm` (no `resp.rect` arithmetic remains in the brush block), dabs still record normalized coords, and a painted stroke still re-renders (the `painted` → `render_preview` path at `gui.rs:571-574` is preserved). Behavior is unchanged from a user's view.
  - Baseline green (fmt/clippy 0-warnings/test). No change to rendered pixels or to `poll_render`/`spawn`.

### U1-C3 — Zoom & pan
- Implements: drastic-UI viewport goal (navigation)            Priority: High
- Crates/files: `gui/canvas.rs` (zoom/pan state on the transform + input handling), the toolbar/status-bar controls (U0 shell surfaces).
- Depends on: U1-C2            Blocks: — (U2 consumes the transform regardless of zoom level).
- Heads-up:
  - **This extends the *same* `ViewTransform` from C2 with zoom and pan state — it does not introduce a second mapping. The transform stays the single source of truth (README global rule 3); zoom/pan are just the `zoom` scalar and `pan` offset C2 already left room for. U2's handles must keep working at any zoom, which they do for free if everything reads the one transform.**
  - **Zoom state.** A `Zoom` notion with: `Fit` (the C2 default — scale recomputed from the panel each frame), `Percent(f32)` for `100%` and the discrete levels `25% 33% 50% 67% 100% 150% 200% 400% 800%` (a fixed ladder; "+"/"−" step along it, clamped at the ends). `Fit` is special: it tracks the panel, so it isn't a fixed percent. Store the *intent* (Fit vs a level), derive `displayed_scale` each frame — keep the transform stateless-per-frame as C2 established.
  - **Wheel-zoom centered on the cursor — the one bit of real math.** On scroll, step the zoom and adjust `pan` so the image point *under the cursor stays under the cursor*: capture `img_norm = screen_to_image_norm(cursor)` *before* the zoom change, then set `pan` so that after the new scale, `image_norm_to_screen(img_norm) == cursor` again. This is the standard "zoom toward cursor" and it's the only place pan and zoom couple — pin it with a test (a cursor-anchored zoom keeps the anchored normalized point fixed). egui gives scroll delta via `ctx.input(|i| i.smooth_scroll_delta)` / `raw_scroll_delta`; gate zoom to the canvas hover so scrolling a panel doesn't zoom the image.
  - **Pan.** Drag-to-pan with **space-drag** (hold Space + left-drag) or **middle-mouse drag** — both common, both unambiguous vs the left-drag that U2 tools will claim. Pan is a screen-space offset added into the transform's `offset`; clamp so the image can't be dragged entirely out of view (keep at least a sliver, or re-center when smaller than the panel). When zoom is `Fit` and the image fits, pan is inert/zero.
  - **Controls + shortcuts.** Toolbar/status-bar: a zoom-% readout (also satisfies C5's status need), a Fit button, a 100% button, and +/− steppers. Keyboard: `+`/`=` zoom in, `-` zoom out, `0` = Fit, `1` = 100% (these are the conventional bindings; register them in the central input handling, and feed U7-C5's cheat-sheet). Don't swallow keys the panels need (text fields, sliders) — gate to canvas focus/hover.
  - **The 100% honesty note — document it, don't fix it here.** At "100%" the preview being displayed is the **1600px downscale** (`preview = full.downscaled(PREVIEW_MAX_DIM=1600)`, `gui.rs:30`), *not* the full-resolution image — so "100%" is 1 preview-pixel per screen-pixel, which is **not** true 1:1 sensor-pixel peeping. Real pixel-peeping would need to render a full-res tile of the visible region on demand (a tile cache), which is **out of scope** for this epic. Add a short doc-comment at the zoom code stating this limitation and that >100% is upscaled preview (soft), so a later card (or U7) can pick up true 1:1 tiling. Do **not** silently let the user believe 200% is sensor-sharp.
  - **No new render work.** Zoom/pan are pure view transform; they never trigger `render_preview` (the texture doesn't change), so they cost nothing on the worker and repaint cheaply. Make sure a zoom/pan gesture requests a repaint but **not** a re-render.
- Acceptance:
  - Zoom and pan behave: wheel zooms toward the cursor, space/middle-drag pans, the ladder steps 25%→800%, `0`=Fit and `1`=100% work, and the **status bar shows the current zoom %**.
  - The `ViewTransform` remains the single source of truth — there is still exactly one `screen_to_image_norm`/`image_norm_to_screen`; zoom/pan only feed its scale/offset. A **test** `wheel_zoom_keeps_cursor_anchored` asserts the normalized image point under the cursor before a cursor-centered zoom maps back to (approximately) the same screen position after.
  - A pan gesture and a zoom gesture request a repaint but do **not** spawn a render (no `render_preview` call); the texture is reused. (Verified by inspection / a small assertion that the dirty/render path isn't entered for a pure view change.)
  - The 100%-is-preview-not-1:1 limitation is documented at the zoom code. Baseline green (fmt/clippy/test).

### U1-C5 — Pixel readout / loupe
- Implements: drastic-UI viewport goal (precision feedback); the shared pick-pixel mechanism            Priority: Low
- Crates/files: `gui/canvas.rs` (hover → pixel sample + optional magnifier), `gui/panels/statusbar.rs` (the readout label).
- Depends on: U1-C2            Blocks: — (but *feeds* U5-C2 clipping and U6-C6 WB eyedropper, which reuse this pick-pixel path).
- Heads-up:
  - **On hover, show the pixel under the cursor's RGB in the status bar; optionally a magnifier loupe. Small on its own, but it establishes the canonical "what pixel is under the pointer" path that U5-C2 (clipping read-back) and U6-C6 (gray-eyedropper white balance) both consume — so build the sampling cleanly, not as a one-off.**
  - **The sample, through the one transform.** On canvas hover, `screen_to_image_norm(cursor)` (C2) gives normalized `[0,1]`; if it's inside `[0,1]` (over the image, not the gray surround), convert to a texture pixel index and read the value. Two values are meaningful and the card should surface at least the first: the **sRGB 8-bit** display value (what `to_color_image`/`to_srgb8` produced — the bytes already uploaded to the texture; this is what the user sees) and, optionally, the **working linear** RGB (the `ImageBuf` value before the output transform). Prefer reading the sRGB from the same byte buffer the texture was built from so the readout matches the screen exactly; reading working-linear needs the rendered `ImageBuf` kept around (the preview render result) — wire whichever the status bar shows and label the space.
  - **Where the pixels are.** The texture is uploaded from `to_color_image(img)` (`gui.rs:333-342`, `1232-1235`) which discards the byte buffer after upload. To read back the sRGB, either keep the last rendered preview `ImageBuf` on the `App` (then `to_srgb8`-sample on demand, or sample the linear value and convert) or cache the `ColorImage`'s pixels. Keeping the last preview `ImageBuf` is the more useful primitive (U6-C6's WB eyedropper wants the *linear/working* pixel to compute a neutral, not the sRGB byte) — so **stash the rendered preview `ImageBuf`** when `poll_render` delivers a `Preview` (`gui.rs:315`), and sample from it. That single stash is the shared pick-pixel substrate the later cards build on.
  - **Coordinate honesty.** The sampled pixel is from the **preview** (downscaled) image — fine for a readout/eyedropper (the value is a faithful area-average), but note it's not the full-res pixel (same 1600px caveat as C3). Normalized `[0,1]` → preview pixel is `(norm * (w-1)).round()`.
  - **The loupe (optional, the magnifier).** A small floating zoomed view around the cursor — sample an N×N neighborhood from the preview `ImageBuf` and paint it enlarged near the pointer, with a crosshair on the center pixel.
