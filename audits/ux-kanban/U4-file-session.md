# U4 — File & session UX

> Depends on **U0** (the modular `gui/` tree and the **U0-C5** menu/toolbar/status shell this epic hangs Open / Open Recent / Export onto). **Single-image scope (decision 1):** `latent` opens **one RAW at a time**, switchable in place — there is **no** folder browser, no filmstrip, no catalog/library. This epic makes that one-at-a-time flow excellent: a native Open dialog, drag-and-drop, recent files, a real export dialog, a welcome state, and a small persisted config — replacing today's CLI-only entry and the bare `text_edit_singleline(&mut self.output)` export field (`gui.rs:476-479`).
>
> **The refactor seam (the spine of this epic):** today `gui::run` (`gui.rs:25-84`) **develops synchronously before the window exists** — `develop_to_image(input)?` at `:29`, downscale + `Arc`, sidecar load (`:37-55`), then `eframe::run_native`. So the program *cannot start without a file*, and opening a second file would mean relaunching. **U4-C1 inverts this:** `run` becomes able to start with **no image** (welcome state, U4-C5), and the develop-and-load step becomes a method on the running `App` that runs **off the UI thread** (rule 5) through the existing worker (`RenderState`/`spawn`/`poll_render`, `gui.rs:254-329`). U4-C1 designs that seam and hands the *visible loading state* to **U8-C1**; every other card in this epic sits on it. Opening must **re-develop** (new `full`/`preview`/`variants`/`sidecar`/`title`/`output`, reset view/history) — not patch settings onto the old base.
>
> **`rfd` justification (the one approved new crate, decision 2 / rule 6):** native OS file/save dialogs are not something to hand-roll in egui, and `rfd` is the standard, well-maintained choice. On Linux it speaks the **XDG Desktop Portal** (`ashpd`) when available and falls back to GTK, so it works under Wayland and Flatpak without us shipping a toolkit. We use it for **Open** (U4-C1), **drag is native egui** (U4-C2, no `rfd`), and **Save** (U4-C4). Pin the version; no other new crate — config persistence uses `serde` (already in the tree via the sidecar) + `std`, with the **OS config dir** resolved by `directories` *(the only judgement call; justified in U4-C6, with a `std`-only fallback noted)*.
>
> **Cross-cutting obligations carried by every card here:** off-thread develop/export (rule 5); atomic temp-then-rename, non-fatal config + sidecar writes that raise a toast, never crash or truncate (rule 9 / **U8-C2**); forward-compatible config (`#[serde(default)]`, rule 4) so an old/partial config still loads; a bad RAW surfaces as an error toast/modal (**U8-C3**), never a panic or `process::exit`; the render output stays byte-identical (rule 2 — this epic moves files and pixels around, it does not change pipeline math).

### U4-C1 — Native Open (rfd) + in-app file switching (the re-develop seam)
- Implements: drastic-UI file UX (decision 1, single-image); the "app can run with no file" refactor    Priority: High
- Crates/files: `latent-app/Cargo.toml` (add `rfd`, pinned); `latent-app/src/gui/app.rs` (the `App` struct + the new `open_path` / `develop_into` seam); `latent-app/src/gui/dialogs.rs` (the `rfd` Open picker, RAW extension filter); `latent-app/src/main.rs` (let `gui::run` start with **no** path).
- Depends on: U0-C1 (module tree), U0-C5 (menu bar — File→Open lives there)    Blocks: U4-C2, U4-C3, U4-C5, U8-C1
- Heads-up:
  - **This is the structural card of the epic. It does two coupled things: (a) make the develop-and-load step callable on a *running* `App` (so a second file swaps in place, no relaunch), and (b) make `App` representable with *no* image open (so the program can start on the welcome state). Get the seam right and U4-C2/C3/C5 and U8-C1 are thin; get it wrong and they all fight `run`'s "develop before the window" shape.**
  - **Today's shape to invert:** `run(input, backend)` (`gui.rs:25-84`) calls `develop_to_image(input)?` (`:29`) *before* `eframe::run_native`, builds the `Arc<ImageBuf>` full/preview (`:30-31`), the title (`:34`), the export `output` default (`:35`), loads the sidecar into a `Document` (`:37-55`), and only then constructs `App { full, preview, variants, sidecar, saved, title, output, … }` (`:61-79`). Every one of those becomes something the *open* path produces, not the *launch* path.
  - **The "no file" representation.** The fields `full`/`preview`/`variants`/`sidecar`/`saved`/`title`/`output`/`texture` are currently always-present (`gui.rs:104-136`). To start file-less, factor the *loaded-document* state into one optional bundle — a `Session` (or `Open`) struct holding `{ full: Arc<ImageBuf>, preview: Arc<ImageBuf>, variants, active, sidecar, saved, title, output, meta }` — and make `App` hold `session: Option<Session>` plus the file-independent UI state (`backend`, `render`, `local_sel`, brush settings, `curve_channel`, recent list, config). `None` ⇒ welcome (U4-C5). This is a bigger move than a one-liner; it is the whole point of the card. Keep the *active-variant* accessors (`active_history`, `gui.rs:222`) working by routing through the session.
  - **The develop-and-load method.** Lift `run`'s body (`:29-55`) into a pure function — e.g. `fn load_session(input, backend?) -> Result<SessionData, Box<dyn Error>>` returning the developed `full`/`preview`/`meta` + the resolved `Document`/sidecar/title/output — that both `main.rs` (first file, if a path was given) and the running `App` call. **Crucially it must run off the UI thread** (rule 5): wire it through the existing worker rather than blocking `update`. The clean fit is a **new `JobKind::Load { input }`** (sibling of `Preview`/`Export`, `gui.rs:180-186`) whose `RenderJob::run` (`:192`) develops and returns a **new `RenderOutput::Loaded(Box<SessionData>)`**, consumed in `poll_render` (`:307`) to install the `Session` and trigger the first preview render. `develop_to_image` is in `main.rs` (`main.rs:84`) and is `Send`-friendly (returns owned `ImageBuf` + `Metadata`); the backend is already `Arc<dyn Backend>` shared with the worker (`gui.rs:33,135`). **This card builds the seam and the synchronous fallback; U8-C1 owns the visible spinner/skeleton over it** — design `JobKind::Load` so U8-C1 only adds the loading *view*, not new plumbing.
  - **What "open" resets (re-develop, don't patch).** On a successful load, **replace** the whole `Session`: new `full`/`preview` (`Arc`), fresh `variants` from the new file's sidecar (or a default `Settings`), `active = 0`, new `sidecar`/`saved`/`title`/`output`, drop the old `texture` so the stale image can't flash, and reset view/history (the U1 `ViewTransform` fit/zoom/pan resets to fit; cross-ref U1-C2/U1-C3 — leave the seam, U1 owns the reset). The `local_sel`/`curve_channel`/brush UI state reset to defaults. **Do not** carry the previous image's settings onto the new base.
  - **The Open dialog (rfd).** In `dialogs.rs`, `rfd::FileDialog::new().add_filter("RAW images", &[…]).set_directory(last_open_dir).pick_file()` → `Option<PathBuf>`. The extension list mirrors what `latent_raw::unpack` accepts — keep it broad (nef, cr2/cr3, arw, dng, raf, orf, rw2, pef, srw, …) plus an "All files" filter so an unusual extension is still reachable; the *real* gate is `unpack`, not the filter. **`rfd`'s `pick_file` blocks** — that is fine for picking a *path* (the user is in the dialog, the editor isn't expected to animate), but the **develop that follows must be the async `JobKind::Load`**, not synchronous. Remember the chosen file's parent as `last_open_dir` in config (U4-C6) so the next Open starts there.
  - **Wiring + shortcut.** File→Open (U0-C5 menu) and **`Ctrl+O`** (egui `ctx.input(|i| i.modifiers.command && i.key_pressed(egui::Key::O))`, mirroring the existing Cmd/Ctrl+Z handling at `gui.rs:356-368`) both call `open_via_dialog`. Guard against re-entrancy while a `Load` is in flight (the menu item / shortcut no-op when `render.is_busy()` with a load, like Export is disabled mid-render at `gui.rs:481-489`).
  - **Errors, not panics (U8-C3).** A non-RAW or corrupt selection makes `develop_to_image` return `Err` (`main.rs:84-94`; today `main` maps that to `process::exit(1)`, `main.rs:116-119`). Inside the running app that must become a **toast/modal** (U8-C3) and leave the current session untouched — never exit the process, never panic. Carry the error string back through `RenderOutput::Loaded(Result<…>)` (or a `LoadFailed` variant) and surface it in `poll_render`.
  - **Forward-compat / autosave preserved.** The opened file's sidecar still loads through `Document::from_ron` (`gui.rs:39-41`, forward-compatible per rule 4) and autosaves through the existing `autosave` (`gui.rs:228-247`) bound to the *new* sidecar path — switching files must flush/teardown the old session's pending autosave first (it's already gesture-gated on `is_idle`, `gui.rs:229`, so swap only when idle or after a final save).
  - **Note for U6-C5:** the auto-apply-lens-on-fresh-document block (`gui.rs:49-53`, `auto_lens_profile` at `:89-102`) currently runs inside `run`. When you lift `run`'s body into `load_session`, that block moves with it verbatim for now — **U6-C5 removes the auto-apply** and makes lens opt-in. Don't delete it here; just don't duplicate it.
- Acceptance:
  - File→Open (and `Ctrl+O`) opens a native `rfd` picker; choosing a RAW **swaps the image live** — the new image, title, variants, sidecar, and export default all replace the old, the view resets to fit, and **no relaunch** occurs. A manual behavior check (display-driven Open is not unit-testable) plus a unit test on the pure `load_session`/`SessionData` builder (develop a tiny synthetic input stand-in, assert the produced title/output/sidecar paths and that a present sidecar's variants win) — mirror the testability split the codebase already uses (pure `RenderJob::run` is tested without a window, `gui.rs:1313`).
  - The app **starts with no file** and shows the welcome state (U4-C5): `gui::run` accepts `Option<&Path>`/no path and constructs `App { session: None, … }` without developing anything; a test asserts a `None`-session `App` is constructible and that `load_session` is what populates it.
  - A **non-RAW** selection surfaces an **error toast/modal** (U8-C3) and leaves the current session intact — **no panic, no `process::exit`**. A test pins that the load path returns `Err` (not a panic) for a garbage input, paralleling `develop_errors_on_bad_input` (`main.rs:190-202`).
  - The develop runs through `JobKind::Load` on the worker (rule 5) — `update` never calls `develop_to_image` directly; a code check + the existing render-gate test style confirm one load at a time. `cargo fmt --check`, `cargo clippy --workspace --all-targets` (0 warnings), `cargo test --workspace` green.

### U4-C2 — Drag-and-drop open + bare-path CLI arg
- Implements: faster file entry (decision 1)    Priority: Medium
- Crates/files: `latent-app/src/gui/app.rs` (egui dropped-files handling in `update`); `latent-app/src/main.rs` (accept a bare path as shorthand for `open`).
- Depends on: U4-C1 (the `open_path` seam)    Blocks: —
- Heads-up:
  - **Two small affordances, both routing into U4-C1's `open_path`. No `rfd` here — egui already carries drops, and clap already parses args. The only real work is the hover feedback and the clap shape.**
  - **Drag-and-drop.** egui surfaces dropped files on the context: `ctx.input(|i| i.raw.dropped_files.clone())` (a `Vec<DroppedFile>`, each with a `path: Option<PathBuf>`). In `update` (after `poll_render`, alongside the keyboard block at `gui.rs:356-368`), take the **first** dropped file with a `path` and call the same `open_path` U4-C1 exposes — re-develop, identical reset semantics, identical error toast for a non-RAW drop (U8-C3). Single-image scope (decision 1): **ignore extra dropped files** (open only the first; don't queue a "batch"). Show a **hover hint** while a drag is over the window — read `i.raw.hovered_files` and paint a centered "Drop a RAW to open" overlay (the welcome state, U4-C5, already shows a drop hint when empty; this is the same hint surfaced over an open image). Web-target file *bytes* (`DroppedFile::bytes`) are irrelevant — `latent` is native and develops from a path; if a drop has no `path`, ignore it.
  - **Bare-path CLI.** Today the only window entry is `latent open <input> [--gpu]` (`main.rs:54-62,114`); there is no way to *just* name a file. Make `latent photo.nef` shorthand for `latent open photo.nef`. Cleanest with clap: keep the `Open`/`Develop` subcommands, but allow a **bare positional** when no subcommand is given (a top-level optional `Option<PathBuf>` + `#[command(subcommand)] Option<Command>`, or clap's `args_conflicts_with_subcommands`/external-subcommand pattern). Resolve precedence explicitly: a real subcommand wins; otherwise a lone path means `open` (CPU backend by default, matching `--gpu` default-false at `main.rs:176-180`); nothing at all launches the **welcome state** (U4-C1/U4-C5) rather than erroring. Keep the existing `Develop`/`Open` parse tests green (`main.rs:127-181`) and add bare-path tests beside them.
  - **GPU pref interplay.** A bare path can't pass `--gpu`; it should honor the **persisted GPU preference** from config (U4-C6 / U7-C6) when present, else default CPU. Leave the seam (read the pref in `select_backend`'s caller); the live toggle is U7-C6 — here just don't hard-wire CPU if a pref exists.
- Acceptance:
  - Dropping a RAW onto the window opens it (re-develops, same reset/error semantics as Open); a hover hint shows while a file is dragged over; extra files in one drop are ignored (single-image). Display-driven, so a manual check; the path→`open_path` dispatch is the same tested seam as U4-C1.
  - `latent photo.nef` (no subcommand) opens the editor on that file; `latent` with no args launches the welcome state; `latent open …` and `latent develop …` still parse exactly as before — **tests** `cli_parses_bare_path_as_open`, `cli_no_args_is_welcome`, alongside the preserved `cli_parses_open_gpu` / `cli_parses_develop` (`main.rs:127-181`). Baseline green (fmt/clippy/test).

### U4-C3 — Recent files (persisted)
- Implements: session continuity (decision 1)    Priority: Medium
- Crates/files: `latent-app/src/gui/app.rs` (File→Open Recent submenu, recent-list mutation on open); `latent-app/src/gui/config.rs` (the persisted list — U4-C6 owns the file).
- Depends on: U4-C1 (open seam), U4-C6 (config persistence)    Blocks: —
- Heads-up:
  - **A most-recent-first, de-duplicated, capped list of absolute paths, living in the app config (U4-C6) and surfaced as File→Open Recent (U0-C5 menu) and on the welcome screen (U4-C5). The logic is small and *pure* (a `push_recent(list, path, cap)` function) — keep it out of the UI so it's unit-testable.**
  - **Data + ordering.** Store `recent_files: Vec<PathBuf>` in the config struct (U4-C6, `#[serde(default)]`). On every successful `open_path` (U4-C1), **canonicalize** the path (so the same file via different relative paths de-dups), move it to the front, remove any earlier copy, and **cap** the length (8–10; pick one and pin it). A failed open (bad RAW) does **not** add a recent. Implement as a free function `push_recent(&mut Vec<PathBuf>, PathBuf, cap)` so the dedup/cap/order is tested without a window.
  - **Surfacing.** File→Open Recent (U0-C5) lists the entries newest-first; show the **file name** with the parent dir as a hover/secondary (full paths get long). The welcome state (U4-C5) shows the same list as big click targets. Clicking an entry calls the same `open_path` (re-develop, same error semantics).
  - **Prune missing.** A recent path whose file no longer exists must not be a trap. **Prune-on-display** (filter `path.exists()` when building the menu/welcome list — cheap, no surprise mutation) and **prune-on-open-failure** (if opening a recent fails because it's gone, drop it from the list and toast, U8-C2). Persist the pruned list on the next config save (U4-C6 saves-on-change). Don't block startup `stat`-ing every entry on a slow disk — prune lazily as the menu opens.
  - **Persistence is U4-C6's job.** This card *reads and mutates* `config.recent_files`; the atomic temp-then-rename write, the OS config dir, and the load-on-startup all live in U4-C6 (which this depends on). Save-on-change (after a successful open mutates the list) so recents survive a crash, not just a clean exit.
- Acceptance:
  - Opening files populates File→Open Recent and the welcome list newest-first, **de-duplicated** and **capped**; the entries **persist across runs** and **open on click** (re-develop). A **test** `push_recent_dedups_caps_and_orders` drives the pure list function (push existing → moves to front, no dup; exceed cap → oldest dropped; newest first). Manual check for the menu/welcome wiring (display-driven).
  - A recent entry whose file is missing is **pruned** (not shown / dropped on failed open) without crashing — a **test** on the prune filter over a list with a non-existent path. Baseline green (fmt/clippy/test).

### U4-C4 — Export dialog (format / depth / quality, last dir, progress + toast)

The dialog: a **format** choice (JPEG/PNG/TIFF), a **bit depth** toggle **constrained to what the format allows** — 8 only for JPEG, 8/16 for PNG and TIFF, defaulting to the `recommended_depth` for that format — and a **JPEG quality** slider that shows **only when format = JPEG**. Default-select the format from the chosen extension so picking `photo.png` pre-selects PNG/16. **Don't invent new encoder capability** — `save_auto` already routes 16-bit to `save_16` and 8-bit to `save`; quality wiring may need a small `latent-export` touch *only if* the JPEG encoder's quality isn't yet parameterized (check `save_buffer_with_icc`'s `JpegEncoder::new`, `latent-export/src/lib.rs:256` — `new` is default-quality; `new_with_quality` exists). The dialog's constraints should make a 16-bit-JPEG choice *unreachable* in the UI, but the export path must still **degrade gracefully** — route the `Result` into a **fail toast** (U8-C2/U8-C4) carrying the error message, not the old plain-string `self.status` label (`gui.rs:490-492`). Reuse `export_status` (`gui.rs:206-211`) as the message mapper feeding the toast.
  - **Progress + toast (U8-C2/U8-C4).** While the export runs (it's already off-thread), show **progress** — at minimum an indeterminate "Exporting…" indicator (today a status string set at `gui.rs:284`); U8-C4 owns determinate progress if the render reports it. On completion, a **success toast** ("Exported to …") or **fail toast** (the encoder error). The status `String` field (`gui.rs:121`) and its label (`:490-492`) are replaced by the toast queue (U8-C2) — leave the seam so U8-C2 owns the toast widget; this card produces the events.
  - **One export at a time.** Keep the busy-gate (`gui.rs:273-275,481`) — the Export action no-ops / is disabled while a render or export is in flight.
- Acceptance:
  - The bare path field + button are **gone**, replaced by an Export dialog: choosing **format + depth + path** exports the **full-resolution** image **off-thread** (same worker path as today) and writes the chosen depth; a **test** extends `render_job_export_writes_and_reports` (`gui.rs:1313-1343`) to pass an explicit `depth` through `JobKind::Export` and assert the encoded depth matches (8-bit JPEG vs 16-bit TIFF), proving the depth choice reaches `save_auto`.
  - **Invalid combos error gracefully:** an unsupported extension or a (UI-unreachable but defensively handled) 16-bit-JPEG export surfaces a **fail toast** with the encoder's typed message — **no panic, no silent no-op**; a test asserts the export result is `Err` and `export_status` produces a "Export failed:" message for it (paralleling `export_status_maps_ok_and_err`, `gui.rs:1259-1270`).
  - The **last export dir persists** (config, U4-C6): the next Save dialog opens there; the default file name derives from the source stem. JPEG quality control shows only for JPEG (or is documented as a follow-up if the encoder isn't parameterized). Baseline green (fmt/clippy/test).

### U4-C6 — App config persistence (window, recent, panel width, export dir, GPU pref, theme)
- Implements: session/UX continuity (decision 2 — minimal deps, OS config dir)    Priority: Medium
- Crates/files: `latent-app/src/gui/config.rs` (the serde config type, OS-config-dir load/save, atomic write); touches `latent-app/Cargo.toml` (`directories`, justified below) and `app.rs`/`main.rs` (load at startup, save on change).
- Depends on: U0-C1 (module tree)    Blocks: U4-C3 (recent files), U3-C3 (panel width), U7-C6 (GPU pref)
- Heads-up:
  - **A small serde config in the OS config dir, loaded once at startup and saved (atomically) when a tracked value changes. It is the *backing store* for several other cards — recents (U4-C3), the side-panel width (U3-C3), and the GPU preference (U7-C6) — so define the struct here with room for all of them, every field `#[serde(default)]` so a partial/old config still loads (rule 4). This card owns the *file*; the consumer cards own *what they put in it*.**
  - **Shape.** A `Config { window_size: Option<(f32,f32)>, recent_files: Vec<PathBuf>, side_panel_width: Option<f32>, last_export_dir: Option<PathBuf>, last_open_dir: Option<PathBuf>, gpu: bool, theme: Theme }` (extend as cards need; all defaulted). `Theme` can be a small enum (Dark default) pairing with U0-C2. Keep it **flat and serde-derived** — RON (the sidecar already uses RON via `Document::to_ron`/`from_ron`, `latent-edit:912,927`, so no new format crate) or JSON; pick RON to match the project and reuse the established serde stack — **no `confy`** (it adds a dep and a format opinion for what is a dozen fields; `serde` + `std` + the dir crate is enough, decision 2 / rule 6).
  - **Where it lives — the one dependency judgement.** The config belongs in the **OS config dir** (`~/.config/latent/config.ron` on Linux, the platform-appropriate dir elsewhere). `std` has **no** portable config-dir API, so this needs either `directories` (`ProjectDirs::from("", "", "latent").config_dir()`) or `dirs`. **`directories`** is the right minimal pick — tiny, no transitive weight, purpose-built, and it gives the correct per-OS path (XDG on Linux, `~/Library/Application Support` on macOS, `%APPDATA%` on Windows). Pin it. *(If even that is unwanted, a `std`-only fallback is `$XDG_CONFIG_HOME`/`$HOME/.config` on Linux — but that's a per-OS hand-roll; prefer `directories` and justify it as the second small crate, the way `rfd` is the first.)*
  - **Atomic, non-fatal writes (rule 9).** Write **temp-then-rename**: serialize to a string, write to `config.ron.tmp` in the same dir (same filesystem, so `rename` is atomic), `fs::rename` over the real file. **Create the config dir if missing** (`create_dir_all`). A failed write (read-only dir, disk full) **raises a toast** (U8-C2) and is otherwise ignored — **never crash, never truncate** the existing file (the temp-then-rename guarantees the old file survives a failed write). This mirrors the sidecar's autosave discipline (`gui.rs:228-247`), which this card should reuse the *shape* of (it currently writes directly with `fs::write`, `:241` — the config gets the stronger temp-then-rename per rule 9; consider lifting a shared `atomic_write(path, text)` helper that the sidecar can later adopt).
  - **Load at startup, save on change.** Load in `main.rs`/`run` before constructing `App`; a **missing** config ⇒ `Config::default()` (first run, no error), a **corrupt** config (bad RON) ⇒ `default()` + a one-time toast, **never a crash** (rule: corrupt config falls back to defaults). Hold the `Config` in `App`; **save on change** — debounce to avoid writing every frame (write when a tracked value actually differs from the last saved, the same "did it change?" gate the autosave uses at `gui.rs:233`). Save window size on resize (read `ctx.input(|i| i.viewport().inner_rect)` / the eframe `on_exit`/`save` hook), export/open dirs on a successful dialog, recents on open, panel width / GPU / theme when their cards change them.
  - **Window size apply (startup).** On launch, apply the persisted `window_size` via `eframe::NativeOptions { viewport: ViewportBuilder::default().with_inner_size(…) }` (today `NativeOptions::default()`, `gui.rs:59`) — couples with **U0-C6** (window/DPI defaults); leave U0-C6 to own the *defaults*, this card supplies the *persisted override* when present.
- Acceptance:
  - The config **round-trips**: a `Config` serialized and re-read yields the same values — a **test** `config_roundtrips` over the serde type (no I/O), plus a **test** `config_write_is_atomic_and_load_recovers` writing to a temp dir and reading back. The **OS config dir** is resolved via `directories` and the dir is created if missing.
  - A **missing** config loads as defaults (first run, no error); a **corrupt** config falls back to **defaults without crashing** (rule 9) — a **test** `corrupt_config_falls_back_to_default` feeds garbage bytes and asserts `Config::default()` is returned (not a panic/`Err` that aborts startup).
  - Writes are **temp-then-rename** and **non-fatal**: a failed write raises a toast (U8-C2) and leaves the prior config intact — a **test** asserts a `.tmp` is used and the real file is replaced atomically (and that a write to an unwritable path returns the error to be toasted, not a panic). The struct carries the fields U4-C3 / U3-C3 / U7-C6 consume, all `#[serde(default)]`; an old config missing newer fields still loads (serde-default test). Baseline green (fmt/clippy/test).

## Epic done when

`latent` is a real single-image editor session rather than a CLI one-shot: **File→Open** (and `Ctrl+O`) opens a native `rfd` picker and **re-develops the chosen RAW into the running window off the UI thread** — no relaunch — because `run` no longer develops before the window exists but starts on a **welcome state** with `session: None` and loads files through a worker `JobKind::Load` (U4-C1, the seam every other card sits on); files can also be **dropped** on the window or named as a **bare CLI arg** (`latent photo.nef`), with no-args launching the welcome (U4-C2); a **persisted, de-duplicated, capped recent-files list** is reachable from File→Open Recent and the welcome screen, pruning missing paths (U4-C3); the bare export path field is replaced by an **Export dialog** — `rfd` Save picker, format/depth (mirroring the `develop` CLI's 16-for-tif/png, 8-for-jpg rule) and JPEG quality, last-dir memory, running the existing off-thread export and surfacing the encoder's typed errors (unknown extension, 16-bit JPEG) as **toasts** (U4-C4); an empty-state **welcome** screen (big Open, recents, drop hint, version) shows when nothing is open and transitions cleanly into the editor (U4-C5); and a small **serde config** in the OS config dir (via `directories`) — window size, recents, side-panel width, last export/open dirs, GPU preference, theme — is loaded at startup and written **atomically, non-fatally** on change, falling back to defaults on a missing or corrupt file (U4-C6). The two new crates are **`rfd`** (native dialogs, XDG-portal-aware on Linux) and **`directories`** (the OS config-dir path); the render pipeline math is untouched (rule 2). The async-loading *view* is handed to **U8-C1**, the toast widget to **U8-C2/U8-C4**, the error modal to **U8-C3**, the lens auto-apply removal to **U6-C5**, and the live GPU toggle / panel-width consumers to **U7-C6 / U3-C3** — each seam is left explicit here.
