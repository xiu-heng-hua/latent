# U8 — States, feedback & polish

> Polish epic. It makes the app **honest about what it's doing** — loading, working, succeeding, failing — using only egui-native chrome and the worker machinery U0 already established, with **no new crate** (Global rule 6). The keystone is **C1: move the initial develop off the UI thread.** Today `gui::run` calls `crate::develop_to_image(input)?` *synchronously before `eframe::run_native` ever opens a window* (`gui.rs:29`); a slow RAW decode is a frozen, windowless start, and a bad file means the window **never opens** — `main` turns the `Err` into `eprintln!` + `process::exit(1)` (`main.rs:116-119`). C1 opens the window first, develops on the existing `RenderState`/`spawn`/`poll_render` worker, and shows a spinner while it runs. That single change is what makes an **in-app Open** possible at all (U4-C1 re-develops into the *running* app), and it forces the app to hold an explicit **load state** (no-image / loading / ready / error) that U4-C5's welcome screen and U8-C3's error modal both branch on.
>
> On top of that load state: **C2** replaces the bare `status: String` (`gui.rs:122`) with a small **toast queue** for transient feedback (export done, save failed, GPU fallback) — auto-dismissing, stacking, egui-painted, while a persistent status line stays in the status bar for steady state (U0-C5); **C3** turns a decode error into a **friendly modal** with Retry / Open-another / Dismiss instead of killing the process — the GUI stops calling `exit(1)`, the CLI `develop` path keeps it; **C4** surfaces the (already-correct, but invisible) "an export is running" state as a visible **indeterminate progress** affordance; **C5** adds an **About** dialog (app version, linked lensfun version via `latent_lens::version`, active backend) and a one-time **first-run hint** pointing at Open / drag-drop.
>
> **Keep the worker, don't rewrite it.** Every card here is *additive* to the existing off-thread render/export path: C1 adds a `RenderOutput::Developed`-style message and a load-state field; C4 reads the `render.is_busy()` flag that already exists (`gui.rs:163`). Nothing changes `RenderJob::run`'s pure render/export contract or the coalescing gate (`gui.rs:255-258, 307-329`), and nothing touches pixel math (Global rule 2 — `to_color_image` / `latent_export::to_srgb8` / `latent_pipeline::render` stay byte-identical; the tripwire `to_color_image_matches_the_export_transform` stays green).
>
> **Shared surface:** the load-state machine lives in `gui/app.rs` (a new `enum LoadState` field on `App`, replacing the implicit "`texture.is_none()` means not-ready" check at `gui.rs:352, 513`); `gui/state.rs` gains the develop message and a `DevelopJob` mirroring `RenderJob`; `gui/toasts.rs` (new) owns the toast queue; `gui/dialogs.rs` (new) owns the error + About modals and the first-run hint; `gui/panels/statusbar.rs` (from U0-C5) shows the persistent status line + progress. Take the cards in order: **C1** establishes the load-state machine and async develop (the enabler), **C2** adds the toast queue, **C3** adds the error modal on top of C1's `Error` state and U4-C1's in-app open, **C4** wires progress into C2's toasts + the status bar, **C5** finishes with About + the first-run hint.
>
> **egui 0.32.3 wiring notes** (pinned, `Cargo.lock`): a spinner is `egui::Spinner` / `ui.spinner()`; a modal is `egui::Window::new(..).collapsible(false).resizable(false).anchor(Align2::CENTER_CENTER, …)` shown over a dimmed full-screen `egui::Area` (egui 0.32 also has `egui::Modal` — prefer it if its API on the pinned version covers the dim+focus we need; otherwise the `Window`+`Area` pair is the portable fallback). Toasts are a custom `egui::Area` stack anchored bottom-right, animated with `ctx.animate_value_with_time` / `request_repaint_after`. `env!("CARGO_PKG_VERSION")` gives the app version at compile time; `latent_lens::version()` (`latent-lens/src/lib.rs:37`) gives the linked lensfun `(major, minor, micro, bugfix)`. The first-run flag is remembered in the app config from **U4-C6**.

### U8-C1 — Async loading states (initial + on-Open develop off the UI thread)
- Implements: board "states & polish"; Global rule 5 (never block the UI thread) — the enabler for in-app Open            Priority: High
- Crates/files: `latent-app/src/gui/app.rs` (`run` setup, `update` first-frame flow, the new `LoadState`), `latent-app/src/gui/canvas.rs` (the not-ready branch → spinner/skeleton), `latent-app/src/gui/state.rs` (a `DevelopJob` + `RenderOutput::Developed`-style message).
- Depends on: U0-C1            Blocks: U4-C1 (in-app open re-develops into the running app), U4-C5 (welcome state is the `NoImage` arm), U8-C3 (error modal is the `Error` arm).
- Heads-up:
  - **This is the keystone of the epic. Today `gui::run` develops the RAW *before the window exists* — `let (full, meta) = crate::develop_to_image(input)?;` at `gui.rs:29`, then builds `preview`/`full`/`title`/`output`/the sidecar document, *then* calls `eframe::run_native` (`gui.rs:57`). A slow decode is a frozen, windowless launch; a decode error short-circuits with `?` so the window never opens and `main` exits (1). Move that develop onto the existing worker so the window opens immediately on an empty canvas and develops in the background. This is precisely what U4-C1's in-app file switch needs (re-develop into a *running* app), and what U8-C3 needs (an error has somewhere to land — a modal — instead of a process exit).**
  - **The load-state machine (the gotcha — the app must hold an explicit state, not infer it from `texture.is_none()`).** Today readiness is implicit: `update` treats `self.texture.is_none()` as "not rendered yet" and shows a centered `"Rendering…"` label (`gui.rs:352, 513-519`). That conflates *four* distinct states the rest of the epic must distinguish. Add an explicit field on `App`:
    ```
    enum LoadState {
        NoImage,                 // nothing open — U4-C5 welcome screen draws here
        Loading { path: PathBuf },// develop in flight — spinner + "Developing <name>…"
        Ready,                   // full/preview/meta present — normal editor
        Error  { path: PathBuf, message: String }, // decode failed — U8-C3 modal
    }
    ```
    `Ready` is the only state in which the right panel's sliders, Export, and the brush canvas are meaningfully interactive — gate them on it (in `Loading`/`Error`/`NoImage` the panels can show but their actions are disabled or absent). The existing `texture.is_none()` check inside `Ready` still distinguishes "developed, first preview render not back yet" (keep the spinner there too).
  - **What `develop` produces, and why it can't be a plain `RenderJob`.** `develop_to_image` (`main.rs:84`) returns `(ImageBuf, latent_raw::Metadata)` and the current `run` does real work with *both* halves: the full-res `ImageBuf` becomes `full`/`preview` (`gui.rs:30-31`), and `meta` drives `auto_lens_profile` + sidecar/title/output setup (`gui.rs:34-54, 89`). So the develop message must carry **the developed image *and* the metadata** (and the source path), not just an image. Add a sibling to `RenderJob` in `state.rs` — a `DevelopJob { path: PathBuf }` whose `run()` calls `crate::develop_to_image(&path)` and returns a new `RenderOutput::Developed(Result<DevelopOutput, String>)` where `DevelopOutput` bundles `{ full: ImageBuf, meta: latent_raw::Metadata, path: PathBuf }`. Keep the **`Send`** discipline the existing worker relies on (`gui.rs:170-171`): `ImageBuf` and `Metadata` are plain data, no lensfun `Database` crosses the thread (the lens *lookup* — `auto_lens_profile`, `gui.rs:89-102` — stays on the **main thread** in `poll_render` after the develop returns, exactly where the `Database::load` non-`Send` handle is safe; do **not** move `auto_lens_profile` into the worker).
  - **Where the post-develop setup moves.** Everything `run` does *after* the develop today (build `preview = downscaled(PREVIEW_MAX_DIM)`, `full`/`backend` `Arc`s, `title`, `output`, read the sidecar `<raw>.ron`, `auto_lens_profile` on a fresh document, `saved`, `variants`) must move out of `run`'s pre-window block into a method that runs **when the `Developed` message arrives** — e.g. `App::on_developed(ctx, DevelopOutput)`. On `Ok`, it builds those fields, flips `LoadState::Loading → Ready`, and kicks the first `render_preview` (the same dirty-on-first-frame path, `gui.rs:352, 506-508`). On `Err(message)`, it flips to `LoadState::Error { path, message }` (U8-C3 renders the modal from that). `run` itself shrinks to: pick `output`/`backend`, build an `App` in `LoadState::Loading { path }` with empty `variants`/no `full`/no `texture`, **spawn the `DevelopJob`**, then `run_native`. (The window now opens before any decode.)
  - **`full`/`preview`/`variants` become "not yet present".** Currently these are non-optional fields set before the window (`gui.rs:107-111`). With async develop they don't exist until `Developed` arrives. Make them `Option` (or hold them inside the `LoadState::Ready` arm) and have `update`/`render_preview`/`export` early-return (or no-op) unless `Ready`. This is the mechanical cost of the move; keep it tight — the worker types (`RenderJob`, `RenderState`, `spawn`, `poll_render`) are unchanged except `poll_render` learns to match the new `RenderOutput::Developed` arm and dispatch to `on_developed`.
  - **The spinner/skeleton (`canvas.rs`).** Replace the bare `ui.label("Rendering…")` placeholder (`gui.rs:514-516`) with a real loading affordance: an `egui::Spinner` plus a `"Developing {filename}…"` line, centered on the neutral canvas surround (U0-C2's `CANVAS_SURROUND`). Use the same widget for the `Ready`-but-`texture.is_none()` sub-state (first preview render in flight) with text "Rendering…". Keep `ctx.request_repaint()` while any worker is in flight (the existing call at `gui.rs:517`) so the spinner animates and the message is polled.
  - **Cross-cutting / acceptance pressure:** opening a *large* RAW must show the spinner immediately and keep the window interactive (you can move it, hit menus) while develop runs — the explicit acceptance. Because develop is now `Send` plain-data over the channel, the lens `Database` non-`Send`-ness is not a regression (it never crosses; the lookup stays main-thread). Don't fork the develop into a second thread-spawn helper — reuse `spawn` by generalizing it to take "any job that returns a `RenderOutput`" (or add a tiny `spawn_develop` sibling that mirrors `spawn`'s `ctx.request_repaint()`-on-done contract, `gui.rs:291-302`).
- Acceptance:
  - `gui::run` no longer calls `develop_to_image` before `run_native`; the window opens in `LoadState::Loading` and the develop runs on a worker thread. Opening a large RAW shows a spinner (not a freeze); the window is draggable/clickable during develop — verified manually.
  - A `LoadState` enum with `NoImage`/`Loading`/`Ready`/`Error` arms exists on `App`; readiness is read from it, not inferred from `texture.is_none()` (the implicit check at `gui.rs:352, 513` is subsumed). Panels/Export/brush are interactive only in `Ready`.
  - `RenderOutput` gains a `Developed(Result<DevelopOutput, String>)`-style arm; `DevelopJob::run` is **`Send`** (plain `ImageBuf`+`Metadata`+`PathBuf`, no lensfun `Database`) and unit-testable off any window. A **test** `develop_job_reports_error_for_bad_input` (mirroring `main.rs`'s `develop_errors_on_bad_input`, `main.rs:190`) asserts a garbage input yields the `Err(message)` arm, not a panic — proving the error reaches the load state instead of `exit(1)`.
  - `auto_lens_profile`/sidecar/title/`saved`/`variants` setup runs in `on_developed` on the **main thread** (the lens lookup never crosses the worker boundary); on success the first `render_preview` fires and the editor reaches `Ready`. The existing five relocated tests and the worker coalescing test stay green.
  - Baseline green: `cargo fmt --check`, `cargo clippy --workspace --all-targets` (zero warnings), `cargo test --workspace`; `to_color_image_matches_the_export_transform` unchanged (no pixel change).

### U8-C2 — Non-blocking toasts (success / info / error queue)
- Implements: board "states & polish"; transient feedback that doesn't hijack a steady-state label            Priority: Medium
- Crates/files: new `latent-app/src/gui/toasts.rs` (the `Toast`, `ToastKind`, `Toasts` queue + its draw); call sites in `gui/app.rs` (replace `self.status = …` transient writes), `gui/panels/statusbar.rs` (the persistent status line stays here).
- Depends on: U0-C1            Blocks: U8-C4 (export-done / failure reports via a toast).
- Heads-up:
  - **Build a small egui-native toast queue — NO new crate (Global rule 6; this is *not* `egui-toast`).** Today every transient outcome is jammed into one `status: String` field that a label prints when non-empty (`gui.rs:122, 284, 490-492`), and it's also written by failure paths (`autosave`'s "Save failed: {e}" / "Serialize failed: {e}", `gui.rs:243, 245`; `export_status`'s "Export failed:", `gui.rs:208-210`). A single sticky string can't stack two outcomes, never auto-clears, and mixes *steady state* (the autosave/render indicator) with *one-shot events* (export finished). Split the two concerns: **transient events → toasts** (auto-dismiss, stack); **steady state → the status bar** (U0-C5's `statusbar.rs`).
  - **The queue (the shape).** In `toasts.rs`:
    ```
    enum ToastKind { Success, Info, Error }
    struct Toast { kind: ToastKind, text: String, shown_at: f64, ttl: f32 }  // ttl secs
    #[derive(Default)] struct Toasts { items: Vec<Toast> }
    impl Toasts {
        fn push(&mut self, kind, text);                 // success/info/error helpers
        fn ui(&mut self, ctx: &egui::Context);          // draw + expire, once per frame
    }
    ```
    `ui` draws an `egui::Area` stack anchored bottom-right (above the status bar), one frame per toast colored by kind (Success/Info/Error reading U0-C2's `theme` accent/gray/an error red token — add `theme::ERROR`/`theme::SUCCESS` if absent, keeping the neutral-chrome rule). Auto-dismiss: drop a toast when `ctx.input(|i| i.time) - shown_at > ttl`; fade the last ~0.3s with `ctx.animate_value_with_time` and `ctx.request_repaint_after(remaining)` so the queue animates without a busy-loop. Error toasts get a longer TTL (or are click-to-dismiss / sticky) — a failure the user should actually see shouldn't vanish in 2s. Stack newest-on-top (or bottom — pick one and be consistent), capping the visible count so a burst doesn't fill the screen.
  - **Migrate the transient writers; keep the steady-state line.** Add `toasts: Toasts` to `App`, call `self.toasts.ui(ctx)` once per frame in `update` (after `poll_render`). Route the *event* outcomes to it: export finished (the `RenderOutput::Export(status)` arm, `gui.rs:316` — success → `Success` toast, "Export failed" → `Error` toast; split `export_status` into success/failure so the kind is known, or parse it — prefer threading a `Result` so the kind isn't string-sniffed), and `autosave`'s write/serialize failures (`gui.rs:243-245` → `Error` toast instead of setting `status`). The **persistent** status line (render-in-flight "Rendering…", an autosave "Saved"/"Saving…" indicator, the active backend) stays a steady label owned by `statusbar.rs` (U0-C5) — toasts are for things that *happen*, the status bar is for things that *are*. A GPU-fallback notice (`select_backend`'s `eprintln!("GPU unavailable …")` at `main.rs:74`) is a candidate `Info`/`Error` toast once that string reaches the GUI (thread the fallback fact through `run` alongside the backend label U0-C5 already threads).
  - **Don't double-report.** After this card the bare `status` field is either gone or demoted to the steady-state status-bar string only; an export must report via **toast**, not also via a sticky label (the old `if !self.status.is_empty() { ui.label(...) }` at `gui.rs:490-492` is removed or moved to the status bar). Audit every `self.status = …` write (`gui.rs:243, 245, 284, 316`) and classify each as transient (toast) or steady (status bar).
  - **Cross-cutting / test:** the draw is display-unverifiable, but the queue logic is pure and must be pinned. A **test** `toast_queue_expires_after_ttl` pushes a toast at t=0, advances a simulated clock past its ttl, and asserts the expiry pass removes it (and a not-yet-expired one survives) — factor the expiry as a pure `fn retain_unexpired(&mut self, now: f64)` the `ui` method calls so it's testable without a `Context`. A **test** `export_failure_is_an_error_toast` asserts a failed export maps to `ToastKind::Error` (proving the success/failure split from the old string).
- Acceptance:
  - A `toasts` module with a stacking, auto-dismissing queue (Success/Info/Error) exists and is drawn once per frame; an **export reports via a toast** (success and failure distinguished by kind), and a **failed sidecar save toasts** (`autosave`'s error path, `gui.rs:243-245`) instead of only setting a label.
  - A persistent status line remains in the status bar (U0-C5) for steady state (render-in-flight, autosave indicator, backend) — toasts and the status line don't duplicate the same message.
  - **Tests:** `toast_queue_expires_after_ttl` (pure expiry, no `Context`) and `export_failure_is_an_error_toast` pass. No new crate added (`egui-toast` is *not* a dependency).
  - Baseline green (fmt/clippy zero-warnings/test); image tripwire unchanged.

### U8-C3 — Error modals (bad RAW → friendly modal, not `process::exit(1)`)
- Implements: board "states & polish"; the GUI stops killing the process on a decode error            Priority: High
- Crates/files: new `latent-app/src/gui/dialogs.rs` (the error modal), `latent-app/src/gui/app.rs` (render the modal from `LoadState::Error`, wire Retry/Open-another/Dismiss), `latent-app/src/main.rs` (the GUI path no longer maps `gui::run`'s error to `exit(1)`; the CLI `develop` path keeps it).
- Depends on: U8-C1 (the `Error` load state + async develop), U4-C1 (Open-another needs the in-app open).            Blocks: —
- Heads-up:
  - **A bad or unreadable RAW must show a friendly, dismissable modal inside the running app — not vanish to stderr and `process::exit(1)`. Today the GUI has *two* fatal-error doors: (1) `gui::run` returns `develop_to_image`'s `Err` via `?` (`gui.rs:29`) before the window opens, and (2) `main` turns any `Err` from `gui::run` into `eprintln!("error: {e}")` + `std::process::exit(1)` (`main.rs:116-119`). U8-C1 already closed door (1) by moving develop off-thread into the `LoadState::Error` arm; this card renders that arm as a modal and closes door (2) for the GUI.**
  - **`main`'s error path splits by command (the precise change).** `main` (`main.rs:106-119`) runs one `match` over `Command::{Develop, Open}` and funnels both into the shared `if let Err(e) = result { eprintln!; exit(1) }`. Change it so **only `Command::Develop`** keeps the `eprintln!` + `exit(1)` (a CLI tool *must* exit non-zero on failure — `develop_errors_on_bad_input`, `main.rs:190`, and the `tests/cli.rs` exit-code contract depend on it). `Command::Open` (the GUI) should **not** exit(1) on a *decode* error — that error now surfaces as the in-app modal (U8-C1's `Error` state), so `gui::run` should return `Ok(())` for a recoverable decode failure and only return `Err` for an *unrecoverable* window/platform failure (the existing "could not start the editor window" map at `gui.rs:82`, which legitimately still exits non-zero — there's no UI to show a modal in). Concretely: the decode `Err` no longer escapes `run` via `?` (U8-C1 routed it into `LoadState::Error`); `run` only returns `Err` for `run_native` itself failing. So `main`'s shared exit path stays correct *because* `run` stops returning decode errors — but make the intent explicit and documented (a comment at `main.rs:114-119` stating the GUI shows decode errors in-app; the CLI exits). Do **not** change `develop`'s behavior.
  - **The modal (`dialogs.rs`).** Render when `LoadState::Error { path, message }`. Use an egui modal: `egui::Modal` if the pinned 0.32.3 API gives the dim-backdrop + focus-trap cleanly; otherwise a full-screen dimmed `egui::Area` (semi-transparent scrim over the whole viewport) with a centered `egui::Window::new("Couldn't open file").collapsible(false).resizable(false).anchor(Align2::CENTER_CENTER, …)`. Content: a friendly headline ("Couldn't open <filename>"), the underlying `message` (the `develop_to_image` error text — "camera color matrix is singular", a LibRaw unpack error, etc.) in a smaller/mono style, and three buttons:
    - **Retry** — re-spawn the `DevelopJob` for the same `path` (flip back to `LoadState::Loading`); for a transient FS hiccup this just works.
    - **Open another…** — invoke U4-C1's native Open (rfd) to pick a different RAW and develop *that* (this is the dependency on U4-C1 — without an in-app open there's nowhere to go but quit).
    - **Dismiss** — flip to `LoadState::NoImage`, i.e. the welcome / empty state (U4-C5), so the app stays alive and reachable with no file open.
  - **The empty state must remain reachable (the gotcha).** After Dismiss the app is in `NoImage` — that *is* U4-C5's welcome screen, so the two must agree on the state (U8-C1 defined `NoImage`; U4-C5 draws it). Don't leave the app in a dead `Error` state with no exit; every modal button leads somewhere live (`Loading`, a new `Loading`, or `NoImage`). If U4-C5 isn't landed yet, `NoImage` can be a minimal "No file open — File → Open…" placeholder that U4-C5 fleshes out (mark `// U4-C5`).
  - **Cross-cutting / test:** the modal draw is display-unverifiable, but the *routing* is testable and is the point. A **test** `bad_input_yields_error_state_not_exit` constructs the develop result for a garbage file (reuse U8-C1's `DevelopJob` error arm) and asserts `on_developed` lands the app in `LoadState::Error { message, .. }` (not a panic, not an exit) — this is the regression guard that the GUI no longer dies on a bad RAW. Keep `main`'s CLI exit-code tests (`main.rs:190`, `tests/cli.rs`) green: assert the `develop` subcommand still returns `Err` (→ exit 1) on bad input, unchanged.
- Acceptance:
  - Opening a corrupt / unreadable RAW in the **GUI** shows a centered, dismissable modal with the error message and Retry / Open-another / Dismiss; the app stays alive and the welcome/empty state (`NoImage`) is reachable via Dismiss. Verified manually with a garbage file.
  - `main`'s error handling is documented as split: `Command::Develop` keeps `eprintln!` + `exit(1)`; `Command::Open` shows decode errors in-app (no `exit(1)` for a recoverable decode failure — `gui::run` returns `Ok` for those, `Err` only for an unrecoverable window/platform failure). `develop`'s behavior is unchanged.
  - **Tests:** `bad_input_yields_error_state_not_exit` (GUI routes a bad RAW to `LoadState::Error`, no exit); the existing `develop_errors_on_bad_input` (`main.rs:190`) and `tests/cli.rs` exit-code behavior stay green (CLI still exits non-zero).
  - Baseline green (fmt/clippy zero-warnings/test); no pixel change.

### U8-C4 — Export / render progress indication
- Implements: board "states & polish"; make the already-correct "busy" state *visible*            Priority: Low
- Crates/files: `latent-app/src/gui/app.rs` (read `render.is_busy()` / the export-in-flight fact), `latent-app/src/gui/panels/statusbar.rs` (the in-progress affordance).
- Depends on: U8-C2 (export-done reports via a toast), U4-C4 (the export dialog is where determinate progress would live).            Blocks: —
- Heads-up:
  - **A long full-res export currently goes silent: `export` sets `status = "Exporting…"` (`gui.rs:284`) and disables the Export button via `add_enabled(!self.render.is_busy(), …)` (`gui.rs:481-485`), but there's no spinner and the only feedback is one easy-to-miss label. Surface the in-progress state visibly. The pipeline cannot report *fractional* progress today (the render/export is a single opaque `render(...)` + `save_auto(...)` call in `RenderJob::run`, `gui.rs:192-201` — no callback, no percent), so an **indeterminate** spinner + "Exporting…" is the honest and accepted affordance. Do not fake a determinate percent.**
  - **What's already true (don't re-implement, surface it).** The off-thread export already exists (U0/the worker) and the busy gate already exists: `RenderState::is_busy()` (`gui.rs:163`) is true while *any* job (preview or export) is in flight, and the Export button is already disabled in that state. The missing piece is *visibility*. Add to the status bar (U0-C5's `statusbar.rs`): when an **export** is in flight, an `egui::Spinner` + "Exporting…" (and `ctx.request_repaint()` so it animates). Distinguish an export from a routine preview re-render so a 30ms preview render doesn't flash "Exporting…" — the cleanest way is a small flag/`JobKind`-aware bit: when `export()` spawns, set an `exporting: bool` (cleared when the `RenderOutput::Export` arm is consumed in `poll_render`, `gui.rs:316`), and show the export affordance off *that*, not raw `is_busy()`. A routine preview render can show a subtler "Rendering…" steady-state indicator (the persistent status line from U8-C2), reserving the prominent spinner for the long export.
  - **The disabled Export is the affordance, made obvious.** The acceptance asks that the in-progress Export action be *visibly* disabled (it already is functionally — `add_enabled(false, …)`). Make it read as in-progress: while exporting, the Export button label can become "Exporting…" with the spinner beside it (or a disabled button + the status-bar spinner), so the user sees *why* it's disabled, not a dead grey button. If U4-C4's export *dialog* has landed, the indeterminate spinner lives in the dialog too (the dialog stays open showing "Exporting…" until the `Export` toast from U8-C2 arrives, then closes) — that's the U4-C4 coupling.
  - **On completion → toast (the U8-C2 hand-off).** When the export finishes, the result is reported by U8-C2's toast (success names the path, failure is an error toast), and the spinner/"Exporting…" clears. So this card is mostly *wiring the in-flight visibility*; the completion feedback is already C2's job — don't add a second completion label.
  - **Note for the future:** if the pipeline ever grows a progress callback (it can't today), the determinate bar slots in where the spinner is — leave the affordance a single function (`statusbar` reads an `Option<f32>` progress that's always `None`/indeterminate now) so a future determinate source is a one-line change. Mark it `// determinate when the pipeline reports progress`.

## Epic done when

The app is honest about every state it's in, with no UI-thread blocking and no process-killing on a bad file. Opening any RAW — including a large one — **opens the window immediately** and develops on the worker thread behind a spinner (C1), having moved the synchronous pre-window `develop_to_image` (`gui.rs:29`) onto an additive `DevelopJob` / `RenderOutput::Developed` path and the app onto an explicit `NoImage`/`Loading`/`Ready`/`Error` load-state machine that U4-C5's welcome and U8-C3's modal both branch on. Transient outcomes — export finished, sidecar save failed, GPU fallback — surface as stacking, auto-dismissing **egui-native toasts** (C2), while the status bar keeps a persistent steady-state line; the bare `status: String` is split into those two homes. A corrupt or unreadable RAW shows a **friendly modal** (Retry / Open-another / Dismiss) and the app stays alive — the GUI no longer `process::exit(1)`s on a decode error, while the CLI `develop` path keeps its exit-code contract (C3). A long export shows a visible **indeterminate spinner + "Exporting…"** with the Export action visibly disabled-because-busy, the completion reported by a toast (C4). **Help→About** reports the app version, the linked **lensfun** version (`latent_lens::version`), and the active backend, and a subtle **first-run hint** points new users at Open / drag-drop exactly once, remembered in the app config (C5). Throughout, the off-thread render/export worker is reused (not rewritten), no new crate is added (Global rule 6), and the rendered image is byte-identical (Global rule 2 — `to_color_image_matches_the_export_transform` stays green); each change is pinned by a pure, window-free test.
