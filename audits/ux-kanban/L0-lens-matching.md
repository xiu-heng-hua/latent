# L0 — Lens database matching (brand-agnostic)

> Independent correctness fix — **do first** (board sequencing step 1, [`README.md`](README.md) §"Recommended sequencing"). The user is actively hitting it: a Nikon body + Nikkor never matches because lensfun's camera lookup is an *exact* string compare and EXIF strings don't match the DB spelling. This epic fixes it the way lensfun and LibRaw themselves intend — fuzzy camera resolution + LibRaw's normalized vendor names — with **no per-vendor string table anywhere** (Product decision 5, [`README.md`](README.md) §"Product decisions"). The epic stays **independent of the U\* epics**; it *feeds* **U6-C5** (the opt-in lens-correction panel), whose detect-on-enable path relies on detection actually working.
>
> **Shared surface:** the lens lookup in [`latent-lens/src/lib.rs`](../../latent-lens/src/lib.rs) (`Database::find_profile`, ~`:79-116`), the EXIF metadata it is fed — [`latent-raw/src/lib.rs`](../../latent-raw/src/lib.rs) (`Metadata` ~`:166-209`, read in `read_metadata` ~`:832-850`) — and the one consumer that wires the two together, [`latent-app/src/gui.rs`](../../latent-app/src/gui.rs) (`auto_lens_profile` ~`:89-102`). **C1** fixes the matcher inside `find_profile`; **C2** feeds it cleaner input from LibRaw. The two cards are independent of each other (different crates, no shared signature change) and can land in either order; both are needed for a reliable match. Take C1 first — it is the headline bug.
>
> **Authoritative references** (lensfun source, on GitHub at `libs/lensfun/database.cpp`; the implementer should re-confirm the function bodies there, and the LibRaw `libraw_iparams_t` doc for the normalized fields):
> - `lfDatabase::FindCameras` (the C entry `lf_db_find_cameras`) is **EXACT**: it sorts/compares with `__find_camera_compare` → `_lf_strcmp` on `Maker` and `Model`. EXIF `"NIKON CORPORATION"` is never byte-equal to DB `"Nikon"`, and EXIF model `"D7000"` is never equal to DB `"Nikon D7000"`. **This is the bug.**
> - `lfDatabase::FindCamerasExt` (the C entry `lf_db_find_cameras_ext`) is **FUZZY**: it scores candidates with `lfFuzzyStrCmp` and returns them best-first. lensfun's own reference tool `apps/lenstool/lenstool.cpp` calls `FindCamerasExt(maker, model)` and takes `cameras[0]` — this is lensfun's intended way to absorb EXIF maker/model spelling differences, brand-agnostically, with no per-vendor table.
> - LibRaw's `idata.normalized_make` / `idata.normalized_model` are *standardized* vendor/model names, **always populated**, that map rebrands onto the primary vendor and clean EXIF strings (e.g. `"NIKON CORPORATION"` → `"Nikon"`). They are the upstream half of the same idea: feed the matcher names that are already close to the DB spelling.

### L0-C1 — Fuzzy camera resolution
- Implements: Product decision 5 (brand-agnostic matching)            Priority: High
- Crates/files: [`latent-lens/src/lib.rs`](../../latent-lens/src/lib.rs) (`Database::find_profile`, `:94-116`)
- Depends on: —            Blocks: **U6-C5** (the lens panel's detect-on-enable relies on detection working)
- Heads-up:
  - **The whole bug lives in one call.** `find_profile` (`:94-116`) calls `ffi::lf_db_find_cameras(self.db, maker, model)` at `:95`. That is lensfun's `FindCameras`, an **exact** `_lf_strcmp` on `Maker` and `Model` (`__find_camera_compare` in `database.cpp`). The user's EXIF `"NIKON CORPORATION"` / `"D7000"` never byte-equals the DB's `"Nikon"` / `"Nikon D7000"`, so the camera list comes back empty, `find_profile` returns `None` at `:100`, and no lens is ever looked up. Swap the call for the **fuzzy** `ffi::lf_db_find_cameras_ext(self.db, maker, model, 0)` — lensfun's `FindCamerasExt`, which scores with `lfFuzzyStrCmp` and returns matches best-first. **Keep taking the first/best result** (`let camera = *cameras;` at `:102`), exactly as lensfun's own `apps/lenstool/lenstool.cpp` does (`FindCamerasExt(...)` then `cameras[0]`). Both entry points are **already bound** in the generated bindings; `lf_db_find_cameras_ext(db, maker, model, sflags: c_int) -> *mut *const lfCamera` differs from `lf_db_find_cameras` only by the trailing `sflags` — pass **`0`** (no special search flags, the lenstool default). No new FFI, no `build.rs` change.
  - **Document *why* fuzzy, in the call-site comment** (not this board — Global rule 8 forbids referencing it). State that `lf_db_find_cameras`/`FindCameras` is an exact `_lf_strcmp` that the EXIF↔DB spelling gap defeats, and that `lf_db_find_cameras_ext`/`FindCamerasExt` is lensfun's fuzzy (`lfFuzzyStrCmp`) matcher — the upstream-intended way to absorb maker/model spelling, used by `lenstool`. Name the lensfun functions so the reasoning is checkable. **No per-brand strings** — no `"NIKON"`, no `.replace("CORPORATION", …)`, nothing vendor-specific; the cleanliness of this fix is the point of Product decision 5.
  - **Model-only fallback (the second half of the robustness story).** A fuzzy maker-qualified search can still come back empty when the *maker* string carries a suffix or extra word the scorer won't bridge. `lenstool` guards against exactly this by allowing a `NULL` maker. So: when `lf_db_find_cameras_ext(db, maker, model, 0)` yields an empty/`NULL` list, retry once with the maker as `std::ptr::null()` (model only) before giving up. A maker-suffix/extra-word mismatch then can't block a match that the model alone would find. Keep the model qualified (don't also null the model) — a bare-model search across the whole DB is what the maker hint exists to avoid.
  - **RAII discipline is load-bearing and must be preserved byte-for-byte.** `find_profile` frees **on every path**: the early-return at `:96-101` frees `cameras` when the list is non-NULL-but-empty (`if !cameras.is_null() { lf_free(cameras …) }`), then `:105` frees `cameras` after extracting `camera`, and the lens block (`:106-114`) mirrors the same NULL-check-then-`lf_free` for `lenses`. The fallback retry adds **one more** allocation that returns `*mut *const lfCamera` and must be freed on *its* every path too — both when it is empty (free + return `None`) and when it succeeds (free after `let camera = *cameras;`). Do not let the first (empty) result leak when you retry: free it before the second call. Mirror the existing pattern exactly; the safety comment at `:91-93` (arrays NULL-terminated, owned by lensfun, freed with `lf_free`) still holds and should still describe the code.
  - The lens half (`lf_db_find_lenses_hd`, `:103-104`) is **out of scope** — `find_lenses_hd` already takes the resolved `camera` and a lens string and is not the failing link. Touch only the camera-resolution step.
- Acceptance:
  - The camera lookup uses `lf_db_find_cameras_ext` with `sflags = 0`, keeps `cameras[0]` as the chosen camera, and adds the model-only (`NULL` maker) retry on an empty maker-qualified result. **No per-brand strings** appear anywhere in the diff.
  - The early-return / `lf_free`-on-every-path plumbing is structurally unchanged (every allocated camera/lens list is freed on every exit, including the new retry's list), and the existing **`find_profile_returns_none_when_no_match`** test (`:467-497`) still passes **with or without a DB installed** — it degrades gracefully when `Database::load` errors (no data package), and on an installed DB still drives the nonsense-camera early-return and the NUL-byte `CString` path to `None` with no leak and no crash. The fuzzy swap must not regress the "nonsense camera → `None`" guarantee (a fuzzy scorer must still reject `"no-such-maker"`/`"no-such-model"`).
  - A **documented manual check** that a real Nikon body + Nikkor now resolves where it didn't before — the live DB query is exercised by hand, like real-RAW decode, since the database is an external CC-BY-SA package not vendored into the repo (the crate doc at `:74-78` already frames the lookup this way). Note in the card/commit that this path is manually verified, not unit-pinned.
  - `cargo fmt --check`, `cargo clippy --workspace --all-targets` (zero warnings), `cargo test --workspace` (313 passing) stay green (Global rule 1).

### L0-C2 — Use LibRaw normalized make/model for lookup
- Implements: Product decision 5 (brand-agnostic matching, upstream half)            Priority: Medium
- Crates/files: [`latent-raw/src/lib.rs`](../../latent-raw/src/lib.rs) (`Metadata` `:166-209`, `read_metadata` `:832-850`); [`latent-app/src/gui.rs`](../../latent-app/src/gui.rs) (`auto_lens_profile` `:89-102`)
- Depends on: —            Blocks: —
- Heads-up:
  - **Feed the matcher pre-cleaned names.** LibRaw exposes `idata.normalized_make` / `idata.normalized_model` — standardized vendor/model strings that are *always populated* and map rebrands to the primary vendor, turning EXIF `"NIKON CORPORATION"` into `"Nikon"`. Reading those for the lensfun lookup makes the match easier independently of C1's fuzzy matcher (the two compound: cleaner input *and* a fuzzy compare). This is the same brand-agnostic principle, sourced upstream — **no per-vendor logic in our code**, LibRaw does the normalization.
  - **Add the fields, keep the raw EXIF for display.** Add `normalized_make: String` and `normalized_model: String` to `Metadata` (`:199-204`, alongside `make`/`model`/`lens`). Read them in `read_metadata` via the **same `c_str_field` helper** (`:787-794`, lossy-UTF-8, NUL-terminated, trimmed) the existing `make`/`model` reads use at `:845-846` — `make: c_str_field(&idata.make)` gains siblings `normalized_make: c_str_field(&idata.normalized_make)` / `normalized_model: c_str_field(&idata.normalized_model)`. **Keep `make`/`model` as the raw EXIF** — they are documented as "as in EXIF" (`:199-202`) and are what a future UI shows the user; the normalized strings are a *lookup aid*, not a replacement.
  - **Bindings are confirmed.** The generated `libraw_iparams_t` carries `pub normalized_make: [c_char; 64]` and `pub normalized_model: [c_char; 64]` (verified in `target/.../latent-raw-*/out/bindings.rs`, fields immediately after `make`/`model`), so `c_str_field(&idata.normalized_make)` type-checks against the same `&[c_char]` slice signature. `idata` is already in scope as `let idata = unsafe { &(*raw).idata };` (`:814`). No `build.rs`/bindgen allowlist change is needed if these already appear in the bindings — confirm they do before assuming, then read them.
  - **`Metadata` does NOT derive `Default`** — its derive is `#[derive(Debug, Clone)]` (`:165`; the `Default` derive at `:107` is on `CblackPattern`, a different struct — don't be fooled). And there is **no `..Default::default()`** spread anywhere; the **four** test `Metadata { … }` literals (at `:873`, `:1120`, `:1203`, `:1264` — each names *every* field, with `make: String::new()` at `:886`, `:1133`, `:1216`, `:1264`) will not compile until the two new fields are added to each. So either add `normalized_make: String::new(), normalized_model: String::new()` to all four literals, **or** first add `Default` to `Metadata`'s derive and convert the literals to `..Default::default()` (cleaner, but a bigger touch — and `Metadata` already derives `Eq` so every field must stay `Eq`/`Default`-able; `String` is, so adding `Default` is safe). Pick the smaller diff that keeps tests compiling; the literals-update path is the safest. This is the only way the new fields "don't break the existing Metadata tests" (the card's explicit constraint).
  - **Prefer normalized, fall back to raw, in the one consumer.** In `auto_lens_profile` (`gui.rs:89-102`) the lookup args at `:94-101` are `&meta.make`, `&meta.model`, `&meta.lens`. Pass the **normalized** make/model when non-empty, falling back to the raw EXIF when a normalized field is empty (a defensive guard even though LibRaw "always populates" them — an empty string would otherwise null the maker hint and force C1's whole-DB model search). A small `if normalized.is_empty() { raw } else { normalized }` per field, no helper crate. The empty-metadata short-circuit at `:90-92` (`meta.model.is_empty() && meta.lens.is_empty()`) stays keyed on the *raw* EXIF `model` — it is a "did we read anything at all" gate, unchanged. `lens` is unaffected (LibRaw has no normalized lens string).
  - This card is independent of C1: it changes *what strings* reach `find_profile`, not *how* `find_profile` matches. Either order works; both compound to make a reliable match.
- Acceptance:
  - `Metadata` carries `normalized_make: String` and `normalized_model: String`, read in `read_metadata` through `c_str_field` from `idata.normalized_make` / `idata.normalized_model`; `make`/`model` remain the raw EXIF strings (doc comments at `:199-202` unchanged in meaning).
  - The new fields don't break the Metadata tests: all four `Metadata { … }` literals (`:873`/`:1120`/`:1203`/`:1264`) construct successfully (either by listing the two new fields or via a newly-derived `Default` + spread), and `cargo test --workspace` stays at full green (313 passing).
  - `auto_lens_profile` passes the **normalized** strings to `find_profile` (raw-EXIF fallback when a normalized field is empty); the empty-metadata short-circuit is preserved.
  - The bindings genuinely expose `normalized_make`/`normalized_model` on `libraw_iparams_t` (verified, not assumed); the workspace **builds** and `cargo fmt --check` / `cargo clippy --workspace --all-targets` (zero warnings) / `cargo test --workspace` are all green (Global rule 1).
