# E0 — Color-management core & perceptual module

*Foundational epic.* The standard-conformant rebuild of the color core plus the shared `Lab`/`LCh` + `L*` perceptual module that every perceptual operation downstream depends on. Implements the [Audit 01 color-science decision register](../decisions/01-color-science.md) (`IP-01 #1…#9`).

**Decision context:** `IP-01 #2`, `#4a`, and `#9` are mutually reinforcing — together they are a deliberate redesign of the color core toward a standard DNG/ICC pipeline (D50 reference white + explicit Bradford chromatic adaptation), not three independent tweaks. `#3`, `#6`, `#8` are confirmed-correct no-ops and are left unchanged (keep their guarding tests green).

**Sequencing inside the epic:** build the matrix/CA infrastructure first — **E0-C2, E0-C3, E0-C4** (full-precision sRGB matrix, D50 working space, Bradford helper) — *before* **E0-C5/E0-C6** (the DNG camera model and the adapted output matrix), which consume them. **E0-C1** (the Lab/LCh module) is independent of the matrix rebuild and can proceed in parallel, but it is the keystone shared artifact consumed by E1 and E5, so land it early. **E0-C7** (highlight rolloff) is independent of the matrix chain.

> Reminders from the [board global rules](README.md#global-rules-apply-to-every-card): CPU↔GPU lockstep (Initiative G) — any new perceptual math (L\*, LCh) added on CPU must be mirrored in WGSL and keep the GPU/CPU render-equivalence test green; a test pins every change; baseline (`cargo fmt --check`, `cargo clippy --all-targets` zero warnings, `cargo test --workspace`) stays green after each card; the perceptual lightness L\* has a **single owner** (E0-C1) — other cards depend on it rather than re-implementing.

### E0-C1 — Lab/LCh module (working→XYZ→Lab→LCh + inverse, D50) and exposed `L*` perceptual lightness
- Implements: `IP-01 #5`            Priority: Critical
- Crates/files: `latent-image/src/color.rs` (new `Lab`/`LCh` types + conversions); a WGSL mirror of the L\* transfer for the GPU perceptual path (e.g. `latent-gpu/.../*.wgsl`); reference helper in `latent-pipeline`.
- Depends on: — (independent of the matrix rebuild; only needs `linear_working_to_xyz`, which exists)            Blocks: E0-C5 (saturation→LCh path), E1-C1/C5/C6 (L\* tone domain, LCh hue mixer, LCh saturation), E5-C3/C4 (perceptual luma sharpen, Lab chroma denoise)
- Heads-up: **This is the keystone shared artifact of the whole perceptual plan** — the `Lab`/`LCh` module and the single `L*` perceptual lightness it exposes are *consumed by epics E1 and E5*; they must depend on this owner rather than re-deriving lightness (global rule "one place per concern"). Implement working→XYZ→Lab→LCh and the full inverse, and **reference everything to D50**: CIE Lab is natively a D50-referenced space, and per `IP-01 #2` the working space is being moved to D50 ProPhoto (E0-C3), so the natural white for the XYZ→Lab step is D50 — no extra adaptation belongs inside Lab. Use the standard CIE Lab forward/inverse with the `δ = 6/29` linear-segment cube-root companding; LCh is just the polar form of (a\*, b\*) so `C* = hypot(a,b)`, `h = atan2(b,a)`. Expose `L*` as *the* single perceptual lightness (the register reconciled the old Rec.709-luma idea to L\* precisely so there is one definition across the grading core — see decision `#5` and README → *Consistency reconciliation*). Gotcha: keep the working→XYZ at unit luminance so `L*` of working white is 100; guard the cube-root branch and `atan2` for the achromatic/`C*≈0` case so hue is well-defined (return 0). **CPU↔GPU lockstep:** the L\* transfer added on CPU must be mirrored in the WGSL shader and exercised by the reference helper, keeping the GPU/CPU render-equivalence test green (global rule 1). Note this card does **not** itself depend on the D50 working-space rebuild landing first — but if E0-C3 lands after, re-verify the D50 white constant is shared, not duplicated.
- Acceptance: working→XYZ→Lab→LCh→Lab→XYZ→working round-trips to `<1e-4` for a spread of in-gamut colors; working white has `L*≈100`, `a*≈b*≈0`; an achromatic/near-gray pixel yields `C*≈0` with a finite (non-NaN) hue; LCh↔Lab polar conversion round-trips. Named tests (new): `lab_roundtrips_working_to_lab_and_back`, `working_white_has_l_star_100_and_neutral_ab`, `lch_is_polar_form_of_lab_and_roundtrips`, `achromatic_pixel_has_zero_chroma_and_finite_hue`. CPU/GPU L\* parity asserted by the render-equivalence harness.

### E0-C2 — Full-precision XYZ→linear-sRGB matrix (derive from Rec.709 primaries)
- Implements: `IP-01 #1`            Priority: High
- Crates/files: `latent-image/src/color.rs` (`XYZ_TO_LINEAR_SRGB` at `color.rs:89-93`, `xyz_to_linear_srgb`)
- Depends on: —            Blocks: E0-C6 (drop-row-norm output matrix relies on this)
- Heads-up: Replace the IEC 4-decimal ("8-bit grade") `XYZ_TO_LINEAR_SRGB` constant with a full-precision matrix. **Prefer deriving it from the Rec.709/sRGB primaries via the same `rgb_to_xyz` used for the working space** (`color.rs:107-123`) and inverting — this gives a *single source of truth* and is consistent with the working-space construction, rather than transcribing a 7-decimal constant. The Rec.709 primaries are R `(0.640, 0.330)`, G `(0.300, 0.600)`, B `(0.150, 0.060)` at D65 `(0.3127, 0.3290)`; `rgb_to_xyz(rec709, D65).inverse()` is your XYZ→linear-sRGB. Why it matters: the 4-decimal matrix is off by up to `3.7e-4` (~24 LSB at 16-bit) and does not map D65 white *exactly* to neutral, and `latent` exports true 16-bit so the precision is visible; it is also the root cause of the `#4b` row-normalization hack. **Key consequence to note for E0-C6:** because this matrix is now derived from primaries with the *same* `rgb_to_xyz`, the working→sRGB product will have **unit row sums natively** — so E0-C6 can simply drop `row_normalized()` instead of hiding a rounding tint. Keep `rgb_to_xyz` itself unchanged (decision `#3`, verified correct). Doc ref: `docs/color-srgb-iec61966-2-1-amd1-bgsrgb.pdf`.
- Acceptance: D65 white `[0.95047, 1.0, 1.08883]` (or the chromaticity-derived white) maps to neutral linear sRGB to `<1e-5` (tighter than the current `1e-3`); black maps to black; the derived matrix matches the published full-precision IEC matrix to `<5e-5`. Named tests: tighten `d65_white_maps_to_neutral_srgb` to the new tolerance; add `xyz_to_srgb_matrix_matches_rec709_primaries` asserting the derived constant against an inverse-`rgb_to_xyz(rec709,D65)` reference; keep `black_maps_to_black`.

### E0-C3 — D50 ProPhoto working space + rebuilt matrices + `LUMA_WEIGHTS`
- Implements: `IP-01 #2`            Priority: High
- Crates/files: `latent-image/src/color.rs` (working-space construction `color.rs:125-136`, `linear_working_to_xyz`, `xyz_to_linear_working`, `LUMA_WEIGHTS` at `color.rs:181`, the `LUMA_WEIGHTS`/GPU note at `color.rs:171-180`)
- Depends on: —            Blocks: E0-C4/C5/C6 (all matrices reference the new working white), E0-C1 (shares the D50 white constant)
- Heads-up: Move the working space from "ROMM primaries pinned at D65" to **standard ROMM/ProPhoto RGB at its native D50 reference white**. Add a `D50_WHITE` chromaticity constant `(0.3457, 0.3585)` and use it in place of `D65_WHITE` for `linear_working_to_xyz`; the ProPhoto primaries stay as-is. This makes the working space a real, interoperable ICC/ISO-identifiable space and lets Lab (E0-C1) reference it natively at D50. **This card changes `LUMA_WEIGHTS`** — the Y row of the rebuilt `linear_working_to_xyz` is the D50-ProPhoto relative-luminance vector, so recompute the const from the new matrix (do not transcribe; the test cross-checks it against the matrix). Update the comments that still describe the D65 rationale ("no chromatic adaptation needed anywhere") — that premise is now *false*: the trade-off accepted here is re-introducing chromatic adaptation (handled by E0-C4/C5/C6). Gotcha: the blue luma weight gets *even smaller* at D50 ProPhoto than the current `~0.0001`; that is colorimetrically correct (keep the by-design note) but is exactly why perceptual ops must use L\* (E0-C1), not this luma. **Flag for the implementer:** two existing unit tests will fail and must be updated to the D50 expectations — `working_space_white_is_neutral_and_round_trips` (now expects D50 white in XYZ, not D65) and `luma_weights_match_the_working_matrix` (the Y row changed). Doc ref: `docs/color-romm-rgb-iso22028-2.pdf`.
- Acceptance: `linear_working_to_xyz().mul_vec([1,1,1])` equals the D50 white in XYZ to `<1e-4`; XYZ→working→XYZ round-trips to `<1e-5`; `LUMA_WEIGHTS` equals the Y row of the rebuilt matrix and sums to 1. Named tests (updated): `working_space_white_is_neutral_and_round_trips` → D50; `luma_weights_match_the_working_matrix` → new Y row.

### E0-C4 — Bradford chromatic-adaptation helper
- Implements: `IP-01 #9`            Priority: High
- Crates/files: `latent-image/src/color.rs` (new Bradford CA helper, e.g. `bradford_adapt(src_white, dst_white) -> Mat3`)
- Depends on: E0-C3 (so the D50 reference white exists as the canonical target)            Blocks: E0-C5 (WB-illuminant→D50 adaptation), E0-C6 (D50→D65 output adaptation)
- Heads-up: Add a standard **Bradford** chromatic-adaptation helper: given a source and destination white (as XYZ, derived from chromaticities via the same `to_xyz` used in `rgb_to_xyz`), return the 3×3 adaptation matrix `M_bfd⁻¹ · diag(ρ_dst/ρ_src, …) · M_bfd`, where `M_bfd` is the standard Bradford cone-response matrix. This is a *direct consequence* of E0-C3 + E0-C5: once the working white is D50 and WB is handled by the DNG model, adaptation is no longer the identity and must be performed explicitly — (a) from the capture/WB illuminant to the reference white inside `camera_to_working` (E0-C5), and (b) from D50-working to D65-sRGB in the output transform (E0-C6). Build it as a *reusable* helper with the published Bradford `M_bfd` constant; do not fold it into either call site. Gotcha: adapting a white to itself must be the identity (within tolerance) — use this as the cheap self-check. Doc ref: the DNG spec `docs/color-dng-spec-1.4.0.0.pdf` describes the same Bradford-style adaptation in the ForwardMatrix path. This card has no behavioral wiring on its own (the matrices that consume it are E0-C5/C6); its test stands alone.
- Acceptance: `bradford_adapt(W, W)` is the identity to `<1e-6` for several whites (D50, D65); adapting D65 XYZ white through `bradford_adapt(D65, D50)` yields the D50 XYZ white to `<1e-4`, and the round-trip `D50→D65→D50` is the identity to `<1e-5`. Named tests (new): `bradford_self_adaptation_is_identity`, `bradford_maps_d65_white_to_d50_white`, `bradford_roundtrips`.

### E0-C5 — Full DNG camera→working model (inverse-ColorMatrix + Bradford WB; drop row-norm + comment; fix decode order)
- Implements: `IP-01 #4a`            Priority: Critical
- Crates/files: `latent-image/src/color.rs` (`camera_to_working` at `color.rs:158-169`, remove the misleading comment at `color.rs:161-165`); `latent-raw/src/lib.rs` (`apply_white_balance` ~`:188-207`, `color_matrix` ~`:425-429` and its row-norm docstring); `latent-app/src/main.rs` (`develop_to_image` decode order ~`:57-72`)
- Depends on: E0-C3 (D50 reference white), E0-C4 (Bradford helper), E0-C1 (so saturation can move to LCh per `#5` rather than re-using luma)            Blocks: E8 output stability (the developed image feeds the export transform)
- Heads-up: **Highest-risk, highest-impact card in this register.** Replace the "white-balance the mosaic by `cam_mul`, then row-normalize the camera→working matrix" approach with the full **Adobe DNG color model**: invert the file's `ColorMatrix` to get camera→XYZ, apply white balance by *scaling camera coordinates* (the diagonal WB), and Bradford-adapt from the white-balance illuminant to the D50 reference white (the DNG `ForwardMatrix`-equivalent path), then XYZ→working (E0-C3). Why: row-normalization keeps neutrals correct but is *not* the reference operator — saturated colors drift up to **0.28 in linear working RGB** for a real Canon profile (visible, camera/WB-dependent); the DNG model renders chromatic colors correctly. **The hard invariant: white balance must be applied exactly once.** Today WB is applied on the mosaic in `apply_white_balance` *and* the matrix is row-normalized to stop the matrix re-applying its own implicit WB — that double-defense must collapse into a single, explicit WB in the DNG model. Decide and document the one place WB lives (in the camera→XYZ/WB-scaling step), and **revisit the decode order** in `develop_to_image` (`latent-app/src/main.rs`) accordingly — if WB moves into the color transform, it must come out of the mosaic pre-pass (or vice-versa), but not both. **Remove the misleading "double-WB fix" / row-norm comment** at `color.rs:161-165` and the matching row-norm rationale in `color_matrix`'s docstring in `latent-raw`. Per `#5` (and E0-C1), saturation no longer uses a working-space luma — if you touch the saturation path here, route it to LCh chroma, not `LUMA_WEIGHTS`. Gotchas: keep the existing `camera_to_xyz` inverse-of-`cam_xyz` (decision `#8`, correct); guard singular matrices (return `None`); the `cam_mul` G2-handling in `apply_white_balance` must survive whatever restructuring you choose. **Sanitize-at-boundary:** RAW `cam_mul`/`cam_xyz` are untrusted metadata — validate/guard them on the way in (coordinate with E2) and trust within. Doc ref: `docs/color-dng-spec-1.4.0.0.pdf`.
- Acceptance: a neutral camera patch `[v,v,v]` (post-WB) stays neutral through `camera_to_working` to `<1e-5` (WB applied exactly once — no neutral drift, no double-apply darkening/tint); for a representative camera `ColorMatrix`, a saturated test color matches the DNG-reference camera→working output to a small tolerance (and differs measurably from the old row-normalized result, proving the operator changed); the developed-image decode order produces the same neutral as before. Named tests: keep/extend `camera_to_working_keeps_a_neutral_patch_neutral` and `color_matrix_keeps_a_neutral_patch_neutral` (now under the DNG model); keep `white_balance_neutralizes_a_gray_patch`; add `camera_to_working_matches_dng_reference_for_a_saturated_color` and a `wb_applied_exactly_once` regression (the same neutral does not darken/tint vs a known reference).

### E0-C6 — Working→sRGB output matrix (D50→D65 adapted, full precision, no row-norm)
- Implements: `IP-01 #4b`            Priority: High
- Crates/files: `latent-image/src/color.rs` (`linear_working_to_linear_srgb` at `color.rs:145-156` and its docstring)
- Depends on: E0-C2 (full-precision XYZ→sRGB), E0-C3 (D50 working white), E0-C4 (Bradford D50→D65)            Blocks: E0-C7 (rolloff feeds the same output path), E8 (export/preview output transform)
- Heads-up: Rebuild `linear_working_to_linear_srgb` as the **correct adapted output matrix**, not a row-norm hack. The chain is now: linear working (D50) → XYZ (D50) → **Bradford-adapt D50→D65** (E0-C4) → linear sRGB (E0-C2's full-precision XYZ→sRGB matrix at D65). Compose those into a single `Mat3` at construction. **Drop `row_normalized()`** — with E0-C2 the working→sRGB product has unit row sums *natively* (both ends are now derived from primaries with the same `rgb_to_xyz`), so "drop row-norm" means *replace the hack with the genuinely-neutral adapted matrix*, not just delete a line. Rewrite the docstring: it currently claims a `~1e-4` drift "pinned" by row-normalization — that was understating the real `3.25e-4` shift and is obsolete. Why it matters: the row-norm was masking the rounded-matrix tint rather than fixing it; the adapted matrix is correct end-to-end and exports true neutral at 16-bit. Gotcha: the order matters — adapt *after* working→XYZ and *before* XYZ→sRGB; getting the Bradford direction backwards introduces a warm/cool cast on white. **E8 depends on this matrix being stable** — the export path and live preview both consume it via `latent-export`.
- Acceptance: a neutral working gray `[v,v,v]` maps to an exactly-neutral linear sRGB gray to `<1e-5` (tighten from today's `1e-3`); the matrix rows sum to 1 to `<1e-5` *without* row-normalization; D50 working white maps to D65 sRGB neutral. Named tests: tighten `working_to_srgb_keeps_neutral_neutral` to `<1e-5`; add `working_to_srgb_rows_sum_to_one_without_normalization` and `working_to_srgb_adapts_d50_white_to_d65_neutral`.

### E0-C7 — Hue-preserving highlight rolloff (max/luminance shared factor)
- Implements: `IP-01 #7`            Priority: Medium
- Crates/files: `latent-export/src/lib.rs` (`highlight_rolloff` at `:42-51`, `to_display` at `:56-59`); update the pinned-value tests
- Depends on: E0-C6 (the output matrix it composes with)            Blocks: E8 (export pinned-value stability)
- Heads-up: Change `highlight_rolloff` from **independent per-channel** compression to a **hue-preserving** form: compute *one* compression factor from the triplet (the max channel, or luminance) and scale the whole RGB triplet by that single factor, so the color's hue is preserved as it compresses into display range. Keep the `0.98` knee (that part is correct-by-design). Why: per-channel rolloff compresses only the hot channel of a near-clipped color, shifting hue toward the secondaries in extreme highlights (a magenta/yellow drift on blown highlights). Implementation note: today `to_display` calls `srgb_encode(highlight_rolloff(lin[c]))` per channel *after* the matrix — the rolloff must now operate on the *triplet* before per-channel sRGB encoding: derive `f = rolloff(m)/m` where `m` is the max (or luminance) of the post-matrix linear triplet, scale all three by `f`, then sRGB-encode each. Gotcha: guard `m ≤ 0` (and the `m ≤ knee` pass-through) so the factor is `1.0` and you never divide by zero; below the knee nothing changes, so all existing in-range colors are untouched. **Update the export pinned-value tests** — `save_writes_a_readable_image` currently pins working `1.0`→`254` and headroom `2.0`→`255` (a neutral triplet, so the shared factor reproduces those exact codes — verify they still hold), and `highlight_rolloff_is_identity_below_knee_and_monotone_above` must be re-expressed against the new triplet-scaling form (a neutral input still gives a monotone, identity-below-knee curve). **E8 depends on these pinned values being stable** — pin the new expected codes explicitly so a future rolloff change can't silently drift. Doc ref: rolloff/headroom rationale in the existing `to_display` docstring.
- Acceptance: a saturated near-clipped color preserves its hue (chromaticity/ratio of channels) through the rolloff — the per-channel-vs-shared difference is asserted on a hot color where the old code shifted hue; a neutral triplet still maps `1.0`→`254` and `2.0`→`255`; the rolloff is identity below the knee and monotone above. Named tests: update `save_writes_a_readable_image` (neutral codes still 254/255) and `highlight_rolloff_is_identity_below_knee_and_monotone_above`; add `highlight_rolloff_preserves_hue_on_a_saturated_color`.
