# E8 — Export, CLI & app

> Depends on **E0's output transform** (E0-C6/C7) being stable — the working→sRGB matrix and hue-preserving rolloff feed `to_display`/`to_srgb8` in `latent-export`, which every card here exercises. This epic does **not** touch color math; it covers the export *engineering* — ICC tagging contract, the untagged-fallback hole, the `ImageResult` contract, depth reachability — plus the CLI/GUI plumbing and the off-UI-thread render.
>
> Realizes the **engineering** findings of [`../decisions/code-review/04-export-lens-app.md`](../decisions/code-review/04-export-lens-app.md): **H2** (unknown extension → typed error), **M1/M2** (`save`/`save_16` honor `ImageResult` + zero-dim guard), **L2** (reachable 16-bit, `--depth`), **H1**+**L3** (render off the UI thread, return the texture handle), and the CLI/test gaps of §6. The lens-FFI findings of that same register — **M3** (center normalization) and **M4** (CA/vignetting finiteness guards) — are **E3** (E3-C2 / E3-C1), **not** this epic; do not author them here.
>
> **Shared surface:** `latent-export/src/lib.rs` (`save`, `save_16`, `save_buffer_with_icc`), `latent-app/src/main.rs` (the `Develop` subcommand + `develop`), `latent-app/src/gui.rs` (`render_preview`, `export`, `update`). Take the cards in order: **C1/C2** harden the export contract (typed errors); **C3** makes the 16-bit path reachable and routes by depth; **C4** moves render/export off the egui thread (the most user-visible fix); **C5** adds CLI exit-code/parse/error-path tests and the lens-lookup stub.
>
> **Cross-epic constraints (load-bearing — read before C4):**
> - **E7-C4 makes `Backend: Send + Sync`** (`CR-02 M3`). That bound is what lets the render worker in **C4** hold `&dyn Backend` (or move a backend) across a thread boundary. C4 *depends on* E7-C4; if E7-C4 has not landed, C4 cannot compile the worker. The current `trait Backend` (`latent-pipeline/src/lib.rs:297`) has **no** `Send+Sync` — confirmed.
> - **The lensfun `Database` handle is deliberately NOT `Send`/`Sync`** (`CR-04 §3`; the one-line doc note is added in E3). It must **never** go onto the render worker. The lens *lookup* already runs once, pre-window, on the main thread (`gui.rs:81-94`, `auto_lens_profile`); the resolved `LensProfile` is plain data living in `Settings::geometry.lens` and **is** `Send`. C4 moves the *render* (which reads the already-resolved `LensProfile`), not the DB.

### E8-C1 — Reject unknown export extensions with a typed error
- Implements: CR-04 **H2** (folds in **L4**)            Priority: High
- Crates/files: `latent-export/src/lib.rs` (`save_buffer_with_icc:120-156`, the `_ => buf.save(path)` arm at `:154`); doc note at `:117-119`. GUI affordance: `latent-app/src/gui.rs:312-318` (export path field) — optional inline hint.
- Depends on: —             Blocks: E8-C5 (the test that locks this contract)
- Heads-up:
  - **The headline color-management fix.** The `_ => buf.save(path)` arm (`export/lib.rs:154`, confirmed) is reachable for any extension that isn't `png`/`tif`/`tiff`/`jpg`/`jpeg` — `.bmp`, `.webp`, a typo like `.jpe`, or no extension at all — and writes **with no ICC profile** via the high-level `image::save`. The output color is then untagged and color-managed viewers misinterpret it, with **zero** signal to the user. This app's stated contract is "an sRGB ICC profile is embedded"; emitting an untagged file silently is the worst failure mode it has. Reachable directly from the CLI (`latent develop in.raw out.bmp`) and from the GUI export field.
  - **Decision is option (a): reject, do not warn-and-fall-through.** Replace the `_` arm with a returned `Err`. Use a typed `image::ImageError` so the function keeps its `image::ImageResult<()>` signature and both front-ends surface it unchanged (see *Surfacing*). The cleanest carrier is `image::ImageError::Unsupported(image::error::UnsupportedError::from_format_and_kind(...))` with a `GenericFeature` message, **or** an `ImageError::Parameter` — pick whichever the crate exposes for a clean, message-carrying error; the message is the contract, not the variant. Message must be **actionable**: `unsupported output extension '<ext>'; use png, tif/tiff, or jpg/jpeg`. Include the empty/no-extension case in the **same** error (currently `extension()` → `unwrap_or_default()` yields `""`, which falls into `_`).
  - Do this in `save_buffer_with_icc` (the one place that branches on `ext`), so both `save` and `save_16` inherit it — do not duplicate the match. Also **fix the stale doc** at `:117-119` which says "an unknown extension falls back to an untagged save"; after this card it returns an error, so the comment must say so.
  - **Surfacing — no extra plumbing (verified):** the CLI already maps `Err` → `error: {e}` + `std::process::exit(1)` (`main.rs:87-90`); the GUI `export` already maps `Err` → the status line (`gui.rs:179-182`). So a returned typed error reaches the user on both front-ends with **no** new wiring. (This is why the decision is a contract change, not a plumbing change.)
  - **L4 (GUI affordance, optional within this card):** the export-path field (`gui.rs:312-318`) is free text; once C1 returns a typed error the GUI already shows it on attempt, so the floor is met. The small extra UX win is an inline "unsupported extension" hint / disabled Export button when the typed extension isn't in the supported set — share the *same* supported-set list as `save_buffer_with_icc` (one source of truth; do not hard-code the set twice). Keep this light; the contract test (C5) is the binding deliverable.
- Acceptance:
  - Calling `save`/`save_16` (or `save_buffer_with_icc`) with a `.bmp` / `.webp` / no-extension path returns `Err`, never writes an untagged file. The error message names the bad extension and the supported set.
  - All supported extensions (`png`, `tif`, `tiff`, `jpg`, `jpeg`) still write with the ICC embedded (existing `save_writes_a_readable_image` / `save_*_embeds_a_readable_icc_profile` tests stay green).
  - The `:117-119` doc no longer claims an untagged fallback.
  - **Test** (lands in C5, named here): `save_rejects_unknown_extension` asserts `save(&img, "out.bmp")` is `Err` and that the message mentions `bmp` and `png`; fails before the card (silent `Ok`), passes after. Baseline green (`cargo fmt --check`, `cargo clippy --all-targets` zero warnings, `cargo test --workspace`).

### E8-C2 — `save`/`save_16` honor `ImageResult` + zero-dimension guard
- Implements: CR-04 **M1** (zero-dim guard) + **M2** (map the `from_raw` `.expect` to `Err`)            Priority: Medium
- Crates/files: `latent-export/src/lib.rs` (`save:86-90`, `save_16:95-106`).
- Depends on: —             Blocks: E8-C5 (the 0×0/0×N contract test)
- Heads-up:
  - **Two related contract holes in the same two functions.** Both `save` and `save_16` return `image::ImageResult<()>` but can either panic or silently emit a degenerate file instead of returning `Err`.
  - **M1 — zero-dimension guard (the common case):** for `w==0 || h==0`, `to_srgb8` produces a `Vec<u8>` of length `0` (= `0*0*3`), so `RgbImage::from_raw(0, 0, vec![])` returns **`Some`** and the `.expect` at `:88` does **not** fire — a 0×0 image slips straight through `len == w*h*3` to the encoder, with format-dependent, unvalidated behavior (some encoders error, some write a degenerate file, and the C1 untagged path also becomes reachable for it). `save_16` loops `0..0`, writes an empty `ImageBuffer`, and hands it to the same encoder. Neither is a meaningful output. **Add a guard at the very top of both** `save` (`:86`) and `save_16` (`:95`): if `img.width() == 0 || img.height() == 0`, return a typed `Err` (`ImageError::Parameter` / `Unsupported`) with message `cannot encode an image with a zero dimension`. Put the guard first so it precedes any allocation or matrix work.
  - **M2 — map the `from_raw` `.expect` (the overflow case):** `to_srgb8` builds a `Vec<u8>` of length `w*h*3`; for *pathological non-zero* dimensions `from_raw`'s internal length check can return `None`, tripping the `.expect("buffer length matches the image dimensions")` (`:88`) — a **panic in a library function on a public API** instead of a returned `Err`. Practically unreachable for real RAW (OOM hits first), but it is a genuine contract wrinkle and the fix is trivial: replace the `.expect` with `match RgbImage::from_raw(...) { Some(out) => …, None => Err(ImageError::Parameter(...)) }` (or `.ok_or_else(...)?`), message e.g. `image dimensions overflow the pixel buffer`. The M1 guard already covers the *zero* case; M2 covers the *overflow* case — both are needed, they are disjoint.
  - **Scope discipline:** `save_16` builds its buffer with `ImageBuffer::new(w, h)` + `put_pixel` (no `from_raw`, no `.expect`), so it only needs the M1 guard, not M2. Apply the guard to both; apply the `from_raw` mapping to `save` only.
  - **Keep N2 as-is:** the `srgb_icc()` `.expect("encode sRGB ICC profile")` (`:114`) is a pure, input-independent `moxcms` call and is **verified safe** — do **not** change it. M2 is specifically the *input-dependent* `from_raw` expect; do not conflate them.
  - These guards run before C1's extension check (dimension validity is independent of format), so a 0×0 `.png` should report the zero-dimension error, not the extension. Order: dimension guard → output transform → `from_raw` map → `save_buffer_with_icc` (which does the C1 extension check).
- Acceptance:
  - `save(&img, path)` and `save_16(&img, path)` return `Err` (no panic, no file written) when `img.width() == 0` or `img.height() == 0`.
  - `save`'s `from_raw` failure path returns a mapped `Err` rather than panicking; the `.expect` is gone.
  - Valid images are unaffected (existing `save_writes_a_readable_image`, `save_16_writes_a_true_16bit_file` stay green).
  - **Tests** (land in C5, named here): `save_rejects_zero_dimension` (0×0 and 0×N → `Err`, asserts the message and that no file appears at the path) and, if feasible to construct, a `from_raw` `None` path covered by `save_returns_err_on_unbuildable_buffer` (or documented as overflow-only). Each fails before / passes after. Baseline green.

### E8-C3 — Reachable 16-bit export (`--depth`, auto for tiff/png)
- Implements: CR-04 **L2** (the `Develop` depth/format plumbing)            Priority: Medium
- Crates/files: `latent-app/src/main.rs` (`Command::Develop` args `:25-30`, `develop:73-77`, `main` dispatch `:81-86`); mirror the choice in `latent-app/src/gui.rs:179` (`export`).
- Depends on: E8-C1 (extension is validated/known at routing time), E8-C2 (the depth functions honor their contract)            Blocks: E8-C5 (the depth-routing test)
- Heads-up:
  - **The tested 16-bit path is dead from the CLI.** `develop` (`main.rs:73-75`) **always** calls `latent_export::save` (8-bit), never `save_16`, even for `.tiff` where 16-bit is the entire point (confirmed). `save_16` exists, is tested (`save_16_writes_a_true_16bit_file`, `save_16_embeds_a_readable_icc_profile`), and is **unreachable**. A wide-gamut, highlight-rolled working pipeline (E0-C6/C7) produces gradients that **band at 8 bits** — the high-quality output the rolloff exists for is currently inaccessible.
  - **Add a `--depth` arg and auto-select by format.** On `Command::Develop` (`:25-30`) add `#[arg(long)] depth: Option<u8>` (accept `8` and `16`; an enum `Depth { Eight, Sixteen }` with a clap `ValueEnum` is also fine and gives nicer `--help`). Routing rule in `develop`: if `depth` is given, honor it; otherwise **default to 16 for `tif`/`tiff`/`png`, 8 for `jpg`/`jpeg`** (JPEG has no 16-bit path). Choose `save_16` vs `save` accordingly. Note PNG genuinely supports 16-bit (`Rgb16`), so the default-16 is meaningful there, not just TIFF.
  - **Derive the extension once, share it with C1.** The format decision needs the lowercased extension — the same value `save_buffer_with_icc` computes. Factor a tiny helper (e.g. `fn output_kind(path) -> Option<Kind>` or just reuse a shared `supported_extension(path) -> Option<&str>`) so the *unsupported-extension* rejection (C1) and the *depth default* (this card) read one source of truth; do not re-list the extension set a third time. If the extension is unsupported, C1's error should still fire — order the depth choice **after** (or fused with) the supported-set check so an `out.bmp --depth 16` reports "unsupported extension", not a depth surprise. (`save`/`save_16` both end in `save_buffer_with_icc`, so the C1 check runs regardless; routing just picks which entry point.)
  - **Mirror in the GUI export** (`gui.rs:179`): `export` currently hard-codes `latent_export::save` (8-bit). Apply the same auto-by-extension rule there so a GUI export to `.tiff`/`.png` is 16-bit too. (The GUI export path comes from `self.output`, default `<input>.jpg` → 8-bit, which stays correct.) Keep the routing logic in *one* place if practical — e.g. a `latent_export`-level `save_auto(img, path, depth: Option<Depth>)` both front-ends call — so the CLI and GUI cannot drift. If you add such a helper, it lives in `latent-export` next to `save`/`save_16`.
  - **L1 (JPEG quality) is a separate register item and not in scope for this card** — but the `Develop` args plumbing here is the same surface L1 would extend (`--quality`). If you touch the args, leave the shape extensible (a struct or clearly-grouped args) so a later `--quality` lands cleanly; do not implement it.
  - Default behavior change is intentional and user-visible: `latent develop in.raw out.tiff` now writes **16-bit** where it wrote 8-bit. That is the fix (honors the high-quality option). Document it in the arg help text (`--depth 8|16 (default: 16 for tif/tiff/png, 8 for jpg/jpeg)`).
- Acceptance:
  - `latent develop in.raw out.tiff` (no `--depth`) writes a **16-bit** file; `out.png` writes 16-bit; `out.jpg` writes 8-bit.
  - `--depth 8` forces 8-bit on a `.tiff`; `--depth 16` forces 16-bit on a `.png`; `--depth 16` on `.jpg` is rejected (no 16-bit JPEG) with a clear error **or** documented to clamp to 8 — pick one and test it.
  - The GUI export routes by the same rule (a `.tiff` export from the GUI is 16-bit).
  - The extension set is defined **once** and shared by the C1 rejection and this depth choice.
  - **Tests** (land in C5, named here): `develop_selects_16bit_for_tiff` / `develop_selects_8bit_for_jpeg` (assert the chosen depth/written `ColorType`, e.g. via a small fixture or by checking the produced file's `image::open(...).color()`), and a CLI-arg test `develop_parses_depth_flag` (`Cli::try_parse_from(["latent","develop","in","out.tiff","--depth","8"])`). Baseline green.

### E8-C4 — Render/export off the egui UI thread (worker + progress)
- Implements: CR-04 **H1** (+ **L3**: return the texture handle, no `self.texture.unwrap()`)            Priority: High
- Crates/files: `latent-app/src/gui.rs` (`render_preview:156-170`, `export:173-183`, `update:187-394` — the synchronous call sites `:335-337`, `:390-392`, and the `self.texture.unwrap()` at `:339-340`); the `App` struct fields `:96-124`.
- Depends on: **E7-C4** (`Backend: Send + Sync`) — required for the worker to hold the backend across the thread boundary; E8-C1/C2 (export returns typed errors the worker forwards) — soft.            Blocks: —
- Heads-up:
  - **The single most user-visible engineering weakness.** Both `export` (full-resolution, `self.full`) and `render_preview` (bounded by `PREVIEW_MAX_DIM = 1600`, `self.preview`) call `render(...)` **inline in `update`** (`:335-337`, `:390-392`). While `render` runs, the egui window neither repaints nor processes input — on a large RAW the editor appears **hung for the entire export**. Move the render off the update thread.
  - **Worker design.** Spawn a `std::thread` (or `poll_promise::Promise`) per render request, communicating over `std::sync::mpsc` (or an `Arc<Mutex<Option<Result<…>>>>` slot). The worker runs `render(&base, &settings, backend)` and sends back the result; for the **preview** that is the rendered `ImageBuf` (the main thread turns it into a `ColorImage`/`TextureHandle` — texture upload must stay on the egui/`ctx` thread, it is not `Send`); for the **export** that is the `latent_export::save(...)` `Result<(), _>` (which *can* run fully off-thread, including the file write). On completion the worker calls `ctx.request_repaint()` so the UI wakes and consumes the result on its next `update`.
  - **Gate re-entry (one render in flight).** Add an `App` field tracking in-flight state (e.g. `rendering: bool` or `pending: Option<Receiver<…>>`). In `update`, when `dirty`/`painted` would trigger a render: if a render is already in flight, mark a "needs another" flag and skip spawning a second (coalesce to the latest settings when the current one finishes) — do not queue unbounded threads. **Disable the Export button while an export is running** (the §UX floor). Show a status: `Rendering…` / `Exporting…`, cleared/replaced on completion (reuse the existing `self.status` string and the export-result formatting at `:179-182`).
  - **Data ownership — clone what the worker needs.** The worker must **own** its inputs: clone the `Settings` (`self.variants[active].current().clone()`) and the base `ImageBuf` it renders (or wrap the bases in `Arc` once — `self.full`/`self.preview` are read-only during a render, so `Arc<ImageBuf>` avoids deep-cloning the full-res image per export). The `Backend` is `Box<dyn Backend>` and must be reachable from the worker: with **E7-C4's `Send + Sync`**, share it as `Arc<dyn Backend + Send + Sync>` (construct once in `run`, clone the `Arc` into each worker) — this is the clean shape now that the trait is `Send+Sync`. Do **not** try to `&`-borrow `self` into the thread.
  - **lensfun constraint (do NOT violate):** the lensfun `Database` handle is **not `Send`/`Sync`** by design (`CR-04 §3`). It is **never** part of a render — the lens lookup runs once, pre-window, in `auto_lens_profile` (`gui.rs:81-94`) on the main thread, and stores a resolved `LensProfile` (plain `Send` data) in `Settings::geometry.lens`. The worker renders from that already-resolved profile. **Keep the DB off the worker.** If you ever need a *fresh* lens lookup mid-session (you do not, for this card), it must happen on the main thread and only the resolved `LensProfile` crosses to the worker. This is precisely why E7-C4 made `Backend` (not `Database`) `Send+Sync`.
  - **L3 — return the texture handle (required once render is async).** `update` currently does `self.texture.as_ref().unwrap()` at `:339-340`, sound today only because the first frame has `dirty = self.texture.is_none()` true (`:188`) and the *synchronous* `render_preview` always sets `self.texture` before that line. Once preview rendering is async, the texture is **not** ready on the first frame — that `unwrap` becomes a first-frame panic. Fix it: when no texture exists yet, render the central panel without the image (show the `Rendering…` status) and `request_repaint`; only paint the `egui::Image` once a `TextureHandle` exists. Have `render_preview`'s completion path *return/hand back* the rendered image so the call site loads the texture explicitly (no reliance on the `self.texture.unwrap()` invariant). At minimum, replace the two `unwrap()`s with a `let Some(tex) = &self.texture else { ctx.request_repaint(); return; }`-style guard.
  - **Preview-equals-export must still hold.** The shared output transform (`to_display`/`to_srgb8`) gives structural preview/file agreement (CR-04 §7 positive). Moving render off-thread must not change *what* is rendered — same `render(base, settings, backend)`, just on another thread. Do not let the async refactor fork the preview vs export code paths beyond resolution (`self.preview` vs `self.full`). Keep the brush-paint flow (`:355-385`, which `render_preview` + `request_repaint` after a dab) working through the same in-flight gate.
  - **Scope:** at minimum move `export` off-thread (the worst case); ideally `render_preview` too (the decision says "ideally"). If you stage it, do export first (it's the clear win and simpler — fire-and-forget a `Result` back), then preview (which needs the texture-upload-on-main-thread handoff). Either way L3's unwrap fix is required the moment preview becomes async; if you only thread `export`, the `unwrap` can stay but still harden it per L3 as the decision marks it required-with-H1.
- Acceptance:
  - Triggering a full-resolution export does **not** freeze the window: the UI keeps repainting and shows an `Exporting…` status; on completion the status becomes `Exported to <path>` or `Export failed: <e>` (the existing `:179-182` formatting), driven by a `request_repaint` from the worker.
  - Only one render/export runs at a time; a second Export attempt while one is in flight is disabled (or coalesced), never spawning an unbounded number of threads.
  - The lensfun `Database` is **not** referenced from any worker thread (lookup stays in `auto_lens_profile`, pre-window, main thread); the worker uses only the resolved `LensProfile`/`Settings`/`Backend`.
  - No `self.texture.unwrap()` panic is reachable on the first frame (L3): a fresh editor with async preview renders without panicking before the first texture arrives.
  - **Tests** (gui threading is hard to unit-test end-to-end; pin the testable seams): a unit test that the in-flight gate coalesces (a `request_render`-style helper sets/respects the `rendering` flag) and a test that the export-result message formatting maps `Ok`/`Err` correctly (factor the `:179-182` formatting into a pure `fn export_status(Result<(),_>) -> String` and test it). Manual/`verify` for the no-freeze behavior. Baseline green; **note the E7-C4 `Send+Sync` dependency in the PR** so it is sequenced after.
