> Independent stream. ### E3-C1 — Thread lensfun geometry through `latent-lens` (`RealFocal`, `Crop`, model, POLY3 TCA, center) + finiteness guards
- Implements: IP-05 §2.3 / §2.4 (data plumbing), CR-04 **M4**            Priority: High
- Crates/files: `latent-lens/src/lib.rs` (`lens_to_profile`, `ca_offsets`, `vignetting_falloff`, `radial_distortion`); `latent-edit/src/lib.rs` (`LensProfile`); `latent-pipeline/src/lib.rs` (`lens_radial`, `Warp`, `apply_geometry`) — signatures/fields only, the *math* lands in C2–C4.
- Depends on: —             Blocks: E3-C2, E3-C3, E3-C4, E3-C5
- Heads-up:
  - **This card is plumbing — it carries the data the later cards need to be correct, and adds the M4 guards. Do not change the normalization math here (that is C2); just make the right numbers reachable.**
  - The current `LensProfile` (`latent-edit:533`) carries only `center`, `distortion[4]`, `ca[2]`, `vignetting[3]` — it has **thrown away** everything needed to normalize at lensfun's scale. lensfun's `NormScale = hypot(36,24)/Crop/hypot(W+1,H+1)/RealFocal` (`docs/geometry-lensfun-modifier.cpp:79`) needs **`Crop`** (the camera crop factor) and **`RealFocal`** (from `lf_lens_interpolate_distortion`'s `lcd.RealFocal`, *not* the nominal focal — they differ on some lenses; see `modifier.cpp:71-75`). Add `crop: f32` and `real_focal: f32` to `LensProfile` (with `#[serde(default)]` so old sidecars still load — E7 forward-compat rule). Default them so an all-default `LensProfile` stays a no-op.
  - lensfun coefficients are model-specific and live in the *Hugin* half-long-edge frame, not the engine's half-diagonal frame; C2 will rescale them. So the **distortion model type must survive** into the profile (or the rescaled coefficients must be computed here): add a `model` discriminant (POLY3 / POLY5 / PTLENS) so C2/C3 can branch (Newton for POLY3/POLY5, direct multiply for PTLENS — see C3). Capture `dist.Model` in `lens_to_profile` (`latent-lens:147`) alongside the terms. Per `docs/geometry-lensfun-lens-models.html`: POLY3 `r_d = r_u(1 - k1 + k1 r_u²)`, POLY5 `r_d = r_u(1 + k1 r_u² + k2 r_u⁴)`, PTLENS `r_d = r_u(a r_u³ + b r_u² + c r_u + 1-a-b-c)`.
  - **Full POLY3 TCA:** `ca_offsets` (`latent-lens:212`) currently collapses POLY3 TCA to its on-axis `vR, vB` constant, dropping the radius dependence. The real model (`lens-models.html:341`) is `r_{d,R} = r_{u,R}(b_R r_{u,R}² + c_R r_{u,R} + v_R)` (XML `br, cr, vr, bb, cb, vb`; `vr/vb` default 1, rest 0; interpolated `Terms` order is the six attrs). Widen the CA carrier from `[r,b]` constants to a per-channel **radial polynomial** so C4 can apply it; LINEAR (`r_d = r_u·k`, attrs `kr/kb`) becomes the degenerate `[0, 0, k]` case. The green channel is the reference (scale 1, no offset). Pick a representation C4 can consume directly (e.g. `[[b_R,c_R,v_R],[b_B,c_B,v_B]]`).
  - **CR-04 M4 — finiteness guards (the concrete ask of this card):** `radial_distortion` (`latent-lens:198`) already guards its output with `.is_finite()` → no-op fallback. `ca_offsets` (`:212`) and `vignetting_falloff` (`:225`) pass interpolated lensfun terms straight through unchecked. Add the **same** guard to both new/updated mappers: a non-finite CA term falls back to the no-op (unit scale / zero offset — green-equivalent), a non-finite vignetting term falls back to unity falloff `[0,0,0]`. A corrupt DB entry must not poison every pixel. (Boundary-sanitize per Global rule 4: validate here, trust downstream.)
  - Keep the FFI RAII/`lf_free`-on-every-path discipline intact (CR-04 §7 positives) — you are editing the mappers, not the lookup.
- Acceptance:
  - `LensProfile` carries `crop`, `real_focal`, the distortion `model`, and a per-channel radial-TCA representation; defaults make a fresh profile a no-op; an old sidecar without the new fields still deserializes (serde-default test).
  - `ca_offsets` and `vignetting_falloff` return the no-op fallback on a non-finite interpolated term — **tests** `ca_offsets_guards_non_finite` and `vignetting_falloff_guards_non_finite` (mirror the existing `degenerate_distortion_falls_back_to_no_op`), each fails before the guard and passes after.
  - A **test** `poly3_tca_keeps_radial_terms` asserts the POLY3 mapper now retains `b_R, c_R` (not just `v_R`), unlike the old on-axis-only path; the existing `poly3_tca_uses_the_on_axis_scale` is updated/superseded (its constant is now just `v_R`).
  - `lens_radial` / `Warp` signatures accept the threaded geometry without yet changing `inv_norm` (that is C2); `cargo test --workspace`, `cargo clippy --all-targets` (zero warnings), `cargo fmt --check` stay green.

### E3-C2 — Half-diagonal (focal-scaled) radius normalization + center-offset scaling; fix the false "matching lensfun" doc
- Implements: IP-05 §2.3 (**Critical**) + §2.3-L1, CR-04 **M3**            Priority: Critical
- Crates/files: `latent-pipeline/src/lib.rs` (`lens_radial:672`, and the `RadialGain`/`Warp` call sites in `apply_geometry`); `latent-lens/src/lib.rs` (`center` mapping `:142`); `latent-edit/src/lib.rs` (fix the doc at `:528`).
- Depends on: E3-C1             Blocks: E3-C3, E3-C5, E3-C6
- Heads-up:
  - **This is the headline Critical fix. Every lensfun coefficient is currently applied at the wrong radial scale — off by ~1.3–1.8× per even order, aspect-dependent, worsening toward the corners. It must land before C3, or Newton inverts at the wrong scale.**
  - The bug: `lens_radial` (`pipeline:675`) sets `inv_norm = 2.0 / w.min(h)`, i.e. radius `1` at the **half-short-edge** — that is the **PanoTools/Hugin** unit, *not* lensfun's. The doc at `latent-edit:528` claiming this "matches lensfun" is **false** (it describes the half-short-edge as the lensfun radius). lensfun normalizes by the **focal-scaled half-diagonal**: `NormScale = hypot(36,24)/Crop/hypot(W+1,H+1)/RealFocal` (`docs/geometry-lensfun-modifier.cpp:79`), where `W = width-1`, `H = height-1` (pixel-centre convention, `modifier.cpp:68-69`). `inv_norm` must become this `NormScale` (built from the `crop`/`real_focal` threaded in C1). Note `hypot(36,24)` is the 35mm full-frame diagonal in mm; `Crop` and `RealFocal` convert the sensor-pixel diagonal into focal-length units — radius `1` is then "one focal length on the sensor", lensfun's natural unit (1).
  - **Coefficient frame mismatch (do not skip):** lensfun does *not* feed raw XML coefficients into a half-diagonal-normalized radius. It **rescales** the Hugin-frame coefficients into the focal-normalized frame in `rescale_polynomial_coefficients` (`docs/geometry-lensfun-mod-coord.cpp:37-71`): `hugin_scaling = RealFocal / hugin_scale_in_millimeters`, `hugin_scale_in_millimeters = hypot(36,24)/CropFactor/hypot(AspectRatio,1)/2`. POLY3: `k1 *= hugin_scaling² / (1-k1)³`. POLY5: `k1 *= hugin_scaling²`, `k2 *= hugin_scaling⁴`. PTLENS: `a *= hugin_scaling³/(1-Σ)⁴`, `b *= hugin_scaling²/(1-Σ)³`, `c *= hugin_scaling/(1-Σ)²`. So either (a) apply `NormScale` to the *radius* and the rescale to the *coefficients* (the lensfun split), or (b) fold both into the coefficients once — but **the radius unit and the coefficient frame must be the same**, or the fix is only half-done. The C1 `model` discriminant is what lets you branch the rescale. (The existing POLY3/PTLENS `(1-k1)`/`D` factoring in `radial_distortion` already encodes part of this; reconcile, don't double-apply.)
  - **Center offset (2.3-L1 / CR-04 M3):** `latent-lens:142` does `center = [0.5 + CenterX, 0.5 + CenterY]`, treating `CenterX/Y` as a fraction of the **full** dimension — wrong on a non-square sensor (lensfun uses the same divisor on both axes). lensfun: `CenterX = (W/2 + size/2·Lens->CenterX)·NormScale` with `size = min(W,H)` (`modifier.cpp:83-85`). So the offset is `CenterX/Y` scaled by **`min(w,h)/2`** in pixels (then by `NormScale` when expressed in normalized coords). Map it explicitly and **pin the `LensProfile.center` convention with a comment** so the wiring assumption is documented (CR-04 M3 ask). The vignetting `RadialGain` call site (`pipeline:698`) and the `Warp` call site (`pipeline:735`) both pull center+`inv_norm` from `lens_radial`, so fixing it once corrects both — keep them sharing one `lens_radial`.
  - The PA vignetting model uses a **different** radius unit (`r=1` at the *corner*, `corrections.html`/`modifier.cpp:34`), unlike distortion/TCA (`r=1` at one focal length). The vignetting `RadialGain` may therefore need its own normalization (corner-anchored) rather than `lens_radial`'s focal-scaled `NormScale` — verify against the PA apply path; this couples to C5. Do **not** silently reuse the distortion `inv_norm` for vignetting without checking the unit.
  - Rewrite the `lens_radial` doc-comment (`pipeline:668-671`) to state the half-diagonal/focal-scaled unit, and replace the false `latent-edit:528` comment with the correct convention.
- Acceptance:
  - `lens_radial` returns `inv_norm == NormScale` (focal-scaled half-diagonal), and the distortion coefficients are in the matching (rescaled) frame; a **test** pins `NormScale` for a known `(w,h,crop,real_focal)` against the hand-computed `hypot(36,24)/Crop/hypot(w,h-as-W+1)/RealFocal` value.
  - Center offset scales by `min(w,h)/2`: a **test** `off_center_scales_by_min_dimension` on a non-square frame asserts a non-zero `CenterX` shifts the center by `CenterX·min(w,h)/2` px (and would *fail* under the old `fraction-of-full-dimension` mapping). Closes CR-04 M3.
  - The false "matching lensfun" doc at `latent-edit:528` is gone, replaced by the correct half-diagonal/focal-scaled statement; `lens_radial`'s doc matches.
  - A coarse **round-trip** check (full grid test is C6): straightening a synthetic grid distorted with a known POLY5 entry lands corner points within sub-pixel of the lensfun expectation — proving the scale, not just the form. `cargo test --workspace`, clippy (0 warnings), fmt stay green.

### E3-C3 — Model-aware Newton inversion for POLY3/POLY5
- Implements: IP-05 §2.2            Priority: Medium
- Crates/files: `latent-pipeline/src/lib.rs` (`Warp::map:250-268`, the `s(r)` evaluation); the lens lowering in `apply_geometry` / `lens_radial`.
- Depends on: E3-C2 (must invert at the correct scale)             Blocks: E3-C6
- Heads-up:
  - **The engine currently applies a single direct `s(r)` multiply for *all* models. For lensfun POLY3/POLY5 that is only first-order-correct (sub-pixel error, but not bit-exact vs lensfun). This card makes POLY3/POLY5 bit-exact by solving the distortion equation by Newton iteration, exactly as lensfun's `UnDist` callbacks do; PTLENS keeps the direct multiply (the register's explicit decision).**
  - Direction matters and is easy to get backwards. The engine's `Warp::map` runs **output → source** (it maps a corrected output pixel to the source pixel to sample) — this is lensfun's **`UnDist`** direction (undistort). lensfun's `UnDist_Poly3`/`UnDist_Poly5` (`docs/geometry-lensfun-mod-coord.cpp:560`, `:634`) solve `Rd = Ru·(1 + …)` for **`Ru`** by Newton's method, *not* by a direct polynomial multiply. POLY3 (rescaled to even form `Rd = Ru + k1·Ru³`): iterate `f(Ru)=Ru³ + Ru/k1 − Rd/k1`, `f'=3Ru² + 1/k1`, seed `Ru=Rd` (`mod-coord.cpp:586-600`). POLY5: `f(Ru)=Ru(1 + k1·Ru² + k2·Ru⁴) − Rd`, `f'=1 + 3k1·Ru² + 5k2·Ru⁴`, seed `Ru=Rd` (`:650-661`). lensfun runs to a tolerance (`NEWTON_EPS`, cap ~5 steps, NaN on non-convergence/negative radius); the register says **2–3 iterations** suffice at our precision — do a fixed small count or the tolerance loop, whichever you test as sub-pixel.
  - **PTLENS stays direct** (register 2.2): keep the current `s(r)` Horner multiply for PTLENS. Rationale to preserve: lensfun *also* has `UnDist_PTLens` (Newton), but the register's decision is to keep the direct multiply for PTLENS where it is the defined `Dist` operation — honor that. The `model` discriminant from C1 is what selects the path: Newton for POLY3/POLY5, direct for PTLENS (and the all-zero no-op short-circuit at `map:258` stays).
  - This only changes the **radial scalar** `s` (the `r → r·s` step at `map:265-267`); the homography (`:251-257`), the `w<=0` guard (`:252`), and the per-channel CA hand-off (`map_channel:272`) are unchanged. Keep the Horner/`f32` style and the single-resample composition (Global rule 5: the geometry stage interpolates exactly once).
  - **GPU lockstep (Global rule 1):** this is `warp` math. There is no GPU warp shader yet (it is built in **E6-C2 / 2.9**); when it lands it must mirror this Newton inversion. Leave a note so E6 picks it up, and keep the CPU `Warp::map` the single reference E6 mirrors — do not fork the math.
- Acceptance:
  - For a POLY3 and a POLY5 entry, `Warp::map` recovers the undistorted radius to sub-pixel: a **test** `newton_inverts_poly3_to_subpixel` / `newton_inverts_poly5_to_subpixel` forward-distorts a radius with `Rd = Ru(1+…)`, runs `map`, and asserts the recovered `Ru` matches within < 0.01 px at the corner (fails under the old direct-multiply, passes after).
  - PTLENS still uses the direct multiply — a **test** asserts the PTLENS path is byte-for-byte the pre-card behavior (no Newton applied).
  - 2–3 iterations (or the chosen tolerance) achieve the sub-pixel bound; non-convergent/negative-radius inputs degrade gracefully (sample-outside / no panic), matching the existing out-of-bounds handling. Baseline green (fmt/clippy/test).

### E3-C4 — Full POLY3 radial TCA in `Warp::map_channel`
- Implements: IP-05 §2.4            Priority: Medium
- Crates/files: `latent-pipeline/src/lib.rs` (`Warp.channel_scale` field, `Warp::map_channel:272`, `has_chromatic:286`, the `Warp` construction in `apply_geometry:744`); consumes the per-channel radial TCA carrier added in C1 (`latent-lens` `ca_offsets`).
- Depends on: E3-C1 (radial TCA carrier), E3-C2 (correct scale)             Blocks: E3-C6
- Heads-up:
  - **Today CA is a single constant per channel: `channel_scale[c]` is one number and `map_channel` scales the offset-from-center by it (`pipeline:274-281`). That captures only the on-axis (constant) TCA term and leaves residual CA on lenses with strong radius-dependent CA. This card makes it the full radius-dependent POLY3 scale.**
  - The model (`docs/geometry-lensfun-lens-models.html:341`): `r_{d,R} = r_{u,R}·(b_R r_{u,R}² + c_R r_{u,R} + v_R)`, same for blue with `b_B, c_B, v_B`; **green is the reference** (no scale). So the per-channel radial scale is `s_c(r) = b_c·r² + c_c·r + v_c` — a function of radius, not a constant. Replace the scalar `channel_scale: [f32;3]` with a per-channel polynomial (e.g. `[[b,c,v];3]`, green = `[0,0,1]`), evaluate `s_c(r)` at the pixel's radius in `map_channel`, and scale the offset-from-center by it. LINEAR (`r_d = r_u·k`) is the degenerate case `s_c(r) = k` (`[0,0,k]`) — keep it working.
  - **Radius convention must match C2/C3.** TCA shares the distortion radius unit (`r=1` at one focal length, the focal-scaled `NormScale` from C2), and lensfun applies anti-TCA on the **undistorted** coordinate (devignette → anti-TCA → undistort order, `corrections.html`). In the engine's fused single-resample `Warp`, the CA scale rides on the offset from center *after* the radial distortion `map` — preserve that composition (`map_channel` calls `map` then scales): the radius for `s_c` is the post-distortion source radius the channel samples at. Keep it folded into one interpolation (the whole point of `Warp`, `pipeline:207-211`) — do **not** add a second CA resample pass.
  - `has_chromatic` (`:286`) must return true when any channel polynomial differs from green's identity (not just `!= [1,1,1]`). The `apply_geometry` construction (`:744`) currently builds `channel_scale: [1+ca[0], 1, 1+ca[1]]` from the old 2-vector — rebuild it from the C1 radial carrier.
  - **GPU lockstep (Global rule 1):** `map_channel` is mirrored by the future `warp.wgsl` (E6-C2 / 2.9). Keep this CPU path the single reference; the per-channel polynomial form must be trivially portable to WGSL (no CPU-only constructs in the hot path). Flag for E6.
- Acceptance:
  - A radius-dependent POLY3 TCA entry is fully corrected: a **test** `poly3_tca_corrects_radial_term` builds a `Warp` with non-zero `b_R/c_R`, checks the red sample radius equals `r·(b_R r² + c_R r + v_R)` at several radii (not just the on-axis `v_R`) — fails under the constant-only path, passes after.
  - LINEAR TCA still works as the degenerate constant (`linear_tca_is_constant_scale` test); green is untouched (identity).
  - `has_chromatic` is true iff red/blue polynomials differ from green's identity; a no-CA profile still resamples all channels in one pass (no spurious per-channel split). Baseline green (fmt/clippy/test).

### E3-C5 — Verify & align PA vignetting convention (divide vs multiply) + flat-field test
- Implements: IP-05 §2.8            Priority: Medium
- Crates/files: `latent-lens/src/lib.rs` (`vignetting_falloff:225`); `latent-pipeline/src/lib.rs` (the vignetting `RadialGain` at `apply_geometry:697-707`, `RadialGain::at:193`).
- Depends on: E3-C1 (guards), E3-C2 (correct vignetting normalization)             Blocks: E3-C6
- Heads-up:
  - **This card is a *verification* card first: confirm against lensfun's own apply code whether the stored PA terms are the falloff (so dividing is right) or the correction gain (so the engine double-inverts), then align the divide/multiply and sign exactly. The model family, the pre-resample placement, and the sign are *believed* correct — prove it, don't assume it.**
  - The evidence already gathered: lensfun's PA model is `C_d = C_s / (1 + k1 r² + k2 r⁴ + k3 r⁶)` (`docs/geometry-lensfun-lens-models.html:395`). `C_d` is the corrected output, `C_s` the source — the polynomial is the **measured falloff** (its `k`s are negative for darker corners → `1 + (neg)` < 1 → dividing by it brightens the corners). The engine stores `vignetting = terms` straight through (`vignetting_falloff:226`) and applies a `RadialGain { poly: vignetting, reciprocal: true }` (`apply_geometry:701-705`), i.e. it **divides** by `1 + k1 r² + …` (`RadialGain::at:199`: `if reciprocal { 1/p }`). That **matches** lensfun's `C_s / (1 + …)`. So the current `reciprocal: true` / pass-through is most likely **correct** — but the card's job is to *confirm* it against the apply path and lock it with a test, and to fix it if the interpolated `Terms` turn out to be a gain instead.
  - **Radius unit caveat (couples to C2):** the PA model's `r = 1` is the image **corner** (`docs/geometry-lensfun-modifier.cpp:34`, `corrections.html`), unlike distortion/TCA whose `r = 1` is one focal length. So the vignetting `RadialGain.inv_norm` must be **corner-normalized**, not the focal-scaled distortion `NormScale`. Check what `lens_radial` now hands the vignetting call site after C2: if C2 routed the distortion `inv_norm` into the vignetting gain, that is wrong for PA — give the vignetting gain its own corner normalization (radius `1` at `hypot(w,h)/2` from center). This is the most likely real bug here; the divide/multiply may already be right.
  - The vignetting step is a **flat-field multiply applied before any resample** (`apply_geometry:694-707`), matching lensfun's "devignetting first, separable from the coordinate ops" (`corrections.html:125`). Keep that placement (it is verified correct) — only the polynomial sign/inversion and the radius unit are in scope.
  - The C1 finiteness guard on `vignetting_falloff` and any sign change must agree: a non-finite or absent term → unity falloff (`[0,0,0]`), a true no-op.
- Acceptance:
  - A written verdict (in the code-comment on `vignetting_falloff` / the `RadialGain` call site) stating, with the `lens-models.html:395` / apply-path citation, whether PA terms are falloff or gain and that the engine's `reciprocal`/sign is aligned to it — and the code matches that verdict.
  - **Flat-field round-trip test** `pa_vignetting_flattens_known_falloff`: synthesize an image with a known PA falloff `1 + k1 r² + k2 r⁴ + k3 r⁶` baked in (negative `k`s, darkened corners), apply the lens vignetting correction, assert the corners return to flat (within tolerance) — this is what catches a wrong divide/multiply *or* a wrong radius unit. Extend the existing `lens_vignetting_flattens_a_radial_falloff` (`pipeline:1652`) to use a real multi-term PA entry and corner-normalized radius.
  - If the radius unit was wrong (likely), the vignetting `RadialGain` is corner-normalized and a **test** pins `inv_norm` at the corner = 1. Baseline green (fmt/clippy/test).

### E3-C6 — lensfun round-trip tests (distortion grid, TCA target, vignetting)
- Implements: IP-05 §2.2 / §2.3 / §2.3-L1 / §2.4 / §2.8 (verification)            Priority: High
- Crates/files: `latent-pipeline/src/lib.rs` (tests); fixtures alongside (synthetic grids/targets/flats with known lensfun entries). No production code change — this card *proves* C2–C5.
- Depends on: E3-C2, E3-C3, E3-C4, E3-C5             Blocks: — (closes the epic)
- Heads-up:
  - **These round-trip tests are how the normalization is *proven* right — the headline Critical fix (C2) is invisible to a unit test of the formula; only a grid straightened to within sub-pixel of lensfun's own output shows the scale is correct. Per Global rule 2, a test pins every change; this card is the integration-level pin for the whole lens stack.**
  - Three round-trips, mirroring the register's named deliverables (`05-geometry-and-optics.md` §"Cross-cutting reminders"):
    1. **Distortion grid** — take a straight-line grid, *forward*-distort it with a real lensfun `<distortion>` entry (POLY5 and POLY3, since they exercise the C3 Newton path; and a PTLENS entry for the direct path), run the engine's correction, assert the lines are straight again and grid intersections land within sub-pixel of the lensfun-undistorted positions. This is the C2-scale + C3-inversion proof.
    2. **TCA target** — a high-contrast edge / point target split into R/G/B at known POLY3 TCA offsets; run the per-channel correction (C4) and assert the channels recombine (R/B align to G) within sub-pixel at several radii — proving the *radial* term, not just on-axis.
    3. **Vignetting flat-field** — a flat (uniform) field darkened by a known PA falloff; correct it (C5) and assert uniformity returns across the frame including corners. (May share the C5 flat-field test or extend it to multi-term + off-center.)
  - **Reference values:** the gold standard is lensfun's own output. If linking lensfun in a test is impractical, compute the expected undistorted/recombined/flattened coordinates from the lensfun model equations *by hand* (the `Dist`/`UnDist` formulas in `docs/geometry-lensfun-mod-coord.cpp`, the `NormScale`/rescale in `modifier.cpp`) and pin those — that is still a faithful lensfun reference because the engine and the reference derive from the same documented math but via independent code paths. Use real coefficient magnitudes from a representative DB entry, not toy values, so the corner error is meaningful.
  - Use **off-center** profiles (non-zero `CenterX/Y`) on a **non-square** frame in at least one grid case, so the C2/2.3-L1 center scaling is exercised (a square frame hides the M3 bug).
  - Keep these CPU-side; the **CPU/GPU parity** tests for the same geometry are **E6** (2.9 / CR-03 F3) once `warp.wgsl` exists — do not author the GPU side here, but write these so E6 can reuse the same fixtures/expected values (Global rule 1 lockstep).
